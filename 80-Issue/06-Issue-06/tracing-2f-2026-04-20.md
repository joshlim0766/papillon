## 관찰 1 — TDD 사이클 한 세션 수행 시 Sonnet 품질 저하

### 현상 (주인님 실측)

- 2f 를 Sonnet 으로 진행 중, 단일 세션 내에서 RED → GREEN → REFACTOR 사이클을 연속 수행.
- 진행이 누적될수록 Sonnet 판단 품질 저하 관찰. 주인님 표현 "빠르게 병신새끼가 되어감. 조금 커지면 착란 일으킬지도 모르겠어."
- Sonnet 자체가 도중 대화 자동 압축 시도 (auto-compaction 발동). 그러나 근본 해결 아님.

### 독립 검증

주인님이 Claude Web Sonnet 에 동일 맥락 공유 → 독립적으로 동일 분석 도달:

> TDD 사이클 자체가 의도적으로 상태를 전환하는 구조. Red → Green → Refactor 가 단순 코딩 순서가 아니라 "지금 뭘 보고 있는가" 의 **컨텍스트 전환**. 한 세션에 다 때려넣으면 모델 입장에서 실패 코드·성공 코드·중간 상태 신호가 뒤섞임.

### 원리 분석

TDD 3단계는 각각 다른 멘탈 모드:

| 단계 | 멘탈 모드 | 핵심 신호 |
|---|---|---|
| RED | "이 동작은 아직 구현 안 됨" 을 **의도적으로 기록** | 실패가 기대값 |
| GREEN | "최소 변경으로 통과만" | 성공이 기대값, 구조 최적화 금지 |
| REFACTOR | "테스트 유지하며 구조 개선" | 이미 녹색, 외부 관찰 불변 |

한 세션 안에 3 단계를 다 두면:
- RED 작성 시의 "실패 기대" 가 GREEN 작업 중에도 맥락에 남아 **과잉 방어 구현** 유도.
- GREEN 의 "최소 변경" 원칙이 REFACTOR 와 섞여 **구조 최적화 타이밍 혼선**.
- 이전 실패 케이스 주변 노이즈가 압축 시 비교 대상으로 끌려들어와 **잘못된 패턴 학습**.

### Issue-06 관점에서의 의미

현 spec v0.1.0-draft 의 **신호 #5 TDD 플래그 헤더** 와 **신호 #7 TDD 구현 규율**은 **지시서 품질** 에 초점. 하지만 이 관찰은 **shackled 소비 로직** (오케스트레이터의 세션 관리) 자체의 요건임을 시사.

즉 Issue-06 의 스코프가 확장될 필요:
- **기존**: TDD-ready 설계서 규격 (지시서 측).
- **추가**: TDD-ready 소비 로직 (shackled 오케스트레이터 측) — 세션 분할 메커니즘.

이 확장을 Issue-06 #4 "shackled tdd_mode 소비 로직 확인" 으로 처리하던 것에 **세션 분할 메커니즘** 을 신규 요건으로 추가.

### 임시 완화책 (2f 진행 중)

### 분석 1 — Opus 자동 분할 패턴

Opus 가 도출한 3 분할은 두 축의 결합:
1. **관심사 축**: 인프라 선제 / 메인 조립 / 운영 보강
2. **TDD 적합성 축**: 처음 2개는 `mode=tdd`, 운영 보강은 `mode=normal` (테스트가 아닌 개선 성격)

이 패턴은 "단일 관심사" 원칙을 **유지하되 관심사를 재분해** 한 것. "UPSERT 서비스" 같은 거시적 관심사가 실구현에서는 "인프라 지원 / 조립 / 운영" 3 서브 관심사로 분해 가능.

### 분석 2 — L1 분할만으론 부족 (L2 필요 신호)

**가장 큰 태스크(TASK-2F-02) 에서 Full GC 발생**. 즉 L1 (지시서 → 태스크) 분할 후에도 단일 태스크가 단일 세션에 안 맞음. L2 (태스크 내부 → RED 그룹) 필요.

- TASK-2F-02 에는 RED 1~12 가 모두 몰려 있음.
- 관찰 1 (시간축) 의 하이브리드 C→A 가 L2 에 해당 — "RED 그룹 × 세션" 분할.

**교차 구조**: 공간 축 L1 + 시간 축 (L2) 가 서로 다른 레이어에서 필요.

### 분석 3 — WORKAROUND 비용

TASK-2F-01 이 `StatsSendResultListener WORKAROUND 임시 전환 포함` 명시로 completed 처리. 의미:

- 인프라 선제 태스크가 **독립 GREEN** 하려면 메인 태스크(2F-02) 미완성 상태의 의존을 임시 처리.
- 의존 분할의 **비용**: 후속 태스크 완료 시 WORKAROUND 제거 확인 필요.
- 상태 파일 `result_summary` 에 WORKAROUND 가 박제돼야 추후 검증 가능 (2f 상태 파일은 문자열로 명시 — 수동 규율).

### Issue-06 관점에서의 의미

- **신호 #1 RED 시나리오 번호**: 기존엔 "RED 1~N" 번호화만 규정. 실측 결과 RED 수 자체가 분할 신호 (N > 6 이면 L2 분할 권고).
- **신호 #8 상태·부작용 경계**: WORKAROUND 허용 조건 + 제거 검증 메커니즘 추가 요건 가능.
- **신규 신호 후보**: "지시서 크기 경계" — 단일 관심사가 L1 분할을 요구하는 임계를 명시 요건화.
- **Issue-06 #4 shackled 소비 로직** 확장: L1/L2 2-레이어 분할 메커니즘 포함.

### 후속 설계

## 관찰 3 — 라운드별 P0/P1 재귀 패턴 + 실패 경로 계약 공백

### 현상 (주인님 실측, Claude 분석)

Phase 2F wtth 코드 리뷰 3라운드에서 P0/P1 이 지속 발생. 각 라운드의 성격이 다름:

| 라운드 | P0/P1 본질 | 예시 |
|---|---|---|
| 1R (1/10) | 지시서 계약 vs 초안 구현 누락 | Validator 미호출, notRetryable 5종/6종, DLT 헤더 스키마 결락 |
| 2R (1/12) | 1R 수정의 부작용 + 테스트 격리 약점 노출 | send future 미확인, parseKey edge, Counter 공유 누적, offset 누적 |
| 3R (1/7) | 2R 수정의 부작용 (blocking 이 만든 새 실패 경로) | Recoverer TimeoutException 재시도 루프, dimensions 공백 markAll 불일치 |

**재귀 패턴**: 앞 라운드 수정이 새 실패 경로를 여는데, 지시서는 **성공 경로만 명시**해서 새 경로의 계약이 매 라운드 공백.

### 근본 원인 3가지

1. **지시서가 성공 경로 중심, 실패 계약 누락**
   - 예: §2.4 는 "notRetryable 목록" + "FixedBackOff" + "commitRecovered" 는 박제했으나 "Recoverer 자체가 예외를 던지면?" 미명시 → 3R P0.
   - 예: §2.6 DltPublisher 의사코드에 "send()" 시그니처만 있고 "send 실패 시 정책" 없음 → 2R M-1 + 3R W.

2. **한 섹션에 관심사 혼재**
   - 예: §2.4 = 코드 예시 + 재시도 정책 + latency budget + notRetryable 목록 4 관심사 혼재.
   - 한 관심사 수정이 다른 관심사 불일치 유발. 묶음 S (notRetryable 6종) 수정 후 budget 수식이 send timeout 미반영 (묶음 W).
   - **공간 축 (§3) 과 연결**: L1 분할이 이 혼재를 해소하는 한 방향.

3. **테스트 강화가 인프라 약점을 시간차로 노출**
   - 1R 묶음 F/G (RED 강화) → 2R 에서 flaky/Counter 공유/offset 누적 드러남.
   - 2R blocking send → 3R 에서 Recoverer 재시도 루프 드러남.
   - 코드·지시서·테스트가 서로 엇갈린 속도로 진화 → 한쪽만 업데이트되면 다른 쪽이 다음 라운드 P0 공급원.

### 해결 방향 3가지 (Claude 분석)

| 방향 | 성격 | 효과 | 한계 |
|---|---|---|---|
| A — 지시서 섹션 분할 | 공간축 (§3 L1) 적용 | 관심사 혼재 해소 | 실패 경로 자체가 누락됐으면 분할해도 빈칸 유지 — **부분 해결** |
| B — **실패 계약 표 박제** | `when X fails → then Y` 표를 각 컴포넌트에 | 3 라운드에 걸쳐 발생한 모든 P0/P1 사전 결정 가능했음 | **핵심 보완** |
| C — 테스트 계약 동기화 | "테스트 없는 동작은 지시서에 없다" 원칙 + RED 의무화 목록 | 테스트-지시서 드리프트 제거 | 심층 보완 (TDD 규율 승격) |

### 실패 계약 표 예시 (방향 B)

```
| 컴포넌트              | 실패 시나리오               | 기대 동작                         | 지표/알람        |
|---|---|---|---|
| DltPublisher.send     | Kafka broker under-min-ISR  | .get(5s) TimeoutException 전파    | n/a              |
| Recoverer             | dltPublisher TimeoutException| swallow + counter + ERROR log + commit | dlt.send.failed  |
| StatsBatchPipelineImpl| dimensions 공백             | markAll(newEvents) 호출 후 return | n/a              |
```

이 표가 1R 이전에 있었다면:
- 1R 묶음 A (Redis silent skip) 논쟁 사전 결정.
- 2R M-1 (send 실패) 사전 결정.
- 3R V (Recoverer 예외) 사전 결정.

| AB-2 (1줄) | 프로덕션 알람 오진 방지 — IAE 를 UPSERT_FAIL 로 오분류하면 DB 장애 오인 |
| AB-3 (1줄) | 테스트 인프라 flaky 예방 — 실제 flaky 후 추적 비용 > 지금 1분 |

**후속 위임 (AB-1 / AB-4 / AB-5 / ARCH 4R)**: 운영 피드백 후 처리. 근거 — 다음 배치 + API 설계가 파이프라인 사용 패턴 결정, 그 전 edge case 방어는 오버엔지니어링 위험.

### 신호 #9 실증 — AB-2 구체 사례

**DltErrorReason.from 수정 1줄** (`DltErrorReason.kt`):
```kotlin
is IllegalArgumentException -> return LOGIC_ERROR  // AB-2 추가
```

이 1줄이 있었다면:
- **1R 에서 사전 결정 가능**: "예외 타입 unmapped → LOGIC_ERROR" 실패 계약 표 항목.
- **4R 까지 끌려오지 않았음**: 프로덕션 알람 오진 (IAE 가 UPSERT_FAIL 로 분류되어 DB 장애로 혼선) 미사전 방어.

**v0.2.0 신호 #9 실패 경로 계약 표 템플릿 후보 반영**:
```
| 컴포넌트             | 실패 시나리오                    | 기대 동작          | 지표/알람                |
|---|---|---|---|
| DltErrorReason.from  | 예외 타입이 cause 체인에 unmapped | LOGIC_ERROR 반환   | dlt.published{reason=LOGIC_ERROR} |
```

### 품질 경계 도달 지점 (wtth 수렴 판정 실측 휴리스틱)

주인님 + Claude 공동 판단한 **"마무리 적절 시점"** 의 구체 신호:

1. **P0 0건 연속 유지**: 2R 이후 P0 없음.
2. **P1 성격 수렴**: 테스트 커버리지 / 리소스 레벨로 이동 (= 계약 자체의 공백 아닌 보완 레벨).
3. **1라운드 반영 건수 감소 곡선**: 1R 11건 → 2R 다수 → 3R 단일 번호(7) → 4R 선별 AB-2/3.
4. **대안 선택지 가용**: "지금 고치기 vs 운영 후 고치기" 의 ROI 비교 가능 단계.

이 4 지표가 동시 충족되면 wtth "사인오프" 내부 전환 조건. common-spec §1 심각도 기준 + `02-wtth/02-design/00-core.md` §4.1 수렴 메커니즘에 연계 가치.

### 후속 위임 패턴 — 실제 경로 발생 시 추가

AB-1/4/5 의 공통 특징: **실제 운영 경로에서 발생해야 실증적 수정 가능**한 edge case 방어. 초기 지시서에 모든 실패 경로를 박제하면 오버엔지니어링.

**신호 #9 실패 경로 계약 표의 적용 범위 가이드**:
- **MUST 포함**: 프로덕션 알람 품질 / 리소스 누수 / 재시도 루프 등 **관측 가능한 운영 피해** 를 초래하는 실패.
- **SHOULD 포함**: 도메인 비즈니스 로직의 분기 실패 (ORIGINAL_NOT_FOUND / BMS_OPTIONAL_FIELD_NULL 등).
- **운영 피드백 후 추가**: 특정 인프라 조합 / 희귀 타이밍 / 부하 패턴 별 edge case.

이 가이드는 v0.2.0 draft 에 "실패 계약 표 작성 범위" 규칙으로 박제.
