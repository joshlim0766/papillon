## 1. 개요

shackled은 papillon 파이프라인의 Phase 4(구현)를 담당하는 독립 스킬이다. 설계서(또는 요구사항 정의서)를 기반으로 AI가 구현을 수행하고 사람이 방향을 확인하는 페어 프로그래밍 협업 도구이다.

### 1.1. 호출 방식

| 호출 방식 | 설명 |
|---|---|
| `/papillon:shackled` | 단독 실행. 파라미터를 직접 제공하거나 shackled이 사람에게 확인하여 진행 |
| papillon 경유 | papillon 오케스트레이터가 Phase 4에서 파라미터를 구성하여 자동 호출 |

### 1.2. 단독 실행 vs 파이프라인 내 실행

| 항목 | 단독 실행 (`/papillon:shackled`) | 파이프라인 내 실행 |
|---|---|---|
| 호출 주체 | 사람이 직접 | papillon 오케스트레이터 |
| 파라미터 제공 | 직접 입력하거나 shackled이 확인 | 오케스트레이터가 구성하여 전달 |
| wtth 호출 | 수행하지 않음. 구현 결과만 반환 | 오케스트레이터가 별도로 wtth 호출 |
| 이후 동작 | 구현 결과를 사람에게 반환. 필요 시 사람이 직접 `/papillon:wtth` 호출 | 오케스트레이터가 Phase 5(코드 리뷰)로 자동 전환 |

### 1.3. 모델

- Sonnet — 구현 작업은 코드 생성·편집이 주이고 설계 판단은 이미 완료된 상태이므로 속도·비용 효율 우선

### 1.4. 시블링 구조

shackled과 wtth은 시블링 관계이다. shackled은 구현에만 집중하고 wtth 호출은 papillon 오케스트레이터의 책임이다. shackled이 wtth을 직접 호출하지 않는다.

### 1.5. 리뷰 계층 구분

두 리뷰는 역할이 다르며 중복이 아니다:

| 리뷰 주체 | 시점 | 역할 | 관점 |
|---|---|---|---|
| shackled 내부 크로스 체크 | 태스크별 구현 완료 후 | "설계대로 구현했나?" — 방향 이탈·누락 점검 | 자기 검수 |
| wtth (오케스트레이터 호출) | 전체 구현 완료 후 | "잘 만들었나?" — 품질·안전성·다각도 점검 | 제3자 검수 |

## 2. 입력 인터페이스

### 2.1. 파라미터 정의

| 파라미터 | 필수 | 설명 |
|---|---|---|
| `context_path` | 필수 | M 체급: 확정된 설계서 경로. S 체급: 인터뷰 요약(`00-summary.md`) 또는 요구사항 경로. 없으면 진행 거부 |
| `weight` | 필수 | 체급 (`S` / `M`). 입력 소스와 흐름이 달라짐 |
| `output_type` | 필수 | 산출물 유형 (`code` / `runbook`). 4-A/4-B 분기 및 페르소나 조합 결정에 사용 |
| `task_id` | 선택 | 중단 후 재개 시 이어받을 태스크 식별자 |
| `exec_mode` | 선택 | 실행 모드 (`auto` / `pair`). 기본값 `auto`. 페어 모드는 확장 포인트 예약 |

### 2.2. context_path 없음 처리

`context_path`가 제공되지 않으면 shackled은 진행을 거부하고 사람에게 다음을 안내한다:
- 설계서 또는 요구사항 경로 제공 요청
- M 체급: 설계서가 없으면 papillon Phase 2~3(설계 리뷰)부터 시작 권장
- S 체급: `00-summary.md` 또는 요구사항 정의서 경로 제공 요청

### 2.3. 단독 실행 시 파라미터 처리

파라미터 없이 단독 실행된 경우 shackled이 사람에게 순서대로 확인한다:
1. 구현할 작업에 대한 설계서 또는 요구사항 경로 (`context_path`)
2. 체급 판단 (문맥 기반으로 제안, 사람이 확인)
3. 산출물 유형 (`code` / `runbook`)

## 3. 프로세스: 기본 모드 (auto)

### 3.1. 진입

1. `context_path`를 로드하여 컨텍스트를 파악한다.
2. `task_id`가 제공된 경우 중단 지점을 식별하고 재개 흐름(§5)으로 분기한다.
3. `task_id`가 없는 경우 신규 진입으로 처리한다.
4. 이전 구현 상태 파일(`.shackled-state.json`)이 존재하면 사람에게 중단된 작업이 있음을 알리고 재개 여부를 확인한다.

### 3.2. 태스크 파싱

#### 3.2.1. M 체급 (설계서 기반)

1. 설계서에서 구현 태스크 목록을 추출한다.
2. 태스크 간 의존 관계를 파악하고 실행 순서를 결정한다.
3. 추출된 태스크 목록과 순서를 사람에게 제시하고 확인받는다.
4. ★ 사람이 순서를 조정하거나 태스크를 추가/제외할 수 있다.

#### 3.2.2. S 체급 (요구사항 기반)

1. 인터뷰 요약(`00-summary.md`) 또는 요구사항 정의서를 읽는다.
2. 구현에 필요한 작업 단위를 도출한다.
3. 도출된 작업 목록을 사람에게 제시하고 확인받는다.
4. ★ 사람이 확인 또는 조정한다.

### 3.3. 태스크별 순차 실행

태스크 하나씩 순서대로 수행한다. 한 번에 전체 태스크를 구현하지 않는다.

각 태스크에 대해 아래 루프를 반복한다:

#### 3.3.1. 구현

작성자 페르소나가 컨텍스트(설계서 또는 요구사항)를 펼쳐두고 해당 태스크를 구현한다.

- `output_type: code` — 코드 작성·편집
- `output_type: runbook` — 런북 섹션 작성 ([50-runbook-template.md](../../50-runbook-template.md) 형식 준수)

#### 3.3.2. 크로스 체크 (§6 참조)

크로스 체커 페르소나가 구현 결과를 설계서(또는 요구사항)와 대조하여 방향 이탈·누락을 점검한다.

- finding이 없으면 다음 단계로 진행한다.
- finding이 있으면 작성자 페르소나가 수정하고 크로스 체크를 재수행한다.
- 내부 크로스 체크를 통과해야 사람 확인 단계로 넘어간다.

#### 3.3.3. 사람 확인

구현 결과를 사람에게 제시한다. 제시 내용:
- 구현된 내용 요약
- 크로스 체크 결과 (이상 없음 확인 또는 수정 내역)
- 다음 태스크 예고

사람 응답에 따라 분기한다:

| 응답 | 처리 |
|---|---|
| 승인 | 다음 태스크로 진행 |
| 피드백 제공 | 피드백을 반영하여 현재 태스크 재구현 → 크로스 체크 → 재제시 |
| 거부 (전면 재작성 요청) | 구현을 폐기하고 태스크 처음부터 재시작 |
| 진행 불가 보고 | §7 진행 불가 처리 흐름으로 분기 |

### 3.4. 인테그레이션

전체 태스크 완료 후 수행한다.

1. 태스크별 구현 결과물을 통합한다.
2. 전체 통합 결과가 컨텍스트(설계서 또는 요구사항)의 목표와 일치하는지 확인한다.
3. 통합 결과를 사람에게 제시하고 최종 확인받는다.
4. ★ 사람이 승인하면 산출물 반환 단계로 진행한다.

### 3.5. 산출물 반환

구현 결과물과 함께 작업 요약 카드(TASK-{ID}) 초안을 생성한다.

- `output_type: code` — 변경된 코드 파일 목록, 작업 요약 카드
- `output_type: runbook` — 완성된 런북 파일 경로, 작업 요약 카드

단독 실행 시: 사람에게 결과를 반환하고 종료한다.
파이프라인 내 실행 시: 오케스트레이터에게 결과를 반환한다.

## 4. 산출물 유형별 분기

프로세스 골격(§3)은 4-A와 4-B가 공유한다. 분기점은 아래와 같다.

| 구분 | 4-A (code) | 4-B (runbook) |
|---|---|---|
| 대표 작업 | 기능 구현, 버그 수정, 리팩토링 | 인프라 조작, 데이터 마이그레이션, 환경 설정 |
| 구현 산출물 | 소스 코드 파일 | 런북 문서 ([50-runbook-template.md](../../50-runbook-template.md) 형식) |
| 페르소나 조합 | CODE(작성자) + BE/FE(크로스 체커) | SRE(작성자) + SEC(크로스 체커) |
| 크로스 체크 기준 | 설계서의 기능 명세, 인터페이스 정의 | 절차 완결성, 롤백 절차 포함 여부, 검증 커맨드 존재 여부 |
| 구현 후 실행 | 없음 (코드만 작성. 실행은 papillon Phase 5~6) | 런북 확정 후 절차 순서대로 실행 + 검증 |

## 5. 전용 페르소나

shackled은 wtth의 전문가 풀과 별도로 구현 특화 페르소나를 사용한다. 페르소나는 작성자와 크로스 체커 쌍으로 구성된다.

### 5.1. 페르소나 조합 결정

`output_type` 파라미터로 조합을 결정한다.

| output_type | 작성자 | 크로스 체커 |
|---|---|---|
| `code` (백엔드) | shackled-CODE | shackled-BE |
| `code` (프론트엔드) | shackled-CODE | shackled-FE |
| `code` (풀스택) | shackled-CODE | shackled-BE + shackled-FE |
| `runbook` | shackled-SRE | shackled-SEC |

백엔드/프론트엔드/풀스택 판별은 설계서 또는 요구사항 내용을 기반으로 shackled이 판단하고 사람에게 확인받는다.

### 5.2. 각 페르소나 역할

| 페르소나 | 역할 |
|---|---|
| shackled-CODE | 코드 작성자. 설계서의 인터페이스·명세를 기반으로 코드를 작성한다 |
| shackled-BE | 백엔드 크로스 체커. API 설계 준수, 데이터 처리 로직, DB 연동 방식을 점검한다 |
| shackled-FE | 프론트엔드 크로스 체커. UI 명세 준수, 상태 관리, API 연동 방식을 점검한다 |
| shackled-SRE | 런북 작성자. 실행 절차를 단계별로 작성하고 검증 커맨드를 포함한다 |
| shackled-SEC | 보안 크로스 체커. 절차의 보안 위험, 권한 과다 부여, 롤백 절차 누락을 점검한다 |

## 6. 크로스 체크 메커니즘

### 6.1. 기본 원칙

크로스 체커 페르소나는 설계서(또는 요구사항)를 펼쳐두고 구현 결과를 대조한다. "잘 만들었는가"가 아니라 "설계대로 만들었는가"에 집중한다.

### 6.2. 체크 항목 (code)

1. 설계서에 명시된 인터페이스(API, 함수 시그니처 등)와 구현이 일치하는가
2. 설계서에 명시된 데이터 흐름·처리 로직이 구현에 반영되었는가
3. 설계서에 명시된 제약 조건(인증, 권한, 유효성 검사 등)이 누락되지 않았는가
4. 설계서에서 정의된 에러 처리 방식이 반영되었는가
5. 이 태스크가 연관된 다른 태스크와의 인터페이스가 충돌하지 않는가

### 6.3. 체크 항목 (runbook)

1. 설계서(또는 요구사항)에서 정의한 작업 목표가 절차에 반영되었는가
2. 각 단계에 검증 커맨드 또는 성공 판단 기준이 포함되어 있는가
3. 롤백 절차가 존재하는가
4. 실행 전제 조건(권한, 환경, 의존 서비스 상태 등)이 명시되어 있는가
5. 단계 순서가 실행 가능한 순서인가 (의존 관계 위반 없음)

### 6.4. finding 처리

크로스 체크 finding은 shackled 내부에서 처리한다. 사람에게 노출하지 않는다.

finding이 있으면:
1. 크로스 체커가 구체적으로 지적한다: "§2.3에서 요구한 X 처리가 누락되었습니다"
2. 작성자 페르소나가 수정한다.
3. 크로스 체커가 재점검한다.
4. finding이 없어지면 사람 확인 단계로 진행한다.

동일 항목에 대해 3회 수정 후에도 해소되지 않으면 진행 불가 처리(§7)로 분기한다.

## 7. 진행 불가 처리

### 7.1. 진행 불가 감지

shackled이 다음 중 하나에 해당하면 구현을 멈추고 진행 불가를 사람에게 보고한다:

| 감지 조건 | 설명 |
|---|---|
| 기술적 막힘 | 구현에 필요한 기술적 결정이 설계서에 없거나 모순된 경우 |
| 컨텍스트 부족 | 구현에 필요한 정보(API 명세, 스키마, 환경 변수 등)가 없는 경우 |
| 설계 불일치 | 설계서의 내용이 실제 구현과 충돌하여 어느 방향으로 구현해도 설계를 위반하는 경우 |
| 내부 크로스 체크 반복 실패 | §6.4의 3회 수정 후 finding 미해소 |

### 7.2. 보고 형식

진행 불가 시 사람에게 다음을 제시한다:

```
[진행 불가] 태스크 {task_id}: {태스크 제목}

막힌 이유: {구체적인 원인}
원인 추적 힌트: {어느 설계 결정에서 비롯된 것인지, 어느 문맥 파일이 관련있는지}
현재까지 완료된 태스크: {목록}
미완료 태스크: {목록}
```

### 7.3. 사람 결정 분기

보고를 받은 사람이 원인 수준에 따라 결정한다. shackled은 원인 힌트만 제공하며 롤백 지점 결정은 사람의 몫이다.

| 원인 수준 | 사람 결정 | 처리 |
|---|---|---|
| 경미 | shackled 내에서 해결 | 사람이 방향을 지시하고 shackled이 재개 |
| 중간 | 설계 수정 후 재진입 | shackled 종료 → 설계 수정 → 리뷰 재진입 → shackled 재호출 |
| 심각 | inquisition 롤백 | shackled 종료 → inquisition 재진입 |

### 7.4. 상태 저장 후 종료

진행 불가로 shackled이 종료될 때 현재 상태를 저장한다. 이후 재개 시 `task_id`를 통해 이어받는다.

---

## 8. 중단 및 재개

### 8.1. 상태 파일

shackled은 진행 중 상태를 아래 파일에 저장한다:

```
{프로젝트}/03-tasks/.shackled-state.json
```

```json
{
  "context_path": "...",
  "weight": "M",
  "output_type": "code",
  "exec_mode": "auto",
  "tasks": [
    { "id": "TASK-A-01", "title": "...", "status": "completed" },
    { "id": "TASK-A-02", "title": "...", "status": "in-progress" },
    { "id": "TASK-A-03", "title": "...", "status": "pending" }
  ],
  "last_updated": "YYYY-MM-DD"
}
```

### 8.2. 재개 흐름

`task_id`가 제공되거나 상태 파일이 발견된 경우:

1. 상태 파일을 로드하여 완료/미완료 태스크 현황을 파악한다.
2. 사람에게 현재 상태를 제시한다 (완료 N건, 진행 중 1건, 미완료 N건).
3. 재개 지점을 확인한다:
   - `in-progress` 상태 태스크부터 재개 (기본)
   - 사람이 다른 지점을 지정하면 그대로 따른다.
4. 이전 컨텍스트를 다시 로드하고 지정된 태스크부터 §3.3 루프를 재개한다.

## 9. 확장 포인트: 페어 모드 (pair)

### 9.1. 인터페이스 예약

`exec_mode: pair`로 호출된 경우, shackled은 현재 페어 모드를 지원하지 않음을 안내하고 기본 모드(auto)로 진행하거나 종료할지 사람에게 선택지를 제공한다.

### 9.2. 설계 방향 (상세 설계 미포함)

페어 모드는 아래 개념을 중심으로 향후 설계될 예정이다:

- 코딩 전문가가 별도 pane에서 AS-IS/TO-BE와 방향을 설명
- 사람이 실시간으로 피드백하고 반복
- 실행 전제: tmux 환경 + 실험 플래그 활성화
- JetBrains MCP 한계 및 tmux 연동 방식은 별도 설계 필요

페어 모드 전담 설계 파일 경로 (미생성): `01-design-shackled-pair.md`
