# AI-2: 수렴 판정 강화 — Sign-off ID Cross-Referencing

**심각도:** P1
**분류:** 공백
**상태:** Closed — wtth §3.3(백업 생명주기 조정) + §4.1(인용 강제 + MCP 문자열 일치 + P0 전수/P1 샘플링) 반영 완료. P1 샘플링 비율 캘리브레이션은 풀사이클 실험 데이터 축적 후 튜닝(후속)
**관련:** [Issue-03-02](../03-Issue-03/02-convergence-verification-gap.md) — 수렴 판정 조건 반영 검증 미포함 (본 건이 구체적 솔루션)

## 문제

wtth 수렴 전환 조건이 "신규 P0/P1 0건"이고, 사인오프에서 경량 모델이 "추가 지적 없음"을 확인하지만, 이전 라운드에서 수용된 사항이 실제로 반영되었는지의 물리적 검증이 없다. 같은 컨텍스트를 공유하는 AI가 확증 편향으로 "당연히 반영됐겠지"하고 넘어갈 수 있다.

## 솔루션: P0/P1 수용 건 인용 강제 + MCP 문자열 일치 검증

- **수정 대상:** `00-core.md` §4.1 사인오프 모드
- **변경 내용:**
  1. 사인오프 에이전트는 단순히 "추가 지적 없음"으로 끝낼 수 없다
  2. 이전 라운드 RDR에서 **수용된 P0/P1 건**에 대해, 각 finding ID별로 반영 지점을 `[파일:위치:해당 구문]` 형태로 **인용을 강제**한다
  3. 오케스트레이터가 MCP를 통해 해당 파일의 해당 위치를 읽어, 인용된 구문과 **단순 문자열 일치 검사**를 수행한다
  4. 불일치 또는 인용 누락 시: 사인오프를 거부하고 수렴 모드로 롤백
- **적용 범위:** P0/P1 수용 건만. P2/P3은 기존 방식 유지 (토큰 비용 통제)
- **비용:** 문자열 일치는 LLM 추론이 아닌 단순 로직이므로 비용 거의 제로. 인용 생성은 사인오프 에이전트의 기존 토큰 버짓 내에서 수행 가능.

<rationale>
외부 도구(git/diff/linter)에 의존하지 않고 도구 독립성 원칙(§4.1.1)을 유지하면서, AI의 확증 편향을 물리적으로 차단하는 방법이다. MCP를 통한 파일 읽기 + 문자열 비교는 외부 바이너리 의존이 아니라 프로토콜 계층의 데이터 접근이므로 원칙과 충돌하지 않는다.
</rationale>

## 보완: Sign-off Sampling + 백업 생명주기 조정 (P1, Round 2)

AI-2의 `[File:Line:Snippet]` 인용 강제는 "수정 구문이 존재한다"는 것만 확인하고, "올바르게 교체되었는가"는 검증하지 못한다. AS-IS/TO-BE 전수 검증은 토큰 비용이 2배. `.pre-review.md` 백업이 리뷰 종료 시 삭제(§3.3 기본값)되어 사인오프 시점에 원본이 없을 수 있다.

**추가 변경:**
1. `.pre-review.md` 삭제 시점을 "리뷰 종료 시"에서 **"사인오프 검증 완료 후"**로 변경. 사인오프에서 AS-IS 참조가 필요하므로.
2. **P0 건**: AS-IS → TO-BE 전수 검증. 오케스트레이터가 `.pre-review.md`에서 AS-IS를, 현재 파일에서 TO-BE를 읽어 교체 발생을 문자열 비교로 확인
3. **P1 건**: 최소 1건 이상 무작위 추출하여 동일 검증. 샘플링 비율은 풀사이클 실험 데이터로 캘리브레이션
4. 불일치 또는 인용 누락 시: 사인오프 거부 → 수렴 모드로 롤백

<rationale>
P0/P1 차등 검증 이유: P0은 데이터 손실/보안 침해급이라 전수 확인이 필수. P1은 건수가 많을 수 있어 전수 검증의 비용이 과다. 샘플링이라도 "검증될 수 있다"는 긴장감이 AI의 성실성을 유도한다.
</rationale>
