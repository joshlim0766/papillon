# 02-스킬 아키텍처 및 wtth 관계

**상태:** Settled

## 확인된 사항
- 4-A(코드)와 4-B(인프라/절차)를 하나의 스킬로 통합
  - 이유: 프로세스 골격 동일 (단계적 구현 → 리뷰 → 수정 반복 → 완성). 산출물 형태만 다름 (코드 vs 런북)
  - 산출물 유형에 따라 내부 분기
- 시블링 구조: papillon 오케스트레이터가 shackled, wtth을 각각 호출
  - shackled은 구현만 하고 산출물 반환
  - wtth 호출은 오케스트레이터 책임
  - 단독 실행 시 구현 결과만 반환, 사람이 필요하면 /papillon:wtth 직접 호출
- 리뷰 계층화 (중복 아님):
  - shackled 내부: "설계대로 구현했나?" — 방향 이탈/누락 점검 (자기 검수)
  - wtth: "잘 만들었나?" — 품질, 안전성, 다각도 점검 (제3자 검수)
