## AI-1. 설계+운영절차 복합 적용 시 SPEC/IX 투입 가이드

**우선순위:** 높음

**배경:** shackled 설계 리뷰에서 기존 5모드(PRD/설계/ADR/운영절차/코드) 중 어느 것도 워크플로우·스킬 명세를 적절히 커버하지 못했다. 별도 `review-workflow` 모드 신설을 검토했으나, 워크플로우 설계서는 "설계"와 "운영 절차"의 성격이 혼재되어 있어 기존 두 모드를 복합 적용하고 전문가 풀만 커스텀하는 것이 현실적이라는 결론에 도달. 별도 모드를 만들면 사용 빈도 대비 관리 비용이 높아짐.

**결론 과정:**
- 초기: "review-workflow 모드가 필요하다" (O1 평가 초안)
- 회고 시: ARCH의 finding은 설계 리뷰 성격, IX의 finding은 운영 절차 리뷰 성격 → 두 모드의 중간 지점
- 최종: 별도 모드 불필요, 기존 모드 조합 + 전문가 풀 커스텀으로 충분

**수정 대상:**
- `04-wtth/02-design/00-core.md` §1 — 복합 모드 적용 가이드 추가. "워크플로우·스킬 명세 리뷰 시 설계 리뷰 + 운영 절차 리뷰를 복합 적용하고, 전문가 풀을 ARCH + SPEC + IX로 구성"
- 동 문서 §1.1 모드 선택 규칙 — 복합 적용 판별 기준 추가

**검증 기준:**
- 워크플로우 설계서를 복합 모드로 리뷰했을 때, 이번 shackled 리뷰와 유사한 범위의 finding이 도출되는가
- 기존 단일 모드 리뷰가 영향받지 않는가

## AI-2. SPEC/IX 페르소나 정식 등록 + 투입 조건

**우선순위:** 높음

**배경:** shackled 리뷰를 위해 SPEC(명세 분석가)과 IX(인터랙션 설계자)를 신규 정의했다. SPEC은 P0 3건 포함 17건, IX는 P0 1건 포함 10건을 도출하여 유효성이 확인됨. 금기 설정으로 역할 경계도 명확. 다만 투입 조건 없이 정식 등록하면 "모든 리뷰에 SPEC/IX를 끼워넣으려는" 경향이 생길 수 있음.

**결론 과정:**
- SPEC↔IX 경계: SPEC이 "정보가 있는가", IX가 "적시에 제시되는가" — 중복 없음
- ARCH↔SPEC/IX: 동일 주제에서 보완적 finding → 묶음 처리로 해결
- 투입 조건 필요성: PoC에 SEC/PM 넣으면 맞는 말이지만 노이즈인 finding 폭발 → 전문가 비대화 방지 필요

**수정 대상:**
- `01-Planning/personas/spec.md` — 현재 파일에 `투입 조건` 필드 추가: "워크플로우·프로세스 명세가 리뷰 대상의 핵심일 때만 투입"
- `01-Planning/personas/ix.md` — 현재 파일에 `투입 조건` 필드 추가: "사람-AI 상호작용이 리뷰 대상에 포함될 때만 투입"
- `01-Planning/personas/_schema.md` — 투입 조건 필드가 DBA 외 다른 페르소나에도 적용 가능함을 반영

**검증 기준:**
- 투입 조건이 명시된 상태에서 오케스트레이터가 일반 설계 리뷰에 SPEC/IX를 추천하지 않는가
- 워크플로우 리뷰 시에는 정상적으로 투입을 추천하는가

## AI-2-1. 전문가 풀 제한 메커니즘

**우선순위:** 높음

**배경:** 전문가 페르소나가 늘수록 "이 리뷰에도 저 전문가 넣으면 좋지 않을까?" 압력이 생김. 실제로 모든 리뷰가 6-7인 풀세트가 되면 토큰 비용과 사람의 finding 처리 부담이 폭발. PoC 레벨에서 SEC/PM을 투입하면 finding은 맞는 말이지만 그 맥락에서는 순수 노이즈.

**결론 과정:**
- 방안 1: 기본 풀 상한 고정 (3인) → 이미 PRD(3인), ADR(2인), 운영절차(2인)에 선례 있음. 설계 리뷰의 6인이 오히려 예외
- 방안 2: 투입 조건을 페르소나 파일에 명시 → DBA 선례 ("DDL 포함 설계에만")
- 방안 3: 토큰 예산 개념 → 과격, 보류
- 최종: 방안 1+2 조합 + 사람 제외 가능 + 리뷰 세션 내 재추천 금지

**수정 대상:**
- `04-wtth/02-design/00-core.md` §1 — 모드별 기본 풀 상한 3인 원칙 추가
- 동 문서 — 리뷰 상태에 `excluded_experts` 필드 추가. 사람이 제외한 전문가를 해당 리뷰 세션(RDR 1개 단위) 동안 유지
- 동 문서 — "사람이 전문가를 제외하면 즉시 반영, 해당 리뷰 내 재추천 금지" 규칙 추가

**검증 기준:**
- 기본 풀 3인 초과 추천 시 사람 확인이 필요한가
- 사람이 제외한 전문가가 같은 리뷰 세션에서 다시 추천되지 않는가
- 클로드 세션이 아닌 리뷰 세션(RDR 단위) 기준으로 제외가 유지되는가

## AI-3. 라운드 간 비교 원칙

**우선순위:** 높음

**배경:** shackled 리뷰 2라운드 진입 시 "라운드 간 diff를 어떻게 할 것인가" 논의 발생. git diff → git 미초기화 시 불가. OS diff/patch → OS 종속성. 설계서는 코드와 달리 구조 변경 시 텍스트 diff 정보 품질이 낮음 (전부 삭제+전부 추가로 표시).

**결론 과정:**
- A. git diff 기반 → git 의존, 수정량 많으면 전체 파일과 같음
- B. 최신 문서 + finding 결정 목록 전달 → 도구 무관, 정보 품질 높음
- 최종: B 방식. 라운드 간 비교는 finding 결정 목록이 담당, 외부 도구 의존 제로

**수정 대상:**
- `04-wtth/02-design/00-core.md` §4 수렴 메커니즘 — 라운드 간 상태 전달 원칙 추가:
  1. 라운드 N 에이전트에게 전달: 최신 문서 원본 + 이전 라운드 finding 결정 목록
  2. "결정 목록의 건은 재언급 금지" 지시
  3. git/diff 등 외부 도구에 의존하지 않음

**검증 기준:**
- git 미초기화 프로젝트에서도 2라운드 이상 리뷰가 정상 작동하는가
- 2라운드 에이전트가 1라운드에서 수용된 건을 재지적하지 않는가

## AI-5. inquisition 핸드오프 브리프

**우선순위:** 높음

**배경:** shackled 초안(Sonnet 작성)에서 P0 4건 + P1 15건이 발생. 원인 분석 결과 inquisition의 역할(수집·정리)은 잘 수행되었으나, Sonnet에게 맥락 기반 주의 사항이 전달되지 않아 세부 명세 공백이 발생. 인터뷰 산출물 대조 결과, 크로스 체크(1줄)와 상태 관리(1줄)처럼 깊이가 부족한 관심사에서 finding이 집중됨.

**결론 과정:**
- inquisition이 더 세밀하게 질문? → 역할 범위 벗어남
- Sonnet에게 범용 체크리스트? → 매번 다른 설계에 맞지 않음
- inquisition(Opus)이 핸드오프 브리프 생성 → 이미 맥락을 가장 깊이 이해, 체급 판단과 유사 수준의 추론, 비용 추가 미미
- 핸드오프 브리프 포함 내용:
  1. 맥락 기반 주의 사항 (예: "상태 전이가 많으니 status enum 빠짐없이 정의")
  2. 깊이 부족 플래그 (예: "크로스 체크: 언급됨, 상세 미정의. Sonnet 자체 설계 필요")
  3. 신뢰도 플래그 (인터뷰이 답변 일관성 평가)

**수정 대상:**
- `05-inquisition/02-design/00-design-inquisition.md` — 체급 판단 단계에 핸드오프 브리프 생성 추가
  - 인터뷰 종료 → 체급 판단 → 핸드오프 브리프 생성 → Sonnet에게 전달
  - 브리프 포맷 정의: 주의 사항 / 깊이 부족 관심사 / 신뢰도
- 동 문서 — 셀프 오딧 항목에 "관심사별 깊이 균형 확인" 추가

**검증 기준:**
- 핸드오프 브리프가 있는 Sonnet 초안 vs 없는 초안의 finding 수 비교 (줄어드는가)
- 깊이 부족 플래그가 표시된 관심사에서 Sonnet이 "자체 설계 필요"를 인지하고 더 상세하게 작성하는가
- 신뢰도 플래그가 낮을 때 사람이 추가 인터뷰를 결정할 수 있는가

## AI-6. wtth 리뷰 토큰 비용 최적화

**우선순위:** 높음

**배경:** shackled 리뷰에서 3인 에이전트 × 2라운드 + 사인오프 3인 = 총 9회 에이전트 호출. 각 에이전트가 설계서 전문 + 참조 문서 2~3개를 매번 읽어 토큰 소모가 큼. 특히 2라운드에서 수정된 섹션은 일부인데 전체를 다시 읽음. 사인오프는 "없음" 한 줄을 받기 위해 Opus급 에이전트 3개를 돌림.

**결론 과정:**
- 1라운드 발산은 전체 문서를 읽어야 하므로 줄일 수 없음
- 2라운드 이후: 변경 섹션만 전달 가능 → 토큰 30~40% 절감 예상
- 사인오프: P0/P1 유무만 확인하면 되므로 경량 모델(Haiku/Sonnet) 충분
- 조기 수렴: 이번에 3라운드 스킵한 ROI 판단을 공식 기준으로

**수정 대상:**
- `04-wtth/02-design/00-core.md` §4 수렴 메커니즘:
  - 2라운드 이후 에이전트 프롬프트에 "변경된 섹션"을 명시하고 해당 섹션 집중 리뷰 지시
  - 사인오프 모드에서 경량 모델 사용 허용
  - 조기 수렴 전환 기준 추가: "P0 0건 + P1 건수 직전 라운드 대비 50% 이상 감소 → 수렴 전환 가능" (사람 승인 필요)
- 동 문서 §4.2 — 1라운드 분석 시 예상 토큰 규모감 제시 (정보 제공 톤)

**검증 기준:**
- 2라운드에서 변경 섹션만 전달했을 때 finding 품질이 전체 문서 전달과 동등한가
- 사인오프에서 경량 모델이 P0/P1을 놓치지 않는가
- 조기 수렴 기준 적용 시 실제로 불필요한 라운드가 줄어드는가

## AI-7. 관심사 분산도 기반 분할 판단 (이중 안전망)

**우선순위:** 높음

**배경:** shackled은 M 체급 단일 태스크로 설계되었으나 1라운드에서 37건이 7개 독립 관심사에 분산. 이는 설계 대상이 단일 M에 담기엔 넓다는 신호였으나, 사람(finding 처리에 집중)도 오케스트레이터(건별 제시에 집중)도 감지 못함. inquisition은 수집 역할이라 판단 불가, Sonnet은 능력 부족. 1라운드 분석이 유일한 개입 지점이나 현재 wtth §4.2는 관심사 분산도를 보지 않음.

추가로, 초안 완료 시점에서도 감지 가능했음 — 500줄 + 7개 섹션이라는 구조적 신호가 있었으나 사람이 "go"를 외치고 넘어감. 리뷰 토큰을 쓰기 전에 알 수 있었음.

**결론 과정:**
- 개입 지점 1 (초안 완료 시): Sonnet이 관심사 분포를 요약하고 예상 규모감 제시. 정보 제공 톤 — "분할을 권합니다"가 아니라 "참고로 이 규모입니다". 비용 거의 없음 (구조만 세면 됨)
- 개입 지점 2 (1라운드 분석 시): finding을 관심사별 group by → 분산도 분석 → "관심사 N개 이상에서 P0/P1 분산 → 분할 재검토 권장". 선택지 제시: 분할 / 유지 / 범위 축소
- 이중 안전망이 필요한 이유: 개입 지점 1에서 사람이 무시해도 개입 지점 2에서 finding 데이터 기반으로 재확인

**수정 대상:**
- `04-wtth/02-design/00-core.md` §4.2 1라운드 결과 분석 — 분석 관점 테이블에 "관심사 분산" 행 추가:
  - 관심사별 finding group by
  - 관심사 3개 이상에서 P0/P1 분산 → 분할 재검토 제안
  - 선택지: 관심사별 분할 / 현재 구조 유지 / 범위 축소
- `03-papillon/02-design/00-design-papillon.md` 또는 shackled/inquisition 설계 — 초안 완료 시 관심사 분포 요약 출력 규칙 추가
- 예상 규모감 제시 형식 정의: 관심사 수, 예상 finding 범위, 예상 토큰

**검증 기준:**
- 관심사 7개 이상 설계서에서 분할 제안이 발생하는가
- 관심사 2~3개 설계서에서는 불필요한 분할 제안이 발생하지 않는가
- 사람이 "유지"를 선택해도 이후 finding 분포로 재확인이 되는가

## AI-8. 리뷰 중간 결과물 처리 방안

**우선순위:** 중간

**배경:** 리뷰 과정에서 생성되는 중간 결과물(에이전트 raw 결과, finding 논의 과정, 설계서 수정 전 원본, 중간 상태)이 세션 종료 시 모두 소멸. 최종 산출물(RDR, 설계서, 평가란)만으로 "무엇이 결정되었는가"는 알 수 있으나, "왜 그렇게 결정했는가"의 상세 맥락(대안 비교, 논의 과정)은 복원 불가. 모두 보존할 필요는 없으나 두 가지는 가치 있음.

**결론 과정:**
- 전부 보존 → 과도, 대부분 최종 산출물에 녹아듦
- git 기반 스냅샷 → OS/도구 종속 (이번 리뷰에서 확립한 원칙과 충돌)
- `.pre-review.md` 백업 → 도구 무관, 순수 파일 복사
- RDR 사유 확장 → 대안 비교가 있었던 건만 선별적

**수정 대상:**
- `04-wtth/02-design/00-core.md` — 리뷰 시작 시 대상 파일 `.pre-review.md` 백업 규칙 추가. 리뷰 종료 후 사람에게 보존/삭제 확인
- `02-common-spec.md` §2 RDR 포맷 — "주요 기각 사유" 섹션에 "대안 비교가 있었던 수용 건의 상세 사유" 기록 가이드 추가 (선택적)

**검증 기준:**
- 리뷰 시작 시 `.pre-review.md` 파일이 자동 생성되는가
- 리뷰 종료 후 보존/삭제 확인이 사람에게 제시되는가
- RDR만으로 핵심 결정의 대안 비교 과정을 복원할 수 있는가

## AI-9. 4-B 런북 실행 전담 스킬 설계 검토

**우선순위:** 낮음

**배경:** 1라운드 ARCH-06에서 shackled이 런북 생성과 실행을 모두 담당하는 것은 관심사 분리 위반이라는 지적. 초기에는 "MVP에서 유지하되 분리 경계만 명시"로 수용했으나, 사람이 "아예 나누자. 생성하고 이어서 실행하면 너무 위험하다"로 결정. 런북 실행은 인프라에 부작용을 일으키는 행위이므로 생성과 동일 스킬에서 연속 수행 시 사람 개입 없이 실행될 위험.

**결론 과정:**
- 분리 경계만 명시 (초기) → MVP에서도 분리 (사람 판단) → papillon §3.4도 동시 수정
- 현재: shackled은 런북 생성까지만, 실행은 별도 스킬/오케스트레이터 책임
- 향후: 런북 실행 전담 스킬이 필요한지, 오케스트레이터가 직접 실행하는 게 맞는지 검토 필요

**수정 대상:**
- 신규 설계 문서 — 런북 실행 전담 스킬 또는 오케스트레이터 실행 메커니즘
- `03-papillon/02-design/00-design-papillon.md` §3.4 — 런북 실행 주체 확정 후 반영

**검증 기준:**
- 런북 생성 후 실행까지 사람 확인 없이 자동 연결되지 않는가
- 실행 중 실패 시 롤백 흐름이 정의되어 있는가
- 실행 상태 추적이 가능한가

## 오케스트레이터 주제별 묶음 처리 (O4 회고에서 도출, O6 미등록)

**배경:** ARCH↔SPEC finding이 동일 주제에서 빈번하게 겹침 (시각만 다름). 사람 입장에서 같은 주제가 2~3건으로 등장하여 finding 건수가 부풀려 보이고 읽는 부담 증가. 이번 리뷰에서 수동으로 묶음 제시했을 때 의사결정이 수월했음.

**수정 대상:**
- `04-wtth/02-design/00-core.md` §2 의사결정 방식 — 동일 주제 finding 묶음 처리를 정식 메커니즘으로 추가. 현재 "동일 주제에 대한 보완적 finding은 주제별로 묶어서 제시"라는 문구는 있으나, 묶음 판단 기준과 제시 형식이 미정의.

**검증 기준:**
- 3인 이상 전문가가 동일 주제를 지적했을 때 자동으로 묶여서 제시되는가
- 묶음 내에서 각 전문가의 관점 차이가 명확히 구분되는가
