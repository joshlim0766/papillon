## 1. 코드 리뷰

### 1.1. 대상

diff, 소스 파일

### 1.2. 전문가 풀

| 기본 풀 | 선택적 투입 |
|---|---|
| CODE, TEST | ARCH, SEC (단독 실행 시 오케스트레이터가 제안 가능) |

### 1.3. 의사결정 방식

AI 자동 수정 + diff 승인/반려/직접수정 (코어 [2.2](./00-core.md#22-ai-자동-수정--diff-승인반려-코드-리뷰) 참조)

### 1.4. 모드 특화 사항

- **papillon 경유 시**: Phase 4-A 완료 후 자동으로 대상 결정
- **단독 실행 시**: 오케스트레이터가 `$ARGUMENTS`를 분석하여 대상을 결정한다. 인자가 없으면 사람에게 아래 중 선택을 요청:
  - staged 변경사항 (`git diff --cached`)
  - 특정 브랜치 대비 변경사항 (`git diff <branch>...HEAD`)
  - 특정 커밋 범위 (`git diff <commit1>..<commit2>`)
- AI 전문가가 지적 + 자동 수정 수행
- 오케스트레이터가 finding 간 코드 영향 범위를 분석하여 diff 단위 결정 (코어 [2.2](./00-core.md#22-ai-자동-수정--diff-승인반려-코드-리뷰) 상세 참조)

- **절차서**: 마이그레이션·권한 변경·인프라 작업 등 일회성 또는 비정기 운영 절차. 런북 규격을 따르되, 반복 실행 전제 항목(스케줄, 트리거 조건 등)은 생략 가능

## 3. 공통 프로세스와의 차이점

### 코드 리뷰

#### 스킵 단계
- **ADR 승격 없음**: 코드 리뷰에서는 아키텍처 결정이 아닌 구현 품질을 검증하므로, 코어 §3.2 ADR 승격 규칙을 적용하지 않는다.
- **수정 경로 리뷰 해당 없음**: 코드 리뷰는 diff 대상이며 코어 §2.3의 수정 매니페스트 기반 프로세스를 사용하지 않는다.

#### 변형 단계
- **의사결정 방식 전면 변경**: finding별 수용/기각(코어 §2.1) 대신, AI 자동 수정 + diff 승인/반려(코어 §2.2)를 사용한다. 이것이 코드 리뷰와 다른 모드의 가장 큰 차이다.
- **라운드 정의 변경**: 발산/수렴/사인오프의 라운드는 "finding → diff → 전부 처리" 단위다. 수렴 전환 조건은 동일하나, 처리 단위가 diff다.
- **묶음 처리 변형**: 코어 §2.1.1의 관심사별 묶음 대신, 코드 영향 범위별 묶음(같은 파일/코드 블록을 건드리는 finding)으로 diff 단위를 결정한다.

#### 추가 규칙
- **동일 diff 반려 3회 경고**: 코어에 없는 코드 리뷰 전용 규칙 (코어 §2.2 참조).
- **diff 기반 순차 제시**: 심각도 순 정렬 후, diff 단위로 사람에게 순차 제시. 승인된 diff 기반으로 다음 diff를 수정한다 — 선행 diff의 변경이 후행 diff에 영향을 줄 수 있으므로 병렬 처리하지 않는다.

### 운영 절차 리뷰

#### 스킵 단계
- **ADR 승격 없음**: 운영 절차는 아키텍처 결정이 아니므로 코어 §3.2를 적용하지 않는다.

#### 변형 단계
- 없음. 코어 §2.1(finding별 수용/기각)을 그대로 사용한다.

#### 추가 규칙
- **런북 규격 적용**: 런북은 [50-runbook-template.md](../../50-runbook-template.md) 규격을 따른다. 규격 미준수 자체가 finding이 될 수 있다.
- **선택적 투입 조건**: DBA는 DB 변경 포함 시, SPEC은 워크플로우·절차 명세 복잡 시, IX는 사람 확인·분기가 포함된 런북에서 투입 제안한다.

<rationale>
코드 리뷰의 의사결정 방식을 diff 기반으로 별도 설계한 이유: 설계 문서는 "finding → 사람 판단 → 수정 반영"이 자연스럽지만, 코드는 "finding + 수정안을 diff로 즉시 보여줌"이 더 효율적이다. 사람이 텍스트 수정안을 읽고 머릿속으로 적용할 필요 없이, AS-IS/TO-BE를 바로 비교할 수 있다.
</rationale>
