# 5-2: 에러/실패 처리 전략 부재

**심각도:** 높음
**분류:** 누락
**상태:** Closed

## 문제

Phase별 정상 흐름은 기술되어 있으나, 비정상 상황 처리가 Phase 4의 "진행 불가 시"에만 부분적으로 있다. 누락된 케이스:

- **wtth 호출 실패 시** (Phase 2, 5): 리뷰 엔진이 비정상 종료되면 어떻게 하는가?
- **세션 끊김**: 컨텍스트 스캔에서 `00-interview/` 재개는 있지만, Phase 2~5 중간 세션 끊김 시 재개 방법이 없다.

shackled 인터뷰 피드백에서도 "진행 불가/롤백 논의가 핵심 관심사"로 기록되어 있는데, 오케스트레이터 수준 반영이 미흡.

## 논의

PRD 재리뷰에서도 관련 건 발견 (M-1). PRD 수준에서도 에러 처리 원칙을 한 줄이라도 명시하면 좋다는 지적.

## 처리

두 가지 조치:

**A. 하위 스킬 호출 실패 공통 원칙 추가 (§3 도입부)**
- 하위 스킬(inquisition, wtth, shackled) 비정상 종료 시 → 오케스트레이터가 사람에게 상황 알림 → 사람이 재실행/스킵 결정
- Phase 4의 shackled 패턴을 전체 스킬에 일반화

**B. 컨텍스트 스캔 테이블 확장 (§3.0)**
- 별도 진행 로그 없이 산출물 기반으로 재개 지점 추론
- 추가된 케이스: 설계서+RDR 조합으로 Phase 2/4/5 진입 전 끊김 감지
- 근거: 사람의 의사결정은 이미 RDR/ADR/interview 파일로 즉시 기록되는 구조이므로, 산출물 존재 여부만으로 Phase 단위 재개 가능. Phase 내부 재개는 해당 Phase 재시작으로 처리 (bounded cost)

## 결과

papillon 설계서 §3 도입부에 스킬 호출 실패 원칙 추가, §3.0 컨텍스트 스캔 테이블에 Phase 2~5 중간 끊김 재개 케이스 3건 추가.
