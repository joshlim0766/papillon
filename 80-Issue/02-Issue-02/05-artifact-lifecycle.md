# 5: 산출물 생태계의 생명주기·정합성 검증 부재

**심각도:** 중간
**분류:** 공백
**상태:** Open

## 문제

프로젝트 하나에서 생성되는 문서 유형이 9종 이상이다:
1. RDR (리뷰 결정 기록)
2. ADR (아키텍처 결정 기록)
3. 작업 요약 카드
4. Phase 요약 카드
5. 상태 파일 (.shackled-state.json)
6. 문맥 파일 (인터뷰)
7. 핸드오프 브리프
8. 런북
9. 피드백 파일

이것들의 생명주기(생성-갱신-폐기)와 상호 참조 관계를 누가 관리하는가?

papillon 오케스트레이터가 "다음 세션에서 index를 읽고"라고 하지만, index가 참조하는 RDR/ADR/태스크 카드가 정합성을 유지하는지 검증하는 메커니즘이 없다.

## 왜 문제인가

1인 운영이라 당장은 괜찮지만, 이 시스템의 가치를 입증해서 팀에 확장하려면 이 관리 비용이 장벽이 된다.

## 제안된 솔루션: 산출물 생명주기 매트릭스

common-spec에 산출물별 생명주기 테이블 추가:

| 산출물 | 생성 주체 | 갱신 시점 | 소비 주체 | 폐기 조건 |
|---|---|---|---|---|
| RDR | wtth | 리뷰 종료 시 | 다음 리뷰 (이전 결정 스캔) | 아카이브 (삭제 안 함) |
| .shackled-state.json | shackled | status 전이마다 | shackled 재개 | 구현 완료 후 ? |
| 00-interview/ | inquisition | 인터뷰 중 | Draft 생성, 핸드오프 브리프 | 설계 확정 후 ? |
| ... | ... | ... | ... | ... |

특히 중요한 건 **폐기 조건**. 현재 미정의.

**적용 시점:** 파이프라인 1~2회 더 돌린 후. 1인 운영이면 "전부 남겨두면 된다"가 통하지만, 팀 확장을 고려하면 잡아두는 게 좋음.
