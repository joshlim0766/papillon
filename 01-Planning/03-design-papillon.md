## 1. 전체 파이프라인

```
사람: 요구사항 전달 + 파이프라인 시작
  │
  ▼
[Phase 1] 인터뷰 + 체급 판정 + 초안
  │  Step 1: 구조화 인터뷰 (Sonnet)

  │    - 인터뷰 결과 + doc-standard 기준으로 설계 초안 작성
  │    - 핵심 제약은 index에 명시적으로 기재
  │
  ▼ (자동 전환, M 체급)

  │  초안 기반으로 판단: 문서 길이 A4 2장 초과? API 수 3개 초과?

  │

  │  wtth 호출 (설계 전문가 풀)

  │  ★ 사람: finding별 수용/기각
  │  → RDR 자동 생성
  │  → ADR 승격 대상 있으면:
  │     [Phase 2.1] ADR 작성 → wtth 호출 → ★ 사람: 의사결정 → RDR 생성
  │       → 승인: ADR 확정
  │       → 방향 기각: ADR 재작성 → Phase 2.1 재진입
  │       → 불필요 판정: ADR 드롭, 구현에서 처리
  │
  ▼ (자동 전환)
[Phase 3] 종결 판단
  │  Agent S가 미해결 P0/P1 기준으로 종결 가능 여부 제안
  │  ★ 사람: 승인 → Phase 4 / 거부 → Phase 2 반복
  │
  ▼ (자동 전환)

  │  확정된 설계서 기반 코딩 수행

  │  4-B. 인프라/수행 작업: 런북 생성 → wtth(운영 절차 리뷰) → ★ 사람 의사결정 → 실행 → 검증

  ▼ (자동 전환, 4-A인 경우)
```

**★ = 사람이 개입하는 유일한 지점**

## 2. Phase별 상세

### 2.1. Phase 1: 인터뷰 + 체급 판정 + 초안

#### Step 1: 구조화 인터뷰 (Sonnet)

1. 사람이 요구사항을 자연어로 전달한다.
2. Sonnet이 요구사항을 분석하여 다음을 수행한다:
   - 기술 키워드, 도메인 개념 추출
   - 각 키워드 뒤에 숨겨진 함의(implication)와 암시적 제약 후보 감지
   - 기술적 결정의 의도, 실행 순서 제약, 핵심 전제 조건에 대해 사람에게 확인 질문
3. 사람이 답변하면, 답변 내용에서 추가 확인이 필요한 부분을 재질문한다.
4. **매 답변 기록**: 사람의 답변이 올 때마다 확인된 내용을 **인터뷰 기록 파일에 즉시 추가**한다. 이후 질문 시 이 파일을 참조하여 이미 확인된 내용과 모순되지 않도록 한다.
   - 저장 경로: `{프로젝트}/00-interview-log.md`
   - 이 파일은 인터뷰 진행 중 누적 업데이트되며, 세션이 끊어져도 재개 가능하다.
   - **중간 요약 제시 타이밍**: 사람이 요청하거나, 질문 주제가 전환될 때 지금까지 확인된 내용을 요약하여 사람에게 보여준다. 사람이 수정/보충하면 파일에 반영 후 계속한다.
5. **인터뷰 종료 규칙**:
   - AI가 "추가 확인 사항이 없다"고 판단하면 종료를 **제안**한다. 사람이 승인하면 종료, 거부하면 사람이 추가로 확인할 항목을 지시한다.
   - 사람이 직접 "이만하면 됐다"고 판단하면 즉시 종료한다.
6. 종료 시 인터뷰 기록 파일을 **최종 확정본으로 정리**한다: 확인된 요구사항, 핵심 제약, 암시적 전제 조건 목록. 이 파일이 이후 체급 판정과 Draft 생성의 입력이 된다.

#### Step 2: 체급 판정 (Sonnet)

인터뷰 결과를 바탕으로 작업의 체급을 판정한다.

**판정 기준:**

| 체급 | 판정 기준 | 예시 |
|---|---|---|
| **S** | 변경 파일 3개 이하 예상, 설계 판단 없음 | 버그 수정, 설정 변경, 유틸 함수 추가 |
| **M** | 단일 관심사, 설계 문서 A4 2장 이내, API 3개 이하 | API 엔드포인트 1개 + 테스트 + DB 마이그레이션 |
| **L** | 독립 실행 가능한 M급 태스크가 2개 이상 | 대시보드 위젯 6개 (각각은 M 이하) |
| **XL** | 순차 Phase가 필요하거나, PRD 수준의 목표/범위 정의가 선행되어야 함 | 시스템 이관, 신규 서비스 구축 |

**경계 사례 처리:**
- 기준이 딱 떨어지지 않는 경우, AI가 후보 체급과 그 근거를 why/how 포함하여 제시한다 (예: "M과 L 사이인데, 하위 작업 간 의존성이 강해 분리 실행이 어려우므로 M을 제안합니다").
- ★ 사람이 최종 판단한다.

**작업 중 체급 변경(승격):**

파이프라인 진행 중 체급이 맞지 않다고 판단되면 AI가 승격을 **제안**한다. 사람이 **승인하면 승격 동작을 수행**하고, **거부하면 현재 체급으로 계속** 진행한다.

| 승격 | 트리거 | 승인 시 | 거부 시 |
|---|---|---|---|
| S → M | 구현 중 설계 판단이 필요해짐 | Phase 1 Step 4(설계 초안)로 이동 | S로 계속 구현 |
| M → L | 크기 점검 또는 리뷰에서 태스크 분할이 필요해짐 | 조감도 + 태스크 목록 생성 → 파이프라인 종료 | M으로 계속 진행 |
| L → XL | 태스크 간 순차 Phase 구분이 필요해짐 | PRD 작성 → 파이프라인 재진입 | L로 계속 진행 |

#### Step 3: 체급별 분기

| 체급 | 동작 |
|---|---|
| **S** | 설계 생략 → 바로 Phase 4(구현) 진입 |
| **M** | 설계 초안 생성 → Phase 1.5(크기 점검) → Phase 2~6 수행 |
| **L** | 조감도(00-index.md) + 태스크 목록 생성 → **파일로 저장하고 종료**. 각 태스크는 별도 세션에서 S/M으로 수행 |
| **XL** | PRD 작성 → Phase 1.5(크기 점검) → Phase 2(wtth PRD 리뷰 모드) → Phase 3(종결 판단) → Phase 분할 + 태스크 목록 생성 → **파일로 저장하고 종료**. 각 Phase 내 태스크는 별도 세션에서 L/M/S로 수행 |

**L/XL의 핵심 원칙:**
- L/XL은 파이프라인의 실행 단위가 아니다. **실행 결과물의 저장소이자 추적 구조**이다.
- 파이프라인이 실제로 도는 단위는 **항상 S/M**이다.
- L/XL의 산출물은 프로젝트 구조(index + 태스크 목록 + RDR/ADR + 작업 요약 카드)로 남는다.
- 다음 세션에서 papillon 호출 시 index를 읽고 "다음 태스크"를 제안한다.

L/XL 산출물 구조: [02-common-spec.md 섹션 8](./02-common-spec.md) 참조.

#### Step 4: Draft 생성 (M 체급, Sonnet)

1. 인터뷰 결과를 기반으로 doc-standard(01-doc-standard.md) 기준으로 설계 초안을 작성한다.
2. 인터뷰에서 확인된 **핵심 제약은 00-index.md에 명시적으로 기재**한다.
3. 초안을 사람에게 보여주고 Phase 1.5로 자동 전환한다.

### 2.2. Phase 1.5: 태스크 크기 점검 (1차, M 체급)

초안이 나온 직후, 리뷰 진입 전에 **실제 산출물 기반으로** 크기를 점검한다.

**Step 1: 단순 판별 (Haiku)**

| 기준 | 판단 방법 |
|---|---|
| 설계 문서 A4 2장 초과 | 초안 문서 길이 측정 |
| 관련 API 3개 초과 | 초안에서 API/엔드포인트 수 확인 |

**Step 2: 분할 제안 (Sonnet, 기준 초과 시만)**

- Haiku가 기준 초과를 판별하면 Sonnet이 분할안을 제안한다.
- ★ 사람이 승인하면 분할 후 Phase 2로, 거부하면 그대로 Phase 2로 진행.
- 리뷰 1라운드 후 wtth이 건수/심각도/패턴을 분석하여 태스크 크기 경고를 제안할 수 있다 (2차 안전망).

### 2.3. Phase 2: 설계 리뷰

1. wtth을 **설계 리뷰 모드**로 호출한다.
2. 기존 wtth 프로세스 그대로 수행 (전문가 제안 → 승인 → 병렬 실행 → 항목별 의사결정).
3. 리뷰 종료 시:
   - RDR 자동 생성 → `docs/decisions/reviews/`에 저장
   - P0/P1 수용 건 중 설계 변경 수반 항목 → ADR 승격 제안
4. ADR 승격 시 → Phase 2.1로 진입:
   - ADR 작성 → wtth로 ADR 리뷰 → ★ 사람 의사결정 → RDR 생성
   - **ADR 리뷰 결과 분기:**
     - 승인 → ADR 확정, Phase 2로 복귀하여 다음 finding 처리
     - 방향 기각 → ADR을 다른 방향으로 재작성 → Phase 2.1 재진입 (RDR에 기각 사유 기록)
     - ADR 자체 불필요 판정 → ADR 드롭, finding 수용은 유지하고 구현에서 처리 (RDR에 드롭 사유 기록)

### 2.4. Phase 3: 종결 판단

1. Agent S가 현재 상태를 분석한다:
   - 미해결 P0: 0건인가
   - 미해결 P1: 0건인가
   - 설계 문서 Status: Work: Completed 가능한가
2. 조건 충족 시 종결을 제안한다.
3. 사람이 승인하면 설계 확정 (Status: Completed + Approved).
4. 사람이 거부하면 사유를 받고 Phase 2로 복귀.

### 2.5. Phase 4: 구현

**4-A. 개발 작업 (코드)**
1. 확정된 설계서의 태스크 목록을 순서대로 구현한다.
2. Phase 5로 자동 전환.

**4-B. 인프라/수행 작업**
1. 태스크 내용을 기반으로 **런북(절차서)을 생성**한다. 런북은 [50-runbook-template.md](./50-runbook-template.md)의 형식을 따른다.
2. wtth을 **운영 절차 리뷰 모드**로 호출한다.
3. ★ 사람: finding별 수용/기각.
4. 절차서 확정 후, 각 단계를 순서대로 실행한다.
5. 각 단계의 검증 커맨드를 수행하여 성공 여부를 판단한다.
6. 전체 완료 확인 후 Phase 6으로 전환.

### 2.6. Phase 5: 코드 리뷰

설계 리뷰(Phase 2)와 달리, 코드 리뷰는 **사람이 코드를 직접 확인**해야 한다.
요약만 보고 수용/기각하는 방식은 코드의 정합성을 판단할 수 없기 때문이다.

**방식: AI 자동 수정 + 사람 diff 전문 확인 (방식 B)**

1. wtth을 **코드 리뷰 모드**로 호출한다 (코드 전문가 풀 사용).
2. 리뷰 대상: `git diff` 기반 변경사항.
3. AI 전문가들이 지적 사항을 도출하고, **수정까지 자동 수행**한다.
4. 수정 결과를 **AS-IS / TO-BE diff 전문**으로 사람에게 출력한다.
5. 사람이 diff를 확인하고 **승인 또는 반려** (반려 시 사유 포함).
6. 반려 시 사유를 반영하여 재수정 → 다시 diff 제시 → 승인될 때까지 반복.
7. Agent S가 종결 제안 또는 사람이 직접 종결.
8. 종결 시 RDR 생성 → Phase 6.

### 2.7. Phase 6: 커밋 + PR + 완료

1. 커밋을 수행한다 (`/commit-plugin:commit` 사용, Human-Time/Ai-Driven-Time 확인).
2. **PR 생성 여부를 프로젝트 CLAUDE.md의 `Git.Platform` 값으로 결정**한다:

   ```markdown
   ## Git
   - **Platform**: github  # github | bitbucket | none
   ```

   | Platform 값 | 동작 |
   |---|---|
   | `github` | `gh pr create` — RDR/ADR/리뷰결과를 PR description에 자동 포함 |
   | `bitbucket` | Bitbucket MCP 또는 수동 안내 — 동일하게 PR description 생성 |
   | `none` / 미설정 | 커밋 + PR 본문 출력 후 종료 |

3. PR 생성 시 본문에 포함되는 내용:
   - 설계 RDR 요약
   - 코드 리뷰 RDR 요약
   - 관련 ADR 목록
   - 변경 파일 및 구현 요약
4. 사람에게 다음 행동을 묻는다:
   - "다음 태스크" → Phase 1로 복귀
   - "종료" → 파이프라인 종료

## 3. wtth 호출 인터페이스

papillon가 wtth을 호출할 때 전달하는 정보:
