## 1. 개요

inquisition은 사람의 요구사항을 체계적으로 추출·정제하는 독립 스킬이다.

- **호출:** `/papillon:inquisition` (단독 실행) 또는 papillon 오케스트레이터가 Phase 1에서 자동 호출
- **모델:** Opus (추론력 기반 품질 우선. 인터뷰는 사람 응답 대기가 대부분이라 AI 응답 속도가 병목이 아니다)
- **페르소나:** [INQUISITOR](./personas/inquisitor.md) — inquisition 전용

### 1.1. 단독 실행 vs 파이프라인 내 실행

| 항목 | 단독 실행 (`/papillon:inquisition`) | 파이프라인 내 실행 |
|---|---|---|
| 호출 주체 | 사람이 직접 | papillon 오케스트레이터 |
| 산출물 | `00-interview/` 디렉토리 (요약 + 문맥 파일 + 체급) | 동일 |
| 체급 판정 | 수행 (요약에 기록) | 동일 |
| 이후 동작 | 사람에게 반환. 사람이 판단하여 후속 작업 결정 | 오케스트레이터가 체급별 분기로 자동 진행 |

## 2. 프로세스

### 2.1. 진입

papillon 컨텍스트 스캔(§3.0) 또는 단독 호출 시, `00-interview/` 디렉토리 존재 여부로 신규/재개를 판단한다.

#### 재개 (`00-interview/` 존재 시)

1. 기존 문맥 파일을 읽어 상태를 복원한다 (Open/Settled/Deferred 현황 파악).
2. 컨텍스트 수집(§2.2 항목 2)을 다시 수행한다 — 세션 간에 데이터가 변경되었거나 새 데이터가 추가되었을 수 있다.
3. 사람에게 현재 상태를 요약 제시한다 (미결 N건, 확정 N건, 보류 N건, 변경된 컨텍스트가 있으면 함께).
4. 사람이 재개 지점을 결정한다.

### 2.2. 신규 인터뷰

1. 사람이 요구사항을 전달한다 (자연어, 구조화된 문서, 이슈 링크, 이미지 등 형태 무관).
2. **컨텍스트 수집**: Opus가 작업에 활용 가능한 데이터를 사람에게 확인한다.
   - 어떤 데이터가 있는지 (소스코드, 설계문서, DB 스키마, KB, API 명세 등)
   - 어떤 형태인지 (파일, 저장소, 외부 시스템 등)
   - 어떻게 접근하면 되는지 (경로, MCP, URL 등)
   - 사람이 지시한 데이터를 읽고 현황을 파악하여 답변한다. 매 턴 종료 시 "더 봐야 할 데이터가 있는지" 확인한다. 사람이 없다고 하면 수집된 데이터를 교차 분석하여 모순, 불일치, 공백을 식별한다. 발견된 항목이 있으면 항목 3의 인터뷰 제안 시 우선 확인 포인트로 함께 제시한다.
3. Opus가 요구사항 + 수집된 컨텍스트를 바탕으로 인터뷰 필요 여부를 판단하고 사람에게 **제안**한다.
   - 인터뷰 제안 시: 구체화가 필요한 포인트를 함께 제시한다.
   - 인터뷰 불필요 판단 시: 이유와 함께 직행을 제안한다. 수락 시 항목 4~10을 건너뛰고, 항목 11(요약 생성) → 항목 12~14(결과물 검증) → 항목 15(체급 판정)로 진행한다.
   - ★ 사람이 최종 결정한다 (제안 수락 / 오버라이드).
4. Opus가 요구사항 + 수집된 컨텍스트를 분석하여 다음을 수행한다:
   - 기술 키워드, 도메인 개념 추출
   - 각 키워드 뒤에 숨겨진 함의(implication)와 암시적 제약 후보 감지
   - 기술적 결정의 의도, 실행 순서 제약, 핵심 전제 조건에 대해 사람에게 확인 질문
5. 사람이 답변하면, 답변 내용에서 추가 확인이 필요한 부분을 재질문한다.

### 2.3. 문맥 관리

6. **문맥 단위 기록**: 하나의 관심사에 대한 질답이 마무리되면, 해당 내용을 정리하여 문맥 파일을 생성한 뒤 다음 관심사로 넘어간다.
   - 저장 경로: `{프로젝트}/00-interview/`
   - Opus가 질답 흐름에서 관심사 경계를 판단하여 파일 생성을 **제안**한다 (예: "여기까지를 인증 전략으로 정리해도 될까요?"). 사람이 확인하면 파일을 생성한다 (예: `01-auth-strategy.md`, `02-data-model.md`).
   - 각 파일에 상태를 표기한다: **미결**(Open) — 아직 확인이 필요한 항목 있음 / **확정**(Settled) — 결론 도출 완료 / **보류**(Deferred) — 의도적으로 미뤄둠, 사유 기재.
   - 다음 관심사 진행 중 이전 관심사에 영향을 주는 내용이 나오면, 해당 문맥 파일로 돌아가 수정한 뒤 현재 관심사로 복귀한다.
   - 세션이 끊어져도 파일별 상태를 보고 재개 가능하다.
7. **문맥 전환 감지**: 사람이 다른 주제로 넘어갈 때, Opus는 다음을 수행한다:
   - 이전 문맥에 미결 항목이 있으면 환기한다 (예: "인증 방식은 아직 OAuth vs API Key 중 결론이 안 났는데, 먼저 데이터 모델로 넘어갈까요?"). 사람이 넘어가겠다고 하면 해당 문맥을 미결 상태로 두고 진행한다.
   - 지금까지 확인된 내용을 요약하여 사람에게 보여준다. 사람이 수정/보충하면 해당 문맥 파일에 반영 후 계속한다.
   - 사람이 직접 요약을 요청한 경우에도 동일하게 제시한다.
8. **확정 사안 번복 처리**: 사람이 확정(Settled)된 문맥을 뒤집으려 할 때, Opus는 다음을 수행한다:
   - `00-interview/` 내 다른 문맥 파일에서 해당 관심사의 키워드와 결정 사항을 검색하여, 구체적인 연결 고리(어느 파일의 어떤 결정이 영향받는지)를 사람에게 제시한다 (예: "인증을 API Key로 바꾸면 `03-session-management.md`의 세션 유지 방식, `04-permission-model.md`의 권한 위임 구조가 재검토 필요합니다").
   - 사람이 영향 범위를 확인한 뒤 판단한다: 변경 범위를 축소 조정 / 영향받는 문맥과 함께 수정 진행 / 전체 인터뷰 재시작.
   - 수정 진행 시 영향받는 문맥 파일의 상태를 미결(Open)로 되돌리고, 해당 문맥들을 순차 재검토한다.

### 2.4. 보정 질문

9. **사람의 표현 패턴에 따른 보정 질문**:
   - 암시적으로만 말함 ("그냥 빠르면 돼") → 구체화 질문 ("응답 시간 기준이 있나요?")
   - 솔루션부터 말함 ("Redis 쓰자") → 의도 역추적 ("Redis를 쓰려는 이유가 캐시 때문인가요?")
   - 핵심 제약을 당연하게 생략 → 도메인 기반 체크리스트로 누락 감지
   - **의도 노출(Meta-Reasoning)**: 질문을 던질 때 "이 질문이 이후 설계의 어떤 부분에 영향을 주는지"를 함께 언급한다 (예: "데이터 보존 기간을 여쭤보는 이유는 DB 파티셔닝 전략과 콜드 스토리지 이관 여부가 달라지기 때문입니다"). 사람이 맥락을 이해하면 더 정확한 답변을 제공한다.

### 2.5. 종료 및 산출물

10. **인터뷰 종료 규칙**:
    - AI가 "추가 확인 사항이 없다"고 판단하면 종료를 **제안**한다. 사람이 거부하면 추가로 확인할 항목을 지시한다.
    - 사람이 직접 "이만하면 됐다"고 판단해도 종료를 개시한다.
    - ★ **사람의 명시적 승인이 있어야** 미결 처리, 파일 생성, 요약, 체급 판정 등 후속 단계로 진행한다. 승인 없이 진행하지 않는다.
    - 미결(Open) 상태인 문맥 파일이 있으면 종료 전에 항목별로 사람에게 처리를 요구한다: **지금 결정**(Settled) / **의도적 보류**(Deferred). Deferred 시 사유와 함께 **결정을 위해 필요한 데이터 또는 조건**을 역으로 제안한다 (예: "추후 '예상 월간 데이터 증가량'이 확보되면 확정 가능합니다"). 모든 미결 항목이 Settled 또는 Deferred가 되어야 종료할 수 있다.
11. 종료 시 `00-interview/00-summary.md`에 전체 요약을 생성한다: 확인된 요구사항, 핵심 제약, 암시적 전제 조건 목록, Deferred 항목과 사유. 인터뷰를 건너뛴 경우(항목 3에서 불필요 판단)는 컨텍스트 수집 결과를 기반으로 요약을 생성한다.
    - 암묵적 피드백 수집 결과를 사람에게 확인한 뒤 `00-interview/feedback.md` 또는 AI 메모리에 저장한다. 상세: [02-common-spec.md §10](../02-common-spec.md).

### 2.6. 결과물 검증

12. **셀프 오딧**: inquisitor가 전체 문맥 파일을 자기 체크리스트(P0~P3)로 재점검한다.
    - 모든 Settled 문맥에 측정 가능한 기준이 있는가 (P0)
    - 컨텍스트 간 모순이 해소되었는가 (P0)
    - 암시적 전제 조건이 추출되었는가 (P1)
    - 관심사 간 의존 관계가 식별되었는가 (P1)
    - 미흡한 항목 발견 시 사람에게 제시하고 추가 질문/수정 여부를 확인한다. 이상 없으면 다음으로 진행한다.
    - 이 시점에서 **예비 체급(preliminary)**을 산정한다. 공식 판정 전이지만 스팟 체크 투입 여부 결정에 사용한다.
13. **경량 스팟 체크** (예비 체급 M 이상만, S는 스킵):
    - 요약과 문맥 파일을 관련 전문가 1~2명에게 전달한다.
    - 전문가 선정: 인터뷰 내용에서 가장 관련 높은 도메인 기준 (예: DB 관련 내용 많으면 DBA, 보안 언급 있으면 SEC).
    - 전문가 역할: 풀 리뷰가 아님. **"이 요구사항에서 우리 관점에서 빠진 질문이 있나?"** — 누락 체크만 수행한다.
    - finding 형식: "X에 대해 확인이 필요합니다" 수준. 심각도는 P1(빠진 질문) 또는 P2(있으면 좋은 질문)만 사용한다.
    - 스팟 체크 결과를 사람에게 제시한다:
      - 사람이 추가 인터뷰 필요하다고 판단 → §2.2로 복귀 (해당 관심사만 재개)
      - 사람이 현재 수준으로 충분하다고 판단 → 다음으로 진행
14. **사람 최종 확인**: 셀프 오딧 + 스팟 체크 결과를 포함한 최종 요약을 사람에게 제시한다. ★ 사람이 "이 요구사항으로 진행해도 좋다"고 승인하면 체급 판정으로 진행한다.

### 2.7. 체급 판정

15. **체급 판정**: 인터뷰 결과(또는 컨텍스트 수집 결과)를 바탕으로 체급을 판정한다. Opus가 아래 두 축을 기준으로 종합 판단한다:
    - **설계 판단의 복잡도**: 의사결정이 필요한 기술적/구조적 선택지가 얼마나 있는가
    - **독립 실행 가능한 단위 수**: 하나의 세션에서 완결할 수 있는가, 분할이 필요한가

    | 체급 | 핵심 기준 | 보조 참고 지표 | 예시 |
    |---|---|---|---|
    | **S** | 설계 판단 없음, 단일 작업으로 완결 | 변경 파일 수, 영향 범위 | 버그 수정, 설정 변경, 유틸 함수 추가 |
    | **M** | 단일 관심사 내 설계 판단 필요 | 문서 분량, API 수 | API 엔드포인트 1개 + 테스트 + DB 마이그레이션 |
    | **L** | 독립 실행 가능한 M급 태스크가 2개 이상 | 태스크 간 의존 관계 | 대시보드 위젯 6개 (각각은 M 이하) |
    | **XL** | 순차 Phase가 필요하거나, PRD 수준의 목표/범위 정의가 선행되어야 함 | Phase 간 의존성 | 시스템 이관, 신규 서비스 구축 |

    보조 참고 지표(파일 수, API 수 등)는 판정의 근거로 참고하되, 기계적으로 적용하지 않는다.
    경계 사례: Opus가 후보 체급과 근거를 why/how 포함하여 제시한다. ★ 사람이 최종 판단한다.

    **사람에게 안내할 때는 코드(S/M/L/XL)가 아니라 자연어로 설명하고 다음 단계를 안내한다:**

    | 체급 | 안내 문구 |
    |---|---|
    | **S** | "작은 단위의 작업입니다. 설계는 딱히 필요 없겠군요. 바로 구현으로 들어가면 됩니다." |
    | **M** | "규모가 어느 정도 있는 작업입니다. 구현 계획을 세워봅시다." |
    | **L** | "상당한 규모가 예상됩니다. 작업 단위를 나누고, 각각에 대해 세부 구현 계획을 세워야 합니다." |
    | **XL** | "매우 큰 작업입니다. 무엇을 왜 해야 하는지부터 정의해 나가야 합니다." |

    확정된 체급과 판정 근거를 `00-interview/00-summary.md`에 기록한다.
