## 1. 진행 불가 감지

shackled이 다음 중 하나에 해당하면 구현을 멈추고 진행 불가를 사람에게 보고한다:

| 감지 조건 | 설명 |
|---|---|
| 기술적 막힘 | 구현에 필요한 기술적 결정이 설계서에 없거나 모순된 경우 |
| 컨텍스트 부족 | 구현에 필요한 정보(API 명세, 스키마, 환경 변수 등)가 없는 경우 |
| 설계 불일치 | 설계서의 내용이 실제 구현과 충돌하여 어느 방향으로 구현해도 설계를 위반하는 경우 |
| 내부 크로스 체크 반복 실패 | [01-cross-check.md](./01-cross-check.md) §5의 P0/P1 finding 3회 수정 후 미해소, 또는 전체 라운드 5회 초과 |
| 사람이 직접 선언 | 코어 프로세스(00-design-shackled.md §3.3.3)에서 사람이 진행 불가를 선언한 경우 |

---

## 2. 보고 형식

진행 불가 시 사람에게 다음을 제시한다:

```
[진행 불가] 태스크 {task_id}: {태스크 제목}

막힌 이유: {구체적인 원인}
원인 추적 힌트: {어느 설계 결정에서 비롯된 것인지, 어느 문맥 파일이 관련있는지}
현재까지 완료된 태스크: {목록}
미완료 태스크: {목록}

가능한 다음 행동:
1. {선택지 1} — {설명}
2. {선택지 2} — {설명}
3. {선택지 3} — {설명}
```

shackled이 원인 수준을 판단하여 §3의 적절한 선택지를 제안한다. 사람은 제안된 선택지 외의 다른 행동도 선택할 수 있다.

---

## 3. 사람 결정 분기

보고를 받은 사람이 원인 수준에 따라 결정한다. shackled은 원인 힌트와 선택지를 제공하며 롤백 지점 결정은 사람의 몫이다.

| 원인 수준 | 사람 결정 | 처리 |
|---|---|---|
| 경미 | shackled 내에서 해결 | 사람이 방향을 지시하고 shackled이 재개 |
| 중간 | 설계 수정 후 재진입 | shackled 종료 → 설계 수정 → 리뷰 재진입 → shackled 재호출 |
| 심각 | inquisition 롤백 | shackled 종료 → inquisition 재진입 |

---

## 4. 상태 저장 후 종료

진행 불가로 shackled이 종료될 때 현재 상태를 저장한다 ([02-state-management.md](./02-state-management.md)). 이후 재개 시 `task_id`를 통해 이어받는다.
