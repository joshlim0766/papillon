## 1. 배경

shackled 설계서를 wtth 설계 리뷰로 검증하면서, 동시에 wtth 자체의 확장 가능성을 테스트한다. 기존 wtth 전문가 풀(ARCH/BE/FE/SRE/SEC/PO)이 워크플로우·스킬 명세 리뷰에 적합하지 않아, 신규 전문가(SPEC, IX)를 정의하고 ARCH + SPEC + IX 3인 구성으로 리뷰를 수행한다.

## 2. 리뷰 목적

### O1. review-workflow 모드 필요성 규명

기존 5모드(PRD/설계/ADR/운영절차/코드) 중 워크플로우·스킬 명세를 커버하는 모드가 없다. 이번 리뷰를 통해 6번째 모드(`review-workflow` 또는 `review-prompt`)의 필요성을 검증한다.

- 기존 설계 리뷰 모드와의 차이점은 무엇인가
- 코어 프로세스(finding 기반, 수렴 메커니즘)는 그대로 사용 가능한가
- 전문가 풀과 체크 관점만 달라지면 충분한가, 프로세스 자체에도 조정이 필요한가

### O2. 신규 페르소나(SPEC, IX) 평가 및 개선

초기 정의된 SPEC(명세 분석가)과 IX(인터랙션 설계자)가 실제 리뷰에서 유의미한 finding을 도출하는지 평가한다.

- 관점/금기/체크리스트가 적절한가
- SPEC과 IX 간 역할 경계가 명확한가 (중복/공백 없는가)
- ARCH와의 역할 분담이 자연스러운가

### O3. inquisition → Sonnet 초안 파이프라인 품질 평가

inquisition이 Draft 직전까지 shackled을 끌고 왔고, 그 결과물을 기반으로 Sonnet이 설계 초안을 작성했다. 이 파이프라인이 리뷰 받을 가치가 있는 결과물을 만들어내는지 평가한다.

- 인터뷰 산출물이 Sonnet에게 충분한 맥락을 제공했는가
- Sonnet 초안의 품질 — 구조, 완결성, 설계 판단의 합리성
- 초안에서 발견되는 문제가 "Sonnet의 한계"인가 "인터뷰 정보 부족"인가

### O4. 기존 ARCH 페르소나의 적응력 평가

ARCH가 소프트웨어 아키텍처용으로 정의되어 있는데, 워크플로우/스킬 설계 맥락에서도 유효한 finding을 내는지 확인한다.

- 관점/체크리스트가 이 맥락에서도 작동하는가
- 조정이 필요하다면 어떤 부분인가

### O5. finding 포맷 적합성

common-spec 6.1의 finding 포맷(문제/왜 문제인가/참조/수정안)이 워크플로우 명세 리뷰에도 자연스러운지 확인한다.

- "참조" 필드에서 코드 위치 대신 프로세스 단계를 참조하는 것이 자연스러운가
- 포맷 변형이 필요한가

### O6. 전체 파이프라인 및 도구 강화

O1~O5의 결과를 종합하여 papillon 파이프라인과 wtth 도구를 개선한다.

- 새 모드 추가 여부 결정
- 페르소나 확정/수정
- 파이프라인 프로세스 조정 사항 도출

## 5. 목적별 평가

### 5.1. O1 평가: review-workflow 모드 필요성

**결론: 별도 모드 불필요. 기존 모드 조합 + 전문가 풀 커스텀으로 충분.**

워크플로우 설계서는 "무엇을 만들 것인가"(설계)와 "어떤 순서로 실행할 것인가"(절차)가 한 문서에 공존한다. 실제 리뷰에서 전문가들의 지적도 설계 리뷰 성격(모듈 구조, 의존성, 파라미터 규격)과 운영 절차 리뷰 성격(단계별 사람 확인, 상태 전이, 진행률 추적)이 혼재되었다. 기존 어느 단일 모드에도 깔끔하게 맞지 않지만, 별도 모드를 신설하기보다 기존 모드를 조합하는 것이 현실적이다.

- **코어 프로세스는 그대로 사용 가능**: finding 기반, 발산→수렴→사인오프 메커니즘이 그대로 작동했다
- **설계 리뷰 + 운영 절차 리뷰 복합 적용**: 두 모드의 관점을 동시에 적용하되, 전문가 풀을 워크플로우에 적합한 구성(ARCH + SPEC + IX)으로 커스텀하면 이번 리뷰와 유사한 결과를 얻을 수 있다
- **별도 모드 비용**: 워크플로우 설계서 리뷰 빈도가 높지 않아 전용 모드의 관리 비용 대비 효용이 낮다

### 5.2. O2 평가: SPEC, IX 페르소나

**결론: 유효하다. 단, 추가 검증 필요 + 투입 제한 메커니즘 필요.**

**유효성:**
- **SPEC**: 1라운드 P0 3건을 포함해 17건 도출. 명세 완결성·분기 조건·파라미터 규격 관점에서 ARCH/IX가 잡지 못하는 구체적 빈칸을 발견. "이 명세만 보고 구현할 수 있는가"라는 판단 기준이 효과적
- **IX**: P0 1건(크로스 체크 은닉)을 포함해 10건 도출. 사람-AI 상호작용 흐름, checkpoint 정보 충분성, 진행률 추적 등 사용자 경험 관점의 finding이 설계 품질을 실질적으로 개선
- **SPEC↔IX 경계**: 금기 설정이 효과적이었다. SPEC이 "정보가 있는가"를 보고, IX가 "정보가 적시에 제시되는가"를 봄. 중복 finding 없었음
- **ARCH↔SPEC/IX 경계**: ARCH는 구조·패턴 일관성, SPEC/IX는 명세·상호작용에 집중. 동일 주제(크로스 체크 은닉, 인테그레이션 등)에 대해 다른 관점의 보완적 finding이 나와 묶음 처리가 자연스러웠음

**추가 검증 필요:**
- 이번은 첫 실전 투입이고 리뷰 대상이 워크플로우 설계서 1건. 다른 유형의 문서(프롬프트 명세, 데이터 파이프라인 설계 등)에서도 동일하게 작동하는지 추가 검증 필요

**투입 제한 메커니즘 (전문가 풀 비대화 방지):**
- 전문가가 늘수록 "이 리뷰에도 저 전문가 넣으면 좋지 않을까?" 압력이 생김. PoC 레벨 빠른 개발에 SEC/PM을 넣으면 맞는 말이지만 노이즈인 finding이 폭발
- **기본 풀 상한 3인**: 모드별 기본 조합은 최대 3인으로 고정
- **선택적 투입 조건**: 페르소나 파일에 투입 조건을 명시 (DBA 선례 — "DDL 포함 설계 리뷰에만 투입"). SPEC: "워크플로우·프로세스 명세가 핵심일 때만", IX: "사람-AI 상호작용이 포함될 때만"
- **사람이 명시적 제외 가능**: 오케스트레이터가 추천하되, 사람이 제외하면 즉시 반영. 해당 리뷰 세션(RDR 1개 단위) 내에서 제외된 전문가를 재추천하지 않음. 제외 결정은 리뷰 상태에 `excluded_experts`로 저장

### 5.3. O3 평가: inquisition → Sonnet 파이프라인

**결론: 리뷰 받을 가치가 있는 결과물을 생산했다. Sonnet 초안 품질 개선을 위해 핸드오프 브리프 추가.**

**파이프라인 평가:**
- 인터뷰 산출물이 전체 구조와 핵심 결정(시블링 관계, 크로스 체크 메커니즘, 4-A/4-B 분기 등)에 대한 충분한 맥락을 제공
- **inquisition의 관심사 커버리지가 우수**: 1라운드 finding이 7개 관심사에 고르게 분산(9/8/7/5/3/3/2건). 한쪽이 비거나 몰리지 않았다는 것은 inquisition이 인터뷰에서 모든 관심사를 빠짐없이 수집하여 Sonnet에게 균형 잡힌 맥락을 전달했다는 증거. finding이 많았던 건 "깊이"의 문제이지 "범위"의 문제가 아니었음
- Sonnet 초안의 품질: 구조적으로 건전했으나 (P0 순환 의존 없음), 세부 명세(분기 조건 MECE, 파라미터 유효 범위, 엣지 케이스)에서 P0 4건 + P1 15건 발생
- 리뷰가 파이프라인의 품질 보증 장치로 기능함을 확인

**원인 분석:**
- inquisition의 소임(인터뷰이가 원하는 걸 체계적으로 정리 → 설계 맥락 생성)은 다하고 있다. 명세 공백은 inquisition의 질문 부족이 아니라 Sonnet이 맥락을 초안으로 변환하는 과정에서 발생
- inquisition이 "분기 조건을 더 세밀하게 질문"하는 건 역할 범위를 벗어남

**개선 지점: inquisition 핸드오프 브리프 추가:**
- inquisition은 Opus이고 인터뷰 맥락을 가장 깊이 이해하는 주체. 인터뷰 종료 → 체급 판단 시점에 **Sonnet용 핸드오프 브리프**를 함께 생성
- 핸드오프 브리프 = 범용 체크리스트가 아닌, 이번 인터뷰 맥락 기반 맞춤형 주의 사항. 예:
  - "상태 전이가 많으니 status enum을 빠짐없이 정의할 것"
  - "시블링 스킬(wtth)이 있으니 패턴 일관성을 확인할 것"
  - "사람-AI 상호작용이 핵심이니 모든 checkpoint에서 제시 정보를 명시할 것"
- 체급 판단과 유사한 수준의 추론이므로 큰 비용 추가 없이 가능
- 매번 다른 설계에 맞춤형이라 범용 체크리스트보다 효과적

### 5.4. O4 평가: ARCH 적응력

**결론: 워크플로우 설계 맥락에서도 유효하게 작동했다. 관점/체크리스트 조정 불필요. ARCH↔SPEC 보완적 중첩에 대한 오케스트레이터 묶음 처리가 필수.**

**적응력:**
- ARCH가 1라운드 10건 + 2라운드 5건 = 총 15건 도출. 모듈 구조, 의존성 방향, 패턴 일관성, 레이어 분리 관점이 워크플로우 설계에도 적용 가능
- 특히 효과적이었던 항목: 시블링 설계 간 패턴 일관성 (진입 분기 테이블, 투명성 정책, 심각도 체계), 관심사 분리 (런북 생성/실행)
- "소프트웨어 아키텍처"를 "워크플로우 아키텍처"로 자연스럽게 매핑하여 적용. 별도 조정 없이도 적응력이 충분

**ARCH↔SPEC 보완적 중첩:**
- ARCH와 SPEC의 finding이 동일 주제에서 빈번하게 겹쳤으나 시각이 다름. ARCH는 "구조적으로 잘못됐다" (패턴 불일치, 관심사 미분리), SPEC은 "명세가 빠졌다" (기준 미정의, 엣지 케이스 미처리). 같은 문제의 다른 면이며 병합 시 수정안이 더 완전해짐
- 중복이 아닌 보완이지만, 사람 입장에서 같은 주제가 2~3건으로 등장하여 finding 건수가 부풀려 보이고 읽는 부담이 증가
- **오케스트레이터의 주제별 묶음 처리가 필수**: 이번 리뷰에서 묶음 제시 시 의사결정이 수월했음. 사람의 피로도를 줄이는 데 핵심적인 역할. wtth 코어에 묶음 처리를 정식 메커니즘으로 반영해야 함

### 5.5. O5 평가: finding 포맷 적합성

**결론: 적합하다. "참조" 필드의 유연성이 자연스럽게 작동.**

- "참조" 필드에서 코드 위치 대신 설계서 섹션 번호(§3.1, §6.5 등)를 자연스럽게 사용
- 문제→왜 문제인가→참조→수정안 구조가 워크플로우 명세 리뷰에서도 원인-결과 전달에 효과적
- 포맷 변형 불필요

## 6. 피드백

### 6.1. 프로세스 피드백

- **코어 메커니즘 검증 완료**: 발산→수렴→사인오프 프로세스가 워크플로우 리뷰에서도 그대로 작동. 1라운드(37건)→2라운드(14건)→사인오프(0건)으로 수렴 추세 명확
- **주제별 묶음 처리가 효과적**: 동일 주제에 대한 다관점 finding을 묶어서 제시하니 사람의 의사결정 부담이 줄어듦
- **P2/P3 일괄 제시 + 체리픽**: 사람이 전체 수용/기각 대신 건별 선택을 원할 때 유연하게 대응 가능했음
- **3라운드 스킵 판단**: ROI 기반으로 사인오프 직행한 결정이 효율적. 수렴 전환 조건(신규 P0/P1 0건)을 엄격히 따르면 1-2라운드 더 돌았을 텐데, 실질적 가치는 미미했을 것
- **라운드 간 비교 원칙 발견**: 리뷰 프로세스 자체에서 "finding 결정 목록 기반 비교, 외부 도구 무의존" 원칙이 도출됨. wtth 코어에 반영 필요

### 6.2. 전문가 피드백

- **ARCH**: 워크플로우 설계에 잘 적응함. 시블링 간 패턴 일관성 지적이 가장 가치 높았음
- **SPEC (신규)**: 명세 완결성 관점에서 P0 3건 포함 가장 많은 finding(17건) 도출. "구현자가 모호함 없이 구현할 수 있는가" 판단 기준이 날카로움
- **IX (신규)**: 사람-AI 상호작용 관점에서 크로스 체크 은닉(P0)이라는 핵심 이슈를 발견. 진행률, 피드백 루프, 종료 안내 등 사용자 경험 개선에 기여

### 6.3. 관심사 분산도 기반 분할 판단 (핵심 인사이트)

이번 리뷰의 가장 중요한 발견은 finding 자체가 아니라 **finding의 분포 패턴이 설계 대상의 복잡도를 드러낸다**는 점이다.

**문제:**
- shackled은 M 체급 단일 태스크로 설계되었으나, 1라운드에서 37건의 finding이 7개 이상의 독립적 관심사(크로스 체크 엔진, 상태 관리, 프로세스 흐름, 입력 인터페이스, 페르소나 시스템, 진행 불가 처리, 인테그레이션)에 분산되었다
- 이는 설계 대상 자체가 단일 M 태스크에 담기엔 너무 넓다는 신호였으나, 사람(finding 처리에 집중)도 오케스트레이터(건별 제시에 집중)도 이를 감지하지 못했다

**원인 체인:**
1. inquisition은 요구사항 수집·정리가 역할이므로 설계 복잡도를 추론할 이유가 없다
2. Sonnet(초안 작성)은 이런 수준의 판단 능력이 없고, 기준을 줘도 오판할 가능성이 높다
3. 사람은 관성에 젖기 쉽고, finding을 하나씩 처리하다 보면 전체 분포 패턴을 놓친다
4. 결국 **1라운드 결과 분석 시점**이 유일한 개입 지점이나, 현재 wtth §4.2는 건수·심각도·패턴만 분석하고 관심사 분산도는 보지 않는다

**해결:**
- wtth 1라운드 분석(§4.2)에서 finding을 관심사별로 group by하고, 분산도를 기반으로 설계 대상 자체의 분할 필요성을 사람에게 제안한다
- 관심사 N개 이상에서 P0/P1이 분산되면 경고: "이 설계 대상이 단일 태스크로 적절한지 재검토를 권합니다"
- 선택지 제시: 관심사별 분할 / 현재 구조 유지 / 범위 축소

### 6.4. 기타

- 리뷰 대상(shackled 설계서)이 약 370줄 → 리뷰 후 약 550줄로 증가. 증가분 대부분이 1라운드에서 발견된 명세 공백 보충
- 총 2라운드 + 사인오프로 완료. 3인 전문가가 병렬로 리뷰하여 각 라운드 소요 시간이 단일 전문가와 거의 동일
