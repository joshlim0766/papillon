# 1: 설계서 vs 프롬프트 스펙 정체성 혼란

**심각도:** 높음
**분류:** 구조
**상태:** Closed — 4개 설계서 전체 + 하위 설계서에 `<rationale>` 태그 적용 완료, common-spec에 컨벤션 문서화 완료 (2026-04-10)

## 문제

모든 설계서(design-papillon, design-inquisition, design-shackled, 00-core)가 "왜 이렇게 설계했는가"(설계 근거)와 "프롬프트에 어떤 지시를 넣어야 하는가"(프롬프트 스펙)를 구분하지 않고 섞어놓았다.

**예시:**
- papillon 설계서 §3 Phase 간 입출력 매핑 테이블 → 프롬프트에 거의 그대로 들어갈 내용
- "Sonnet을 쓰는 이유(속도·비용 효율 우선)" → 설계 근거이지 프롬프트에 넣을 내용 아님

## 왜 문제인가

프롬프트 변환 단계에서 매번 "이 문장을 프롬프트에 넣어야 하나 말아야 하나"를 판단해야 한다.
- 설계 의도가 누락될 수 있음
- 역으로 설계 근거가 프롬프트에 섞여 들어가 토큰을 낭비할 수 있음

## 제안된 솔루션: 3계층 분리 + SPEC/RAT 마킹

**3계층 모델:**

| 계층 | 역할 | 독자 | 예시 |
|---|---|---|---|
| **설계 근거 (Design Rationale)** | 왜 이 구조인가, 기각된 대안, 제약 | 사람 (미래의 자신) | "Sonnet을 쓰는 이유는 속도·비용 효율 우선" |
| **프롬프트 스펙 (Prompt Spec)** | AI가 따라야 하는 규칙, 분기, 입출력, 톤 | 프롬프트 작성자 → AI | Phase 간 입출력 매핑 테이블, 문맥 전환 감지 규칙 |
| **최종 프롬프트 (Skill File)** | 프롬프트 스펙을 토큰 효율적으로 압축 | AI (런타임) | `~/.claude/commands/papillon.md` + `~/.claude/commands/papillon/*.md` |

**현실적 적용 — 기존 문서에 태그 마킹:**

기존 문서를 뜯어고치지 않는다. 각 섹션/항목에 `[SPEC]`(프롬프트에 들어갈 것)과 `[RAT]`(근거만)을 태그로 붙인다. 최종 프롬프트 작성 시 `[SPEC]`만 추출하면 된다.

**예시 — inquisition 설계서 §1:**

현재:
> 모델: Opus (추론력 기반 품질 우선. 인터뷰는 사람 응답 대기가 대부분이라 AI 응답 속도가 병목이 아니다)

분리 후:
> `[SPEC]` 모델: Opus
>
> `[RAT]` 추론력 기반 품질 우선. 인터뷰는 사람 응답 대기가 대부분이라 AI 응답 속도가 병목이 아니다.

**판별 테스트:** "이 문장을 빼면 AI의 행동이 달라지는가?" → 달라지면 `[SPEC]`, 안 달라지면 `[RAT]`.

**적용 시점:** 새 문서 작성할 때부터 적용. 기존 문서는 건드리지 않아도 됨. 나중에 여유가 생기면 `[RAT]` 블록만 별도 파일로 추출 가능.

## 구현 방식: md 안에 XML 태그로 구조적 격리

마크다운 파서는 인식 못하는 태그를 무시하고, Claude는 XML 태그를 잘 인식한다 (Claude 시스템 프롬프트 자체가 md + XML 혼용). 기존 문서 구조를 깨지 않으면서 해당 블록만 감싸면 된다.

**예시:**

```markdown
## 1. 개요

inquisition은 사람의 요구사항을 체계적으로 추출·정제하는 독립 스킬이다.

- **호출:** `/papillon:inquisition` (단독 실행) 또는 papillon 오케스트레이터가 Phase 1에서 자동 호출
- **모델:** Opus

<rationale>
추론력 기반 품질 우선. 인터뷰는 사람 응답 대기가 대부분이라 AI 응답 속도가 병목이 아니다.
</rationale>
```

**이점:**

1. **프롬프트 변환:** `<rationale>` 블록을 grep으로 일괄 제거하면 프롬프트 스펙만 남음. 또는 남겨두고 "rationale 태그 내용은 배경 지식이며 실행 지시가 아니다" 한 줄 추가
2. **시각적 구분:** GitHub/에디터에서 태그가 그대로 보이니 "이 부분은 근거구나"가 바로 보임
3. **비파괴적:** 섹션 넘버링, 상호참조, Context 헤더 전부 유지

**태그 종류 (3종 이하로 시작):**

| 태그 | 용도 | 예시 |
|---|---|---|
| `<rationale>` | 설계 판단 근거, 대안 기각 이유 | "Sonnet을 쓰는 이유는..." |
| `<calibration>` | 경험적 사례, 수치 근거 | 체급 캘리브레이션 데이터 |
| `<note>` | 미래의 자신에게 남기는 메모, TODO | "팀 확장 시 재검토" |

**주의:** 태그를 세분화하면 태그 자체가 관리 비용. 3종 이하로 시작하고 필요해지면 추가.

**적용 방식:** 새로 쓰거나 수정하는 부분부터 적용. 묶음 B 설계 초안부터 시작.
