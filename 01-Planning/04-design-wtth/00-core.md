## 1. 5모드 리뷰 체계

wtth은 리뷰 대상 유형에 따라 전문가 풀과 의사결정 방식을 자동 전환한다.

| 모드 | 대상 | 기본 전문가 풀 | 의사결정 방식 | 상세 |
|---|---|---|---|---|
| **PRD 리뷰** | PRD, 기획 문서 | PM + ARCH + SEC (3명) | finding별 수용/기각 | [01-review-prd.md](./01-review-prd.md) |
| **설계 리뷰** | 설계 문서, 명세서 | ARCH/BE/FE/SRE/SEC/PO | finding별 수용/기각 | [02-review-design.md](./02-review-design.md) |
| **ADR 리뷰** | ADR (아키텍처 결정 기록) | ARCH + SEC (2명) | finding별 수용/기각 | [02-review-design.md](./02-review-design.md) |
| **운영 절차 리뷰** | 런북, 절차서 | SRE + SEC (2명) | finding별 수용/기각 | [03-review-task.md](./03-review-task.md) |
| **코드 리뷰** | diff, 소스 파일 | CODE + TEST (2명) | AI 자동 수정 → 사람이 AS-IS/TO-BE diff 승인/반려 | [03-review-task.md](./03-review-task.md) |

### 1.1. 모드 선택 규칙

- **papillon 경유 시**: 파이프라인 Phase에 따라 자동 결정
  - XL PRD → PRD 리뷰
  - M 설계 → 설계 리뷰
  - Phase 2.1 ADR → ADR 리뷰
  - 4-B 런북 → 운영 절차 리뷰
  - 4-A 구현 후 → 코드 리뷰
- **단독 실행 시**: 오케스트레이터가 `$ARGUMENTS`의 대상을 분석하여 모드를 제안하고 사람이 승인. 전문가 조합도 추가/제거 가능

## 2. 의사결정 방식

### 2.1. finding별 수용/기각 (PRD / 설계 / ADR / 운영 절차 리뷰)

오케스트레이터가 finding을 심각도 순(P0→P1→P2→P3)으로 정렬하여 사용자에게 제시한다. 동일 주제에 대한 보완적 finding(충돌이 아닌 다른 관점)은 주제별로 묶어서 제시한다.

**P0/P1 처리:** 한 번에 하나씩 제시하고, 수용/일부수용/기각을 받는다.
- **일부수용** = finding의 문제 인식은 수용하되, 수정 범위나 방향을 사람이 조정. RDR에 "일부수용" + 조정 내용 기재.

**P2/P3 처리:** 일괄 제시 후 아래 중 선택:
- **전체 수용** / **전체 기각** / **건별 처리** (건별 선택 시 하나씩 수용/기각)

**수정 반영 (HITL, finding별 수용/기각 모드 공통):**
1. 라운드의 finding 전부 수용/기각 처리 완료
2. AI가 수용된 finding 기반으로 **설계 수정안을 제안** (수정 포인트 목록 또는 diff)
3. 사람이 확인 → 승인하면 AI가 반영 / 사람이 직접 반영 / 수정안 조정 요청
4. 사람이 반영 완료를 확인한 후 다음 라운드 시작

**"미해결"의 정의:**
- **미해결** = 수용/기각 판단이 안 된 건, 또는 수용했으나 설계 반영이 완료되지 않은 건
- **해결** = 수용 + 설계 반영 완료 (사람 확인), 또는 기각 (사유 기록)
- Phase 3 종결 조건: 모든 P0/P1이 해결 상태여야 함

### 2.2. AI 자동 수정 + diff 승인/반려 (코드 리뷰)

1. AI 전문가들이 **finding만 도출**한다 (수정안은 텍스트로 제안).
2. 오케스트레이터가 finding 간 코드 영향 범위를 분석하여 **diff 단위를 결정**한다:
   - 같은 파일/코드 블록을 건드리는 finding → 묶어서 하나의 diff
   - 독립된 finding → 개별 diff
3. 오케스트레이터가 묶음 단위로 **코드 수정을 실행**하여 AS-IS / TO-BE diff를 생성한다.
4. diff를 심각도 순(P0→P1→P2→P3)으로 사람에게 순차 제시한다. diff에 여러 심각도가 묶인 경우 최고 심각도 기준.
5. 사람이 diff를 확인하고 아래 중 선택:
   - **승인** — AI 수정안 채택. 다음 diff는 승인된 코드 기반으로 수정
   - **반려** — 사유 포함, AI가 사유를 반영하여 재수정 → 다시 diff 제시
   - **직접 수정** — finding 수용하되 사람이 직접 코드 수정. 수정 완료 후 AI가 최신 코드를 기반으로 다음 diff 진행
   - **기각** — finding 자체를 수용하지 않음. RDR에 기각 사유 기록. AI는 다음 diff로 진행

**코드 리뷰 → RDR "결정" 컬럼 매핑:**

| 코드 리뷰 액션 | RDR "결정" |
|---|---|
| 승인 | 수용 |
| 반려 후 최종 승인 | 수용 |
| 직접 수정 | 수용 |
| 기각 | 기각 |

**동일 diff 반려 3회 시 경고:**

> 동일 수정 사항에 대해 3회 반려되었습니다.
> 1. **직접 수정** — 사람이 직접 코드 수정
> 2. **기각** — 이 finding을 수용하지 않음
> 3. **계속** — AI에게 다시 시도

**코드 리뷰의 "라운드" 정의:**
- **1라운드** = 전문가가 전체 diff를 분석하여 finding 도출 → diff 순차 제시 → 전부 처리(승인/반려/직접수정/기각)
- **2라운드 이후** = 처리된 코드(승인+직접수정 반영된 최신 상태)에 대해 전문가가 다시 분석 → 신규 finding 도출
- 수렴 전환 조건 "직전 라운드 대비 신규 P0/P1이 0건"이 동일하게 적용됨

## 3. RDR / ADR 생성

RDR/ADR 포맷 및 저장 경로: [02-common-spec.md 섹션 2~4](../02-common-spec.md) 참조.

### 3.1. RDR 생성 규칙

- 리뷰 종료 시 자동으로 생성한다. 리뷰 1회 = RDR 1개.
- 5모드 모두에서 생성한다.
- 저장 경로(`docs/decisions/reviews/`)가 존재하지 않으면 자동 생성한다.

### 3.2. ADR 승격 규칙

**설계 리뷰 및 ADR 리뷰**에서 P0/P1 수용 건 중 **되돌리기 어렵고, 여러 모듈에 영향을 주거나, 향후 작업의 전례가 되는 아키텍처 결정**에 대해 승격을 제안한다. 사람이 승인하면 생성한다.

**승격 제안 시점:** finding 전부 수용/기각 처리 완료 후, ADR 승격 대상을 **일괄 제안**한다. finding 처리 도중에 끼어들지 않는다.

**ADR 초안 작성:** AI가 수용된 finding 기반으로 ADR 초안을 자동 생성한다. 사람이 확인/수정 후 확정하고, wtth ADR 리뷰 모드로 호출한다.

**승격 대상 예시 (이에 한정되지 않음):**
- 기술 스택 또는 라이브러리 선택/변경
- 모듈 간 통신 방식 변경
- 데이터 모델의 구조적 변경 (정규화/비정규화, 테이블 분리 등)
- 인증/인가 방식 변경
- 여러 태스크에 영향을 주는 패턴 결정

**승격 대상이 아닌 예시:**
- 특정 위치에 검증 로직 추가
- 단일 API의 에러 처리 보완
- 인덱스 추가/제거
- 단일 모듈 내부 리팩토링

### 3.3. 이전 결정 스캔

리뷰 시작 시 기존 RDR/ADR을 읽어서 이미 결정된 사항을 전문가에게 컨텍스트로 전달한다.

**스캔 범위:** 리뷰 대상이 속한 프로젝트의 `docs/decisions/` 전체. 단독 실행 시 파라미터로 경로가 전달되면 그대로 사용하고, 전달되지 않으면 사람에게 물어본다. 경로가 존재하지 않으면 이전 결정 없음으로 간주하고 스캔을 생략한다.

**전달 방식:** 전문가 에이전트 프롬프트에 이전 결정 목록을 포함하여, 동일 사안을 재지적하지 않도록 지시한다.

**재검토 예외:** RDR의 결정 근거에 조건부 기각이 명시된 경우 (예: "50만건 도달 시 재검토"), 현재 상태가 해당 조건을 충족하면 재지적을 허용한다.

**유사 finding 판단:** 동일 파일/동일 위치의 동일 이슈는 제외. 다른 파일/다른 위치에서 같은 패턴이 발견되면 신규 finding으로 취급한다.

## 4. 수렴 메커니즘

### 4.1. 라운드별 역할 분리

라운드 수를 고정하지 않고, **전환 조건**으로 모드를 전환한다.

| 모드 (내부 용어) | 규칙 | 다음 모드 전환 조건 |
|---|---|---|
| **발산** | 전체 지적 허용 (초기 상태) | 직전 라운드 대비 신규 P0/P1이 0건 → 수렴으로 전환. 또는 사람이 직접 지시 |
| **수렴** | 신규 P0/P1만 허용. 기존 건 재언급 금지 | 신규 finding이 0건 → 사인오프로 전환. 또는 사람이 직접 지시 |
| **사인오프** | 각 에이전트 "추가 지적 없음" 확인. P0/P1 발견 시 수렴으로 롤백, P2/P3은 RDR에 기록하고 사인오프 계속 | 전원 확인 → 리뷰 종료 |

**사람에게 보이는 메시지 (내부 용어를 노출하지 않는다):**
- 발산 → 수렴 전환 시: "새로운 P0/P1 지적이 없습니다. 이제부터 신규 P0/P1만 추가로 확인합니다."
- 수렴 → 사인오프 전환 시: "추가 지적이 없습니다. 각 전문가에게 최종 확인을 요청하겠습니다."
- 사인오프 완료 시: "전문가 전원이 추가 지적 없음을 확인했습니다. 리뷰를 종료할까요?"

**전문가 간 충돌 처리:**

동일 대상에 대해 전문가 간 상반된 finding이 발생하면(예: SEC "암호화 필요" vs SRE "암호화 시 검색 불가"), 오케스트레이터가 충돌 건을 묶어서 양쪽 근거와 함께 사람에게 제시한다. 사람이 방향을 결정한다. 전문가 간 debate는 수행하지 않는다 — 트레이드오프 결정은 비즈니스 맥락을 아는 사람의 몫이다.

**발산 3라운드 초과 시 경고 (내부 용어 노출 금지):**

발산 모드에서 3라운드를 초과하면 사람에게 아래 메시지와 선택지를 제시한다:

> 리뷰가 3라운드를 초과했으나 새로운 주요 지적이 계속 발생하고 있습니다.
>
> 1. **태스크 분할** — 현재 태스크를 쪼개고 파이프라인 재진입
> 2. **설계 초안 재작성** — 누적된 finding을 반영하여 초안부터 다시
> 3. **현재 지적 사항 처리에 집중** — 신규 지적 추가를 중단하고 기존 건만 마무리
> 4. **그대로 계속**

### 4.2. 1라운드 결과 분석 및 프로세스 분기

1라운드 종료 후 오케스트레이터가 **건수 + 심각도 분포 + 패턴**을 분석하여 사람에게 제안한다.

**분석 관점:**

| 관점 | 판단 내용 |
|---|---|
| **건수** | 전체 finding 수 |
| **심각도 분포** | P0/P1 비율이 높은가, P2/P3 위주인가 |
| **패턴 반복** | 동일 패턴의 반복인가 (→ 근본 원인 1건으로 병합 가능) |
| **도메인 분산** | 여러 도메인에 걸쳐있는가 (→ 태스크가 클 가능성) |

**제안 분기 (예시, 오케스트레이터가 맥락을 보고 판단):**

| 상황 | 제안 |
|---|---|
| finding 적고 P0/P1 없음 | "지적 N건이며 P0/P1이 없습니다. 리뷰를 종료할까요?" → 사람이 승인하면 사인오프 생략, 즉시 RDR 생성 후 리뷰 종료 |
| finding 많고 P0/P1 섞여있고 도메인 다양 | "지적이 N건이고 도메인이 분산되어 있습니다. 태스크 분할을 고려하세요." |
| finding 많지만 P2/P3 위주, 동일 패턴 반복 | "지적이 N건이지만 동일 패턴의 반복입니다. 근본 원인을 먼저 해결하면 대부분 해소될 것 같습니다." |
| finding 많고 P2/P3 위주, 전반적 품질 미달 | "지적이 N건이며 전반적인 품질 이슈입니다. 초안을 재작성하는 것이 효율적일 수 있습니다." |

오케스트레이터는 건수를 기계적으로 적용하지 않고, **why/how를 포함하여 제안**한다. 최종 판단은 사람이 한다.
