# Issue-03: 파이프라인 수정 경로 미정의

## 심각도: 높음

## 문제

현재 papillon 파이프라인은 **신규 설계서 작성**만 고려한다. Step 3(Sonnet 초안)은 인터뷰 결과를 입력받아 새 설계서를 생성하고, wtth은 그 설계서를 리뷰한다.

그런데 실제 작업에서는 **기존 설계서를 수정**하는 케이스가 빈번하다:
- 리뷰 후 액션 아이템 반영 (묶음 B: 00-core.md 수정)
- 이슈 해결을 위한 기존 설계서 보강
- 다른 스킬 설계 변경에 따른 연쇄 수정

이 경우 Step 3의 "초안 작성"과 wtth의 리뷰 포인트가 달라져야 하나, 현재 설계에는 수정 경로가 존재하지 않는다.

## 영향

- Sonnet이 "새로 만들기"와 "기존 수정"을 구분 없이 처리하면 기존 맥락을 훼손할 수 있다
- wtth 전문가가 수정 맥락(뭘 왜 바꿨는지) 없이 전체 문서만 보면, 수정-기존 간 정합성 검증이 누락된다
- 묶음 B에서 실제로 발생한 문제: 인터뷰 결과를 기반으로 기존 00-core.md를 수정해야 하는데, 파이프라인에 해당 경로가 없었음

## 솔루션

### 1. 파이프라인에 수정 경로 추가

| | 신규 경로 | 수정 경로 |
|---|---|---|
| Step 3 입력 | 인터뷰 결과 | 인터뷰 결과 + 기존 설계서 |
| Step 3 산출물 | 새 설계서 초안 | 수정된 설계서 + **수정 매니페스트** |
| wtth 입력 | 설계서 전문 | 설계서 전문 + 수정 매니페스트 |
| 리뷰 핵심 | 완결성, 구조, 명세 품질 | 위 전부 + **수정이 기존 논리/흐름/맥락에 녹아드는가** |

### 2. 수정 매니페스트

Step 3(Sonnet)이 수정 완료 후 자동 생성하는 문서. 전문가에게 "뭐가 바뀌었는지"를 알려주는 역할.

포함 내용:
- 변경 섹션 목록 (섹션 번호 + 변경 유형: 신규/수정/삭제)
- 각 변경의 근거 (어떤 인터뷰 결과/액션 아이템에서 기인)
- 영향 범위 추정 (이 변경이 다른 섹션/문서에 미치는 영향)

### 3. wtth 수정 경로 리뷰 포인트

전문가에게 추가로 지시하는 체크 포인트:
- 수정된 섹션이 수정되지 않은 섹션과 논리적으로 일관되는가
- 수정으로 인해 기존 동작/흐름이 변경되는 경우, 그 변경이 의도된 것이고 전체 맥락에서 타당한가
- 수정 매니페스트에 명시된 영향 범위가 실제로 커버되었는가 (빠진 연쇄 수정이 없는가)
- 수정 전후로 용어/개념의 일관성이 유지되는가
