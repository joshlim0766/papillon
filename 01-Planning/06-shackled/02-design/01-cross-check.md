## 1. 기본 원칙

크로스 체커 페르소나는 설계서(또는 요구사항)를 펼쳐두고 구현 결과를 대조한다. "잘 만들었는가"가 아니라 **"설계대로 만들었는가"**에 집중한다.

<rationale>
"잘 만들었는가"를 배제하는 이유: 코드 품질·보안·성능 등 "잘 만들었는가"는 wtth(제3자 리뷰)의 영역이다. 크로스 체크가 이 영역까지 확장하면 wtth과 역할이 중복되고, 내부 크로스 체크가 무거워져 구현 속도가 저하된다.
</rationale>

---

## 2. 심각도 체계

크로스 체크 finding에도 공통 규격(§1)의 P0~P3 심각도를 적용한다. 크로스 체크의 관점("설계대로 만들었는가")에 맞는 매핑:

| 심각도 | 기준 | 예시 |
|---|---|---|
| **P0** | 설계 인터페이스 위반 — 명시된 계약과 구현이 불일치 | API 시그니처 변경, 데이터 모델 불일치 |
| **P1** | 명시된 로직 누락 — 설계에 정의된 처리가 구현에 없음 | 에러 처리 누락, 필수 검증 미구현 |
| **P2** | 제약조건 미반영 — 설계의 비기능 요구가 누락됨 | 성능 제약 미반영, 로깅 누락 |
| **P3** | 경미한 불일치 — 동작에 영향 없음 | 네이밍 컨벤션 불일치, 주석 누락 |

---

## 3. 체크 항목 (code)

1. 설계서에 명시된 인터페이스(API, 함수 시그니처 등)와 구현이 일치하는가
2. 설계서에 명시된 데이터 흐름·처리 로직이 구현에 반영되었는가
3. 설계서에 명시된 제약 조건(인증, 권한, 유효성 검사 등)이 누락되지 않았는가
4. 설계서에서 정의된 에러 처리 방식이 반영되었는가
5. 이 태스크가 연관된 다른 태스크와의 인터페이스가 충돌하지 않는가

---

## 4. 체크 항목 (runbook)

1. 설계서(또는 요구사항)에서 정의한 작업 목표가 절차에 반영되었는가
2. 각 단계에 검증 커맨드 또는 성공 판단 기준이 포함되어 있는가
3. 롤백 절차가 존재하는가
4. 실행 전제 조건(권한, 환경, 의존 서비스 상태 등)이 명시되어 있는가
5. 단계 순서가 실행 가능한 순서인가 (의존 관계 위반 없음)

---

## 5. finding 처리

크로스 체크 finding은 shackled 내부에서 자동 해소한다. 단, 이력은 투명하게 공개한다.

finding이 있으면:
1. 크로스 체커가 심각도와 함께 구체적으로 지적한다: "§2.3에서 요구한 X 처리가 누락되었습니다 (P1)"
2. 작성자 페르소나가 수정한다.
3. 크로스 체커가 재점검한다.
4. finding이 없어지면 사람 확인 단계로 진행한다.

**심각도별 차등 처리:**
- P0/P1: 반드시 해소해야 사람 확인 단계로 진행
- P2/P3: 해소를 시도하되, 3회 수정 후에도 미해소 시 주석으로 기록하고 진행 허용

**반복 상한 (개별 finding):** 동일 finding 1건에 대해 작성자가 3회 수정을 시도했으나 크로스 체커가 여전히 해당 finding을 지적하는 경우:
- P0/P1: 진행 불가 처리([03-blocked-handling.md](./03-blocked-handling.md))로 분기
- P2/P3: 주석 기록 후 진행

**라운드 상한 (전체):** 크로스 체크 전체 라운드가 5회를 초과하면 (매 라운드 새로운 finding이 계속 발생하는 경우), 진행 불가 처리([03-blocked-handling.md](./03-blocked-handling.md))로 분기하거나 사람에게 현황을 보고하고 계속/중단 선택지를 제시한다. 코어 프로세스(00-design-shackled.md §3.3.3) 사람 확인 시 "크로스 체크 N라운드 수행"을 표시하여 내부 반복 횟수를 투명하게 공개한다.

---

## 6. 변천사 로그

크로스 체크 이력을 상태 파일([02-state-management.md](./02-state-management.md))의 `cross_check_log` 배열에 누적 기록한다.

**기록 내용 (라운드별):**
- finding 내용 + 심각도
- 수정 내용
- 해소 여부

**제시 방식:**
- **기본 (코어 §3.3.3)**: 수정 요약만 — "크로스 체크 N회, finding M건 해소" + 방향 변경이 있었으면 변경 전/후 1줄 요약
- **상세 (사람 요청 시)**: 전체 라운드별 finding → 수정 → 해소 이력
