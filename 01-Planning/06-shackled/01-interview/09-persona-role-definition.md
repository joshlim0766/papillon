# 09. 페르소나 역할 정의

**상태:** Settled

**출처:** AI-4 (shackled 전용 페르소나 4축 정의)

---

## 확정 사항

### 공통 원칙

- shackled 페르소나는 wtth 페르소나와 이름이 같지만 역할이 다르다 (wtth: "이 설계가 맞나?" vs shackled: "설계대로 만들었나?")
- 작성자 페르소나(CODE, SRE)는 `_schema.md` 구조를 **셀프 체크** 관점으로 적용한다 — 코드/런북을 작성한 뒤 자기 산출물을 자가 검증
- shackled 크로스 체크와 wtth 코드 리뷰(Phase 5)의 겹침은 **의도된 이중 체크**로 허용한다. 동일한 내용을 다른 관점(설계 정합성 vs 코드 품질)으로 보는 것이 목적이며, 중복 finding은 감수한다

### 페르소나별 역할

| 페르소나 | 역할 | 관점 핵심 |
|---|---|---|
| **shackled-CODE** | 코드 작성자 | 설계서 명세 기반 구현 + 셀프 체크 (버그, 엣지케이스, 리소스) |
| **shackled-BE** | 백엔드 크로스 체커 | 설계서의 BE 명세(API 계약, 트랜잭션, 데이터 모델)가 구현에 반영됐는가 |
| **shackled-FE** | 프론트엔드 크로스 체커 | 설계서의 FE 명세(UI 흐름, 상태 관리, API 연동)가 구현에 반영됐는가 |
| **shackled-SRE** | 런북 작성자 | 절차서 + 인프라 컨텍스트 기반 실행 가능한 런북 작성 + 셀프 체크 (절차 완결성, 롤백, 검증 커맨드) |
| **shackled-SEC** | 런북 크로스 체커 (DevSecOps) | 보안(권한, 시크릿, 감사) + 절차 안전성(실행 순서, 롤백, Step 간 상태 일치) |

### 핵심 결정 근거

- **CODE 셀프 체크**: 코드를 작성하고 자기 코드를 리뷰하지 않는 사람은 없음 — 자연스러운 관점
- **이중 체크 허용**: 크로스 체크는 서로의 접근 패턴을 이용해서 합치하는 건 합치, 다른 건 지적을 통해 수용해나가는 과정
- **shackled-SEC = DevSecOps**: 순수 SEC로는 절차 안전성(Step 간 상태 일치, 실행 순서)을 판단할 도메인 지식이 부족. DevSecOps / 클라우드 보안 엔지니어는 인프라 절차를 이해하면서 보안 관점으로 검증하는 실제 존재하는 도메인 전문가
- **SRE 셀프 체크 범위**: 입력 데이터(사람의 지침) 자체의 오류는 체크리스트로 잡을 수 없지만, "지침에 명시된 리소스의 존재 여부 검증 커맨드 포함" 수준으로 방어 가능

### 실례 참조

- `megabird-infra/migration-execute-plan`: 구체화된 절차서 + 인프라 컨텍스트로 작성된 런북. 사전 점검 누락 1건(가정한 DB 테이블 미존재 — 사람의 점검 지침 오류) 외 트러블 없음
