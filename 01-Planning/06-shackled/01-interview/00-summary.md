# Inquisition Summary

## 요구사항 요약
- papillon Phase 4(구현)를 독립 스킬 "shackled"로 분리
- 4-A(코드 작업)와 4-B(인프라/절차 작업)를 하나의 스킬로 통합 — 프로세스 골격 공유, 산출물 유형 분기
- 페어 프로그래밍 스킬: 설계서/요구사항 기반으로 AI가 구현하고 사람이 방향을 잡는 협업 도구
- `/papillon:shackled`로 단독 실행 가능, 오케스트레이터 경유도 가능

## 핵심 제약
- 설계서 또는 요구사항 정의서 필수 입력 — 없으면 진행 거부
- S/M 체급만 진입 (L/XL은 하위 태스크로 분해되어 진입)
- 시블링 구조 — shackled은 구현만, wtth 호출은 오케스트레이터 책임
- 태스크 하나씩 순차 구현 (한번에 전부 금지)

## 암시적 전제 조건
- 설계서가 inquisition + wtth 리뷰를 거쳐 확정된 상태에서 진입하는 것이 정상 흐름
- 코딩 전문가 에이전트 활용 필수 (기본 모드에서도)
- 진행 불가 시 롤백 지점은 사람이 결정 — AI는 원인 힌트만 제공

## Deferred 항목
| 항목 | 사유 | 결정 조건 |
|---|---|---|
| 페어 모드 (tmux 분할) | 환경 제약 — tmux + 실험 플래그 필요 | 기본 모드 안정화 후 구현 |
| 페어 모드 (IDE 연동) | JetBrains MCP 한계 — 포커스 전환 불가 | IDE MCP 기능 확장 시 재검토 |

## 체급 판정
- **규모:** 규모가 어느 정도 있는 작업입니다. 구현 계획을 세워봅시다.
- **체급:** M
- **근거:** 단일 관심사(구현 스킬) 내에서 설계 판단이 필요함 — 모드 분기(코드/절차), 페르소나 설계, 리뷰 계층화, 진행 불가 처리 등 여러 설계 결정 포함. 하나의 세션에서 설계서 작성까지 완결 가능.
- **다음 단계:** 설계서 작성 → wtth 설계 리뷰 → 스킬 프롬프트 구현

## 추가 인터뷰: 전문가 풀 거버넌스 (2026-04-08)

**출처:** [01-action-items.md](../03-review/01-action-items.md) — AI-1, AI-2, AI-2-1, 미등록(오케스트레이터 주제별 묶음 처리)
**리뷰 기록:** [RDR-2026-04-07-design-shackled.md](../03-review/RDR-2026-04-07-design-shackled.md)

### 요구사항 요약
- AI-1: 설계+운영절차 복합 적용 시 SPEC/IX 투입 가이드
- AI-2: SPEC/IX 페르소나 정식 등록 + 투입 조건
- AI-2-1: 전문가 풀 제한 메커니즘 (3인 상한, excluded_experts, 재추천 금지)
- 미등록: 오케스트레이터 주제별 묶음 처리
- 수정 대상: `04-wtth/02-design/00-core.md` §1~§2, `personas/spec.md`, `personas/ix.md`, `personas/_schema.md`

### 핵심 제약
- "워크플로우 리뷰"는 정식 모드가 아님. 별도 모드 신설 불가 — 기존 모드 조합 + 전문가 풀 커스텀
- 복합 적용 판별: 오케스트레이터 제안 + 사람 확정 (자동 판별 아님)
- 설계 리뷰의 6인 풀은 "뽑아 쓰는 풀"이지 전원 투입이 아님
- 기본 풀 상한 3인 — 초과 시 사람 확인 필수
- excluded_experts는 리뷰 세션(RDR 단위) 기준 유지, 클로드 세션 기준 아님
- 묶음 판단 기준: 동일 관심사(섹션) 기준 group by

### 암시적 전제 조건
- SPEC/IX는 복합 적용 전용이 아님. 개별 모드 단독 투입 가능 (SPEC: 복잡한 API 명세, IX: 복잡한 런북)
- 설계와 운영 절차의 경계가 모호한 문서 유형이 존재하며 정밀한 자동 판별 기준을 세우기 어려움
- 기존 모드별 기본 풀(PRD 3인, ADR 2인, 운영절차 2인)은 이미 3인 이하 — 설계 리뷰만 예외
- AI-1 → AI-2 → AI-2-1 순서 의존 존재

### Deferred 항목
| 항목 | 사유 | 결정 조건 |
|---|---|---|
| excluded_experts 영속 메커니즘 | 저장 위치/형식은 설계 단계 결정 | 00-core.md에서 리뷰 상태 관리 방식 확정 시 |
| 묶음 처리 제시 형식 | 기준만 확정, 형식은 설계 단계 | 설계 초안에서 구체화 |

### 체급 판정
- **규모:** 규모가 어느 정도 있는 작업입니다. 구현 계획을 세워봅시다.
- **체급:** M
- **근거:** 4개 아이템이 "전문가 풀 거버넌스"라는 단일 관심사 안의 설계 판단. 한 세션 완결 가능.
- **다음 단계:** 구현 계획 → 설계 초안 작성
