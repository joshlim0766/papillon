# AI-4: Shackled 진행 불가 시 옵션 기반 대응

**심각도:** P2
**분류:** 공백
**상태:** Open

## 문제

`03-blocked-handling.md` §2의 보고 형식이 "가능한 다음 행동" 선택지를 제시하지만, 현재는 §3의 경미/중간/심각 수준 분류만 있고, 구체적인 기술적 대응 옵션은 사람이 직접 고안해야 한다. 설계 공백이 사소한 경우에도 진행 불가 → 설계 수정 사이클을 돌면 오버헤드가 크다.

## 솔루션: 보고 형식에 AI 추론 옵션 필수 포함

- **수정 대상:** `01-Planning/06-shackled/02-design/03-blocked-handling.md` §2(보고 형식)
- **변경 내용:**
  1. 보고 형식의 "가능한 다음 행동" 섹션에 **"AI가 추론한 대응 옵션 2~3개"**를 필수 필드로 추가
  2. 각 옵션에 **트레이드오프(장점/단점/영향 범위)**를 포함
  3. **옵션 제시의 주체는 shackled 오케스트레이터**(진행 불가를 보고하는 주체)이며, 특정 페르소나(BE/FE 등)가 가설을 수립하는 것이 아님. 역할 경계 유지
  4. 사람이 옵션 중 하나를 선택하거나, 옵션 외의 다른 방향을 지시할 수 있음

<rationale>
BE 크로스 체커가 가설 주체가 되면 안 되는 이유: shackled-BE의 금기에 "설계서 자체에 대한 이의 제기"가 있다. 가설 수립은 사실상 설계 결정이므로 역할 경계 위반이다. 옵션 나열("이런 길이 있습니다")은 정보 제공이고, 가설 결정("이 길로 가겠습니다")은 설계 행위다. 전자만 허용한다.
</rationale>
