## 1. 5모드 리뷰 체계

review-hell은 리뷰 대상 유형에 따라 전문가 풀과 의사결정 방식을 자동 전환한다.

| 모드 | 대상 | 기본 전문가 풀 | 의사결정 방식 |
|---|---|---|---|
| **PRD 리뷰** | PRD, 기획 문서 | PM + ARCH + SEC (3명) | finding별 수용/기각 |
| **설계 리뷰** | 설계 문서, 명세서 | BE/FE/SRE/SEC/PO (기존) | finding별 수용/기각 |
| **ADR 리뷰** | ADR (아키텍처 결정 기록) | ARCH + SEC (2명) | finding별 수용/기각 |
| **운영 절차 리뷰** | 런북, 절차서 | SRE + SEC (2명) | finding별 수용/기각 |
| **코드 리뷰** | diff, 소스 파일 | CODE + TEST (2명) | AI 자동 수정 → 사람이 AS-IS/TO-BE diff 승인/반려 |

### 1.1. 모드 선택 규칙

- **work-flow 경유 시**: 파이프라인 Phase에 따라 자동 결정
  - XL PRD → PRD 리뷰
  - M 설계 → 설계 리뷰
  - Phase 2.1 ADR → ADR 리뷰
  - 4-B 런북 → 운영 절차 리뷰
  - 4-A 구현 후 → 코드 리뷰
- **단독 실행 시**: 오케스트레이터가 `$ARGUMENTS`의 대상을 분석하여 모드를 제안하고 사람이 승인. 전문가 조합도 추가/제거 가능

## 2. 전문가 풀 구성

각 전문가의 상세 정의(관점/금기/판단 기준/체크리스트)는 [51-expert-definitions.md](./51-expert-definitions.md) 참조.

| 모드 | 기본 풀 | 선택적 투입 |
|---|---|---|
| **PRD 리뷰** | PM, ARCH, SEC | — |
| **설계 리뷰** | BE, FE, SRE, SEC, PO | DBA (오케스트레이터가 제안 톤으로 추가 투입을 물어봄) |
| **ADR 리뷰** | ARCH, SEC | — |
| **운영 절차 리뷰** | SRE, SEC | — |
| **코드 리뷰** | CODE, TEST | ARCH, SEC (단독 실행 시 오케스트레이터가 제안 가능) |

## 3. 의사결정 방식

### 3.1. finding별 수용/기각 (PRD / 설계 / 운영 절차 리뷰)

기존 review-hell과 동일. 정렬된 finding을 한 번에 하나씩 사용자에게 제시하고, 수용/일부수용/기각을 받는다.

### 3.2. AI 자동 수정 + diff 승인/반려 (코드 리뷰)

1. AI 전문가들이 지적 사항을 도출하고, **수정까지 자동 수행**한다.
2. 수정 결과를 **AS-IS / TO-BE diff 전문**으로 사람에게 출력한다.
3. 사람이 diff를 확인하고 **승인 또는 반려** (반려 시 사유 포함).
4. 반려 시 사유를 반영하여 재수정 → 다시 diff 제시 → 승인될 때까지 반복.

## 4. RDR / ADR 생성

RDR/ADR 포맷 및 저장 경로: [02-common-spec.md 섹션 2~4](./02-common-spec.md) 참조.

### 4.1. RDR 생성 규칙

- 리뷰 종료 시 자동으로 생성한다. 리뷰 1회 = RDR 1개.
- 4모드 모두에서 생성한다.

### 4.2. ADR 승격 규칙

P0/P1 수용 건 중 **되돌리기 어렵고, 여러 모듈에 영향을 주거나, 향후 작업의 전례가 되는 아키텍처 결정**에 대해 승격을 제안한다. 사람이 승인하면 생성한다.

**승격 대상 예시 (이에 한정되지 않음):**
- 기술 스택 또는 라이브러리 선택/변경
- 모듈 간 통신 방식 변경
- 데이터 모델의 구조적 변경 (정규화/비정규화, 테이블 분리 등)
- 인증/인가 방식 변경
- 여러 태스크에 영향을 주는 패턴 결정

**승격 대상이 아닌 예시:**
- 특정 위치에 검증 로직 추가
- 단일 API의 에러 처리 보완
- 인덱스 추가/제거
- 단일 모듈 내부 리팩토링

### 4.3. 이전 결정 스캔

리뷰 시작 시 기존 RDR/ADR을 읽어서 이미 결정된 사항을 전문가에게 컨텍스트로 전달한다.

**스캔 범위:** 리뷰 대상이 속한 프로젝트의 `docs/decisions/` 전체.

**전달 방식:** 전문가 에이전트 프롬프트에 이전 결정 목록을 포함하여, 동일 사안을 재지적하지 않도록 지시한다.

**재검토 예외:** RDR의 결정 근거에 조건부 기각이 명시된 경우 (예: "50만건 도달 시 재검토"), 현재 상태가 해당 조건을 충족하면 재지적을 허용한다.

**유사 finding 판단:** 동일 파일/동일 위치의 동일 이슈는 제외. 다른 파일/다른 위치에서 같은 패턴이 발견되면 신규 finding으로 취급한다.

## 5. 수렴 메커니즘

### 5.1. 라운드별 역할 분리

라운드 수를 고정하지 않고, **전환 조건**으로 모드를 전환한다.

| 모드 (내부 용어) | 규칙 | 다음 모드 전환 조건 |
|---|---|---|
| **발산** | 전체 지적 허용 (초기 상태) | 직전 라운드 대비 신규 P0/P1이 0건 → 수렴으로 전환. 또는 사람이 직접 지시 |
| **수렴** | 신규 P0/P1만 허용. 기존 건 재언급 금지 | 신규 finding이 0건 → 사인오프로 전환. 또는 사람이 직접 지시 |
| **사인오프** | 각 에이전트 "추가 지적 없음" 확인 | 전원 확인 → 리뷰 종료 |

**사람에게 보이는 메시지 (내부 용어를 노출하지 않는다):**
- 발산 → 수렴 전환 시: "새로운 P0/P1 지적이 없습니다. 이제부터 신규 P0/P1만 추가로 확인합니다."
- 수렴 → 사인오프 전환 시: "추가 지적이 없습니다. 각 전문가에게 최종 확인을 요청하겠습니다."
- 사인오프 완료 시: "전문가 전원이 추가 지적 없음을 확인했습니다. 리뷰를 종료할까요?"

### 5.2. 1라운드 결과 분석 및 프로세스 분기

1라운드 종료 후 오케스트레이터가 **건수 + 심각도 분포 + 패턴**을 분석하여 사람에게 제안한다.

**분석 관점:**

| 관점 | 판단 내용 |
|---|---|
| **건수** | 전체 finding 수 |
| **심각도 분포** | P0/P1 비율이 높은가, P2/P3 위주인가 |
| **패턴 반복** | 동일 패턴의 반복인가 (→ 근본 원인 1건으로 병합 가능) |
| **도메인 분산** | 여러 도메인에 걸쳐있는가 (→ 태스크가 클 가능성) |

**제안 분기 (예시, 오케스트레이터가 맥락을 보고 판단):**

| 상황 | 제안 |
|---|---|
| finding 적고 P0/P1 없음 | "지적 N건이며 P0/P1이 없습니다. 리뷰를 종료할까요?" (간소 프로세스) |
| finding 많고 P0/P1 섞여있고 도메인 다양 | "지적이 N건이고 도메인이 분산되어 있습니다. 태스크 분할을 고려하세요." |
| finding 많지만 P2/P3 위주, 동일 패턴 반복 | "지적이 N건이지만 동일 패턴의 반복입니다. 근본 원인을 먼저 해결하면 대부분 해소될 것 같습니다." |
| finding 많고 P2/P3 위주, 전반적 품질 미달 | "지적이 N건이며 전반적인 품질 이슈입니다. 초안을 재작성하는 것이 효율적일 수 있습니다." |

오케스트레이터는 건수를 기계적으로 적용하지 않고, **why/how를 포함하여 제안**한다. 최종 판단은 사람이 한다.
