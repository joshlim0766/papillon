## 1. 5모드 리뷰 체계

wtth은 리뷰 대상 유형에 따라 전문가 풀과 의사결정 방식을 자동 전환한다.

| 모드 | 대상 | 기본 전문가 풀 | 선택적 투입 | 의사결정 방식 | 상세 |
|---|---|---|---|---|---|
| **PRD 리뷰** | PRD, 기획 문서 | PM + ARCH + SEC (3명) | BE, SRE | finding별 수용/기각 | [01-review-prd.md](./01-review-prd.md) |
| **설계 리뷰** | 설계 문서, 명세서 | ARCH/BE/FE/SRE/SEC/PO | DBA | finding별 수용/기각 | [02-review-design.md](./02-review-design.md) |
| **ADR 리뷰** | ADR (아키텍처 결정 기록) | ARCH + SEC (2명) | BE, DBA, SRE | finding별 수용/기각 | [02-review-design.md](./02-review-design.md) |
| **운영 절차 리뷰** | 런북, 절차서 | SRE + SEC (2명) | DBA | finding별 수용/기각 | [03-review-task.md](./03-review-task.md) |
| **코드 리뷰** | diff, 소스 파일 | CODE + TEST (2명) | ARCH, SEC | AI 자동 수정 → 사람이 AS-IS/TO-BE diff 승인/반려 | [03-review-task.md](./03-review-task.md) |

### 1.1. 모드 선택 규칙

- **papillon 경유 시**: 파이프라인 Phase에 따라 자동 결정
  - XL PRD → PRD 리뷰
  - M 설계 → 설계 리뷰
  - Phase 2.1 ADR → ADR 리뷰
  - 4-B 런북 → 운영 절차 리뷰
  - 4-A 구현 후 → 코드 리뷰
  - 리뷰 대상이 복수 모드에 걸치면 오케스트레이터가 복합 적용을 제안하고 사람이 확정한다.
- **단독 실행 시**: 오케스트레이터가 `$ARGUMENTS`의 대상을 분석하여 모드를 제안하고 사람이 승인. 전문가 조합도 추가/제거 가능. 리뷰 대상이 복수 모드에 걸치면 오케스트레이터가 복합 적용을 제안하고 사람이 확정한다. 복합 적용 시에는 §1.2 확정 흐름이 모드 승인 단계를 대체한다.

### 1.2. 복합 모드 적용

리뷰 대상이 단일 모드에 맞지 않을 때 복수 모드를 복합 적용할 수 있다. 예를 들어 워크플로우와 모듈 구조를 함께 다루는 설계서, 또는 사람 확인 절차가 포함된 런북이 이에 해당한다.

**복합 적용 판별:** 오케스트레이터가 아래 시그널을 기반으로 복합 적용을 제안하고, 사람이 최종 확정한다.

| 시그널 | 설명 |
|---|---|
| 절차 + 구조 공존 | 프로세스 단계(절차)와 모듈 구조(설계)가 한 문서 안에 함께 기술되어 있다 |
| 사람-AI 상호작용 포함 | 사람 확인 지점, 판단 분기, 개입 절차가 설계 요소로 포함되어 있다 |
| 실행 명세 + 구조 설계 겸함 | 산출물이 코드가 아닌 런북·파이프라인 정의와 같은 실행 명세이면서 구조 설계를 동시에 기술한다 |
| 워크플로우·프로세스 명세가 설계서에 핵심적으로 포함 | API 명세에 복잡한 분기 조건이 있거나, 설계서가 운영 절차를 겸하는 경우 |

**복합 적용 확정 흐름:**
1. 오케스트레이터가 복합 모드를 제안하면서 시그널 근거를 함께 제시한다.
2. 사람이 복합 모드를 확정한다.
3. 오케스트레이터가 선별된 커스텀 전문가 풀(이름·도메인·담당 관심사)을 제시한다.
4. 사람이 풀을 조정하거나 승인한다.

**전문가 선별 순서:** (1) 각 관련 모드 풀을 합산 (기본 풀 + 선택적 투입 전체) → (2) 투입 조건 필터 적용 → (3) 추천 상한 3인 내로 압축. 투입 조건은 리뷰 대상 전체를 대상으로 평가한다 — 복합 모드 중 하나라도 투입 조건을 충족하면 투입 대상으로 간주한다 (OR 조건).

**복합 모드 풀 제시 포맷 (확정 흐름 3단계):**

```
| 전문가 | 도메인 | 담당 관심사 |
|---|---|---|
| ARCH | 아키텍처 | §2 모듈 구조 |
| SRE  | 운영     | §4 런북 절차 |

→ 제외할 전문가가 있으면 알려주세요. 없으면 승인으로 진행합니다.
```

**복합 적용 시 규칙:**
- §1.3 추천 상한 3인 규칙이 동일하게 적용된다.
- excluded_experts는 모드 전환 후에도 동일 리뷰 세션(RDR 1개 기준) 내에서 유지된다.
- 복합 모드에서도 RDR은 리뷰 전체로 1개 생성한다 (§3.1 "리뷰 1회 = RDR 1개" 동일 적용).
- 코어 프로세스(finding 기반 수용/기각, 수렴 메커니즘)는 변경 없이 그대로 적용한다.
- 묶음 처리(§2.1.1) 시, 복합 모드에서는 동일 섹션에 대한 각 모드 관점의 finding을 하나의 묶음으로 구성하되 묶음 헤더에 관련 모드를 표기한다 (예: `[묶음: §3 런북 절차 | 운영 절차 + 설계 관점]`).

### 1.3. 전문가 투입 거버넌스

**용어:** "리뷰 세션"은 RDR 1개 단위(wtth 호출 1회)를 가리키며, 대화 컨텍스트 윈도우와 무관하다.

1. **추천 상한 3인**: 오케스트레이터가 한 리뷰에 추천하는 전문가는 최대 3인이다. 3인 초과 추천이 필요하다고 판단하면 사람에게 확인한다 (복합 모드에서는 §1.2의 선별 순서 3단계로 3인 압축을 실행한다). 모드별 기본 풀(설계 리뷰 6인 등)은 "뽑아 쓰는 풀"이며 전원 투입이 아니다. 3인 초과 확인 시 사람에게 제시하는 정보: (a) 추가 전문가 이름·도메인, (b) 투입 근거(투입 조건 충족 사유), (c) 추가하지 않을 경우 예상 커버리지 공백. 선택지: 승인 / 제외할 전문가 선택.

2. **투입 조건 기반 추천**: 페르소나 파일에 `투입 조건`이 명시된 전문가는 해당 조건을 충족하는 리뷰에서만 추천한다. 오케스트레이터가 리뷰 대상을 분석하여 조건 충족 여부를 판단하고 제안, 사람이 확정한다. **경계 사례** — 투입 조건 키워드(워크플로우·프로세스 명세, 사람-AI 상호작용 등)가 리뷰 대상에 존재하지만 핵심 관심사가 아닌 경우 — 는 사람에게 확인한다. 조건이 없는 전문가는 해당 모드의 기본 풀에 항상 포함된다.

3. **사람 제외**: 사람이 전문가를 제외하면 확인 메시지를 제시한다 ("[전문가명] 제외합니다. 확정하시겠습니까?"). 확정 후 해당 리뷰 세션 내에서 제외된 전문가를 재추천하지 않는다. 사람은 리뷰 중 언제든 제외를 해제하고 재투입을 요청할 수 있다.

4. **excluded_experts 영속**: 리뷰 상태를 파일로 저장하여 대화 세션 간 제외 상태를 유지한다.
   - **저장 위치**: 리뷰 대상 파일이 위치한 디렉토리의 `.wtth-review-state.json`
   - **생명주기**: 리뷰 종료(RDR 생성) 시 해당 파일을 삭제한다.
   - **조회 시점**: 리뷰 시작 시 상태 파일 존재 여부를 확인한다. 존재하면 이전 대화 세션이 중단된 것으로 판단하고 사람에게 이어받을지 확인한다. 이어받으면 기존 excluded_experts를 유지하고, 새로 시작하면 파일을 초기화한다. 리뷰 재개 시 현재 상태 요약을 먼저 제시한다: (a) 현재 라운드, (b) 처리된/미처리 finding 수, (c) 제외된 전문가 목록. 이 요약에서 excluded_experts를 재확인하고 해제 기회를 준다.
   - **갱신 시점**: 라운드 변경 시마다 `round` 필드를 갱신한다. 전문가 제외/재투입 시 `excluded_experts`를 갱신한다.

**단일 모드 예시:**

```json
{
  "review_target": "path/to/design.md",
  "mode": ["설계 리뷰"],
  "experts": ["ARCH", "SPEC", "IX"],
  "excluded_experts": ["BE"],
  "round": 2,
  "last_updated": "2026-04-08"
}
```

**복합 모드 예시:**

```json
{
  "review_target": "path/to/workflow-design.md",
  "mode": ["설계 리뷰", "운영 절차 리뷰"],
  "experts": [
    {"id": "ARCH", "modes": ["설계 리뷰"]},
    {"id": "SRE", "modes": ["운영 절차 리뷰"]},
    {"id": "SPEC", "modes": ["설계 리뷰", "운영 절차 리뷰"]}
  ],
  "excluded_experts": [],
  "round": 1,
  "last_updated": "2026-04-08"
}
```

`mode` 허용값: `"PRD 리뷰"`, `"설계 리뷰"`, `"ADR 리뷰"`, `"운영 절차 리뷰"`, `"코드 리뷰"` (§1 테이블의 모드명과 동일). 단일 모드도 배열로 저장한다.

## 2. 의사결정 방식

### 2.1. finding별 수용/기각 (PRD / 설계 / ADR / 운영 절차 리뷰)

오케스트레이터가 finding을 심각도 순(P0→P1→P2→P3)으로 정렬하여 사용자에게 제시한다.

#### 2.1.1. 묶음 처리

동일 관심사(설계서의 섹션 단위)를 다루는 finding을 묶어서 제시한다. 충돌이 아닌 보완적 관점(서로 다른 전문가가 같은 섹션을 각자의 관점에서 지적)을 하나의 묶음으로 구성한다.

**묶음 판단 기준:**
- 섹션 번호가 동일한 finding은 묶는다 (기본 기준).
- 서로 다른 섹션의 finding이라도 오케스트레이터가 동일 관심사로 판단하면 묶을 수 있으며, 이 경우 묶음 헤더에 두 섹션을 모두 표기한다.
- 하나의 finding이 2개 이상 섹션에 걸치면 주요 관심사 기준으로 분류한다. 주요 관심사 판별 우선순위: finding이 명시적으로 참조하는 섹션 > 수정안이 가장 많이 영향을 미치는 섹션. 관련 섹션을 참조로 표기한다.
- 충돌하는 finding(예: SEC "암호화 필요" vs SRE "암호화 시 검색 불가")은 묶음이 아닌 충돌 건으로 별도 처리한다(§4.1 참조).

**묶음 제시 형식:**

```
[묶음: §{섹션번호} {섹션명}]

[이름(도메인ID)] P{N}: {제목}
  문제: ...
  왜 문제인가: ...
  참조: ...
  수정안: ...

[이름(도메인ID)] P{N}: {제목}
  문제: ...
  왜 문제인가: ...
  참조: ...
  수정안: ...

→ 이 묶음에 대한 결정을 내려주세요.
```

묶음 내 finding은 심각도 순(P0→P3)으로 정렬한다. 묶음 자체의 심각도는 묶음 내 최고 심각도 기준이다.

**묶음 처리 방식:**
- **P0/P1 묶음**: 묶음 단위로 제시하고, 묶음 내 각 finding에 대해 개별 수용/기각을 받는다. 묶음의 finding을 모두 처리한 후 다음 묶음으로 이동한다.
- **P2/P3 묶음**: 아래 P2/P3 처리 규칙과 동일하게 적용한다.

**P0/P1 처리:** 묶음 단위로 제시하고, finding별 수용/일부수용/기각을 받는다.
- **일부수용** = finding의 문제 인식은 수용하되, 수정 범위나 방향을 사람이 조정. RDR에 "일부수용" + 조정 내용 기재.

**P2/P3 처리:** 일괄 제시 후 아래 중 선택:
- **전체 수용** / **전체 기각** / **건별 처리** (건별 선택 시 하나씩 수용/기각)

**수정 반영 (HITL, finding별 수용/기각 모드 공통):**
1. 라운드의 finding 전부 수용/기각 처리 완료
2. AI가 수용된 finding 기반으로 **설계 수정안을 제안** (수정 포인트 목록 또는 diff)
3. 사람이 확인 → 승인하면 AI가 반영 / 사람이 직접 반영 / 수정안 조정 요청
4. 사람이 반영 완료를 확인
5. **경량 통합 체크** — 오케스트레이터가 수정 적용 완료 후, 다음 라운드 에이전트에 넘기기 전에 수행:
   - **공유 상태 식별:** 이번 라운드에서 수정된 N개 항목이 공유하는 상태(데이터 구조, 상태 파일, 공통 참조)를 식별
   - **조합 시나리오 확인:** 공유 상태에 대해 "수정 A + 수정 B가 동시에 적용된 상태에서 빈칸이 없는가" 확인
   - **빈칸 발견 시:** 오케스트레이터가 자체 보정하거나 사람에게 확인. 보정 내용은 다음 라운드 finding 결정 목록에 "통합 체크 보정" 항목으로 기록
   - **비용:** 수정된 항목 목록 + 공유 상태만 전달 (전체 문서 재독 불필요). Opus/Sonnet 1회 호출 수준
   - **이상 없으면** 다음 라운드 시작

**"미해결"의 정의:**
- **미해결** = 수용/기각 판단이 안 된 건, 또는 수용했으나 설계 반영이 완료되지 않은 건
- **해결** = 수용 + 설계 반영 완료 (사람 확인), 또는 기각 (사유 기록)
- Phase 3 종결 조건: 모든 P0/P1이 해결 상태여야 함

### 2.2. AI 자동 수정 + diff 승인/반려 (코드 리뷰)

1. AI 전문가들이 **finding만 도출**한다 (수정안은 텍스트로 제안).
2. 오케스트레이터가 finding 간 코드 영향 범위를 분석하여 **diff 단위를 결정**한다:
   - 같은 파일/코드 블록을 건드리는 finding → 묶어서 하나의 diff
   - 독립된 finding → 개별 diff
3. 오케스트레이터가 묶음 단위로 **코드 수정을 실행**하여 AS-IS / TO-BE diff를 생성한다.
4. diff를 심각도 순(P0→P1→P2→P3)으로 사람에게 순차 제시한다. diff에 여러 심각도가 묶인 경우 최고 심각도 기준.
5. 사람이 diff를 확인하고 아래 중 선택:
   - **승인** — AI 수정안 채택. 다음 diff는 승인된 코드 기반으로 수정
   - **반려** — 사유 포함, AI가 사유를 반영하여 재수정 → 다시 diff 제시
   - **직접 수정** — finding 수용하되 사람이 직접 코드 수정. 수정 완료 후 AI가 최신 코드를 기반으로 다음 diff 진행
   - **기각** — finding 자체를 수용하지 않음. RDR에 기각 사유 기록. AI는 다음 diff로 진행

**코드 리뷰 → RDR "결정" 컬럼 매핑:**

| 코드 리뷰 액션 | RDR "결정" |
|---|---|
| 승인 | 수용 |
| 반려 후 최종 승인 | 수용 |
| 직접 수정 | 수용 |
| 기각 | 기각 |

**동일 diff 반려 3회 시 경고:**

> 동일 수정 사항에 대해 3회 반려되었습니다.
> 1. **직접 수정** — 사람이 직접 코드 수정
> 2. **기각** — 이 finding을 수용하지 않음
> 3. **계속** — AI에게 다시 시도

**코드 리뷰의 "라운드" 정의:**
- **1라운드** = 전문가가 전체 diff를 분석하여 finding 도출 → diff 순차 제시 → 전부 처리(승인/반려/직접수정/기각)
- **2라운드 이후** = 처리된 코드(승인+직접수정 반영된 최신 상태)에 대해 전문가가 다시 분석 → 신규 finding 도출
- 수렴 전환 조건 "직전 라운드 대비 신규 P0/P1이 0건"이 동일하게 적용됨

### 2.3. 수정 경로 리뷰

기존 설계서를 수정한 경우(papillon Step 3-B), 오케스트레이터는 전문가에게 **설계서 전문 + 수정 매니페스트**를 함께 전달한다. 수정 경로 리뷰는 복합 모드(§1.2)와 동시에 적용될 수 있다. 이 경우 §1.2의 복합 모드 전문가 선별 및 묶음 처리를 먼저 확정한 뒤, 전문가 프롬프트에 아래 추가 체크 포인트를 추가한다. 신규 경로(Step 3-A)와 동일한 코어 프로세스(finding 기반 수용/기각, 수렴 메커니즘)를 사용한다.

**추가 체크 포인트 (전문가 프롬프트에 포함):**

1. **수정-기존 정합성:** 수정된 섹션이 수정되지 않은 섹션과 논리적으로 일관되는가. 수정으로 인해 기존 동작/흐름이 변경되는 경우, 그 변경이 의도된 것이고 전체 맥락에서 타당한가.
2. **영향 범위 커버리지:** 수정 매니페스트에 명시된 영향 범위가 실제로 커버되었는가. 빠진 연쇄 수정이 없는가. (예: §1에 새 개념을 추가했는데 §4에서 해당 개념을 참조해야 하는 곳이 반영되지 않았다)
3. **용어/개념 일관성:** 수정 전후로 용어와 개념이 일관되게 사용되는가. 새로 도입된 용어가 기존 용어와 충돌하지 않는가.
4. **기존 참조 무결성:** 수정으로 인해 다른 문서에서 이 문서를 참조하는 부분이 깨지지 않는가.

**수정 매니페스트 필수 필드:** 최소한 변경 섹션 목록(섹션 번호, 변경 유형, 근거)과 영향 범위 목록이 포함되어야 한다. 선택 필드: "함께 수정된 파일" 목록 — 포함된 경우 체크 포인트 4번(기존 참조 무결성)에서 영향 범위 항목으로 함께 간주한다. 자동 생성 시 필수 포맷을 따른다.

**수정 매니페스트가 없는 경우 (단독 실행 시, papillon 경유 없이 수정 경로 리뷰를 직접 호출한 경우):** 오케스트레이터가 수정 전 백업(`.pre-review.md`)과 비교하여 변경 섹션을 식별하고, 매니페스트를 자동 생성한다. 자동 생성이 불가능하면 사람에게 변경 요약을 요청한다. papillon 경유 시에는 Step 3-B가 매니페스트를 반드시 생성하므로, 없으면 오류 처리한다. git/diff 등 외부 도구에 의존하지 않는다 (§4.1.1 원칙 일관 적용).

**수정 반영 특화 안내:** finding 수용 후 AI가 추가 수정안을 제안할 때, 매니페스트의 해당 변경 섹션과 나란히 제시하여 사람이 "매니페스트 범위 내 수정인가, 범위 외인가"를 구분할 수 있게 한다.

**영향 범위 추적:** finding 처리 완료 후, 매니페스트의 영향 범위 항목을 체크리스트로 제시하여 커버된 항목과 미확인 항목을 사람이 추적할 수 있게 한다. 리뷰 종료 시 미확인 항목은 RDR의 후속 액션 목록에 자동 포함한다.

## 3. RDR / ADR 생성

RDR/ADR 포맷 및 저장 경로: [02-common-spec.md 섹션 2~4](../02-common-spec.md) 참조.

### 3.1. RDR 생성 규칙

- 리뷰 종료 시 자동으로 생성한다. 리뷰 1회 = RDR 1개.
- 5모드 모두에서 생성한다.
- 저장 경로(`docs/decisions/reviews/`)가 존재하지 않으면 자동 생성한다.
- 수정 경로 리뷰 완료 시, 매니페스트의 영향 범위 중 미확인 항목을 RDR의 후속 액션 목록에 포함한다. 리뷰 종료 시 사람에게 "다음 확인이 필요한 문서: [목록]"을 명시적으로 안내한다.

### 3.2. ADR 승격 규칙

**설계 리뷰 및 ADR 리뷰**에서 P0/P1 수용 건 중 **되돌리기 어렵고, 여러 모듈에 영향을 주거나, 향후 작업의 전례가 되는 아키텍처 결정**에 대해 승격을 제안한다. 사람이 승인하면 생성한다.

**승격 제안 시점:** finding 전부 수용/기각 처리 완료 후, ADR 승격 대상을 **일괄 제안**한다. finding 처리 도중에 끼어들지 않는다.

**ADR 초안 작성:** AI가 수용된 finding 기반으로 ADR 초안을 자동 생성한다. 사람이 확인/수정 후 확정하고, wtth ADR 리뷰 모드로 호출한다.

**승격 대상 예시 (이에 한정되지 않음):**
- 기술 스택 또는 라이브러리 선택/변경
- 모듈 간 통신 방식 변경
- 데이터 모델의 구조적 변경 (정규화/비정규화, 테이블 분리 등)
- 인증/인가 방식 변경
- 여러 태스크에 영향을 주는 패턴 결정

**승격 대상이 아닌 예시:**
- 특정 위치에 검증 로직 추가
- 단일 API의 에러 처리 보완
- 인덱스 추가/제거
- 단일 모듈 내부 리팩토링

### 3.3. 이전 결정 스캔

리뷰 시작 시 기존 RDR/ADR을 읽어서 이미 결정된 사항을 전문가에게 컨텍스트로 전달한다.

**스캔 범위:** 리뷰 대상이 속한 프로젝트의 `docs/decisions/` 전체. 단독 실행 시 파라미터로 경로가 전달되면 그대로 사용하고, 전달되지 않으면 사람에게 물어본다. 경로가 존재하지 않으면 이전 결정 없음으로 간주하고 스캔을 생략한다.

**전달 방식:** 전문가 에이전트 프롬프트에 이전 결정 목록을 포함하여, 동일 사안을 재지적하지 않도록 지시한다.

**재검토 예외:** RDR의 결정 근거에 조건부 기각이 명시된 경우 (예: "50만건 도달 시 재검토"), 현재 상태가 해당 조건을 충족하면 재지적을 허용한다.

**유사 finding 판단:** 동일 파일/동일 위치의 동일 이슈는 제외. 다른 파일/다른 위치에서 같은 패턴이 발견되면 신규 finding으로 취급한다.

## 4. 수렴 메커니즘

### 4.1. 라운드별 역할 분리

라운드 수를 고정하지 않고, **전환 조건**으로 모드를 전환한다.

| 모드 (내부 용어) | 규칙 | 다음 모드 전환 조건 |
|---|---|---|
| **발산** | 전체 지적 허용 (초기 상태) | 직전 라운드 대비 신규 P0/P1이 0건 → 수렴으로 전환. 또는 조기 수렴 전환(§4.1.2). 또는 사람이 직접 지시 |
| **수렴** | 신규 P0/P1만 허용. 기존 건 재언급 금지 | 신규 finding이 0건 → 사인오프로 전환. 신규 P0/P1이 2건 이상 → 사람에게 발산 복귀를 제안. 또는 사람이 직접 지시 |
| **사인오프** | 각 에이전트 "추가 지적 없음" 확인. P0/P1 발견 시 수렴으로 롤백, P2/P3은 RDR에 기록하고 사인오프 계속. **경량 모델(Haiku/Sonnet) 사용 허용** — P0/P1 유무 확인만 수행하므로 Opus급 불필요 | 전원 확인 → 리뷰 종료 |

**사람에게 보이는 메시지 (내부 용어를 노출하지 않는다):**
- 발산 → 수렴 전환 시: "새로운 P0/P1 지적이 없습니다. 이제부터 신규 P0/P1만 추가로 확인합니다."
- 수렴 → 사인오프 전환 시: "추가 지적이 없습니다. 각 전문가에게 최종 확인을 요청하겠습니다."
- 사인오프 완료 시: "전문가 전원이 추가 지적 없음을 확인했습니다. 리뷰를 종료할까요?"

**전문가 간 충돌 처리:**

동일 대상에 대해 전문가 간 상반된 finding이 발생하면(예: SEC "암호화 필요" vs SRE "암호화 시 검색 불가"), 오케스트레이터가 충돌 건을 묶어서 양쪽 근거와 함께 사람에게 제시한다. 사람이 방향을 결정한다. 전문가 간 debate는 수행하지 않는다 — 트레이드오프 결정은 비즈니스 맥락을 아는 사람의 몫이다.

**발산 3라운드 초과 시 경고 (내부 용어 노출 금지):**

발산 모드에서 3라운드를 초과하면 사람에게 아래 메시지와 선택지를 제시한다:

> 리뷰가 3라운드를 초과했으나 새로운 주요 지적이 계속 발생하고 있습니다.
>
> 1. **태스크 분할** — 현재 태스크를 쪼개고 파이프라인 재진입
> 2. **설계 초안 재작성** — 누적된 finding을 반영하여 초안부터 다시
> 3. **현재 지적 사항 처리에 집중** — 신규 지적 추가를 중단하고 기존 건만 마무리
> 4. **그대로 계속**

#### 4.1.1. 라운드 간 상태 전달 원칙

2라운드 이후 에이전트에게 전달하는 입력:

1. **최신 문서 원본** (전체) — 변경 섹션만 전달하지 않는다. 전체 맥락을 보아야 새로운 finding을 놓치지 않는다.
2. **이전 라운드 finding 결정 목록** — 수용/기각/일부수용 + 사유. RDR의 finding 결정 테이블과 동일 형식.
3. **"결정 목록의 건은 재언급 금지"** 지시를 에이전트 프롬프트에 포함한다.
4. **excluded_experts 목록** — 해당 전문가에게는 발언권을 부여하지 않도록 에이전트 프롬프트에 포함한다.

git/diff 등 외부 도구에 의존하지 않는다. finding 결정 목록이 "무엇이 변경되었는가"를 전달하는 역할을 겸한다.

**세션 재개를 통한 라운드 진입 시:** §1.3.4 상태 요약은 사람에게 제시하고, 본 항의 상태 전달은 에이전트에게 전달한다. 두 흐름은 독립적으로 동시에 실행된다.

#### 4.1.2. 조기 수렴 전환

기존 자동 전환("신규 P0/P1 0건")과 별개의 조기 전환 경로:

- **조건:** P0 0건 + P1 건수가 직전 라운드 대비 50% 이상 감소
- **절차:** 오케스트레이터가 사람에게 수렴 전환을 제안 → 사람이 검토하고 AI 의견을 수렴한 뒤 최종 결정
- **사람이 거부하면** 기존 프로세스대로 발산 모드를 계속한다

사람에게 보이는 메시지 (내부 용어 노출 금지):
> "P0이 해소되었고 P1도 크게 줄었습니다. 신규 P0/P1만 확인하는 단계로 전환할까요?"

### 4.2. 1라운드 결과 분석 및 프로세스 분기

1라운드 종료 후 오케스트레이터가 **건수 + 심각도 분포 + 패턴**을 분석하여 사람에게 제안한다.

**분석 관점:**

| 관점 | 판단 내용 |
|---|---|
| **건수** | 전체 finding 수 |
| **심각도 분포** | P0/P1 비율이 높은가, P2/P3 위주인가 |
| **패턴 반복** | 동일 패턴의 반복인가 (→ 근본 원인 1건으로 병합 가능) |
| **도메인 분산** | 여러 도메인에 걸쳐있는가 (→ 태스크가 클 가능성) |
| **관심사 분산** | finding을 관심사별 group by → 관심사 3개 이상에서 P0/P1이 분산되면 분할 재검토 제안. 선택지: 관심사별 분할 / 현재 구조 유지 / 범위 축소 |

**제안 분기 (예시, 오케스트레이터가 맥락을 보고 판단):**

| 상황 | 제안 |
|---|---|
| finding 적고 P0/P1 없음 | "지적 N건이며 P0/P1이 없습니다. 리뷰를 종료할까요?" → 사람이 승인하면 사인오프 생략, 즉시 RDR 생성 후 리뷰 종료 |
| finding 많고 P0/P1 섞여있고 도메인 다양 | "지적이 N건이고 도메인이 분산되어 있습니다. 태스크 분할을 고려하세요." |
| finding 많지만 P2/P3 위주, 동일 패턴 반복 | "지적이 N건이지만 동일 패턴의 반복입니다. 근본 원인을 먼저 해결하면 대부분 해소될 것 같습니다." |
| finding 많고 P2/P3 위주, 전반적 품질 미달 | "지적이 N건이며 전반적인 품질 이슈입니다. 초안을 재작성하는 것이 효율적일 수 있습니다." |

오케스트레이터는 건수를 기계적으로 적용하지 않고, **why/how를 포함하여 제안**한다. 최종 판단은 사람이 한다.

**예상 토큰 규모감 제시:** 1라운드 분석 시, 전문가 수 × 문서 규모 × 예상 라운드 수를 기반으로 전체 리뷰의 예상 토큰 규모감을 정보 제공 톤으로 제시한다. "참고로 이 규모입니다" 수준 — 사람이 비용 대비 효과를 판단하는 데 활용한다.
