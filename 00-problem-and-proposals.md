# review-hell 개선: 문제 진단 및 제안 정리

**작성일:** 2026-04-03
**출처:** 2026-04-02 Sonnet 대화 + Opus 분석

---

## 1. 핵심 문제 3가지

### 1-A. 결정 망각

- review-hell로 수용/일부수용/기각 의사결정을 하지만, 세션이 끝나면 LLM도 사용자도 기억 못 함
- 현재 스킬에 "이미 결정된 사항은 지적하지 않는다" 규칙이 있으나, 과거 결정을 참조할 수단이 없어 **사문화된 상태**

### 1-B. 태스크 과대 → 리뷰 지옥

- 태스크가 크면 전문가 5명 투입 → 유니크 지적 30개/라운드 × 8라운드 = ~240개 처리
- Opus 오케스트레이터가 중복을 걸러도 원본 태스크가 크면 지적 총량은 줄일 수 없음
- **근본 원인은 프로세스가 아니라 태스크 크기**

### 1-C. 결과물 품질 저하

- 태스크가 크면 LLM 컨텍스트 내에서 앞쪽 결정을 뒤쪽에서 잊음
- 설계 문서 자체의 내부 일관성이 깨짐 → 코드 생성 시에도 동일 현상
- 배포하면 이상 동작, 기능 누락 발생

## 2. 제안: 결정 기록 (2계층 구조)

### 계층 1 — Review Decision Record (RDR) → review-hell 내장

리뷰 1회 = RDR 1개. review-hell 단계 6 이후 **자동 생성**.

```markdown
# RDR-2026-04-03-payment-api-설계서

**리뷰 대상:** docs/payment-api-design.md
**날짜:** 2026-04-03
**참여 전문가:** BE, SEC, SRE

| ID | 심각도 | 제목 | 결정 | 사유 요약 |
|---|---|---|---|---|
| SEC-01 | P0 | reviewer_id 인증 미연결 | 수용 | 권한 우회 가능 |
| BE-01 | P1 | 낙관적 락 누락 | 수용 | 동시 수정 시 데이터 유실 |
| BE-02 | P2 | 인덱스 추가 | 기각 | 현재 데이터량에서 불필요 |

### 주요 기각 사유
- **BE-02**: 현 단계 데이터 10만건 미만, 추후 재검토
```

### 계층 2 — 미니 ADR → review-hell이 "승격 제안"

- RDR 항목 중 **P0/P1 수용 + 설계 변경 수반** 건에 대해 ADR 승격 제안
- 모든 finding에 하는 게 아니라 아키텍처 수준 결정만 대상

```markdown
# ADR-001: JWT 인증 방식 채택

**날짜:** 2026-04-02 | **상태:** 확정

**결정:** 세션 대신 JWT 사용
**이유:** stateless 서버, 모바일 확장 고려
**기각:** 세션(메모리 의존), OAuth 단독(과함)
**영향:** auth 미들웨어, 전 엔드포인트
```

### review-hell 스킬 변경점

**단계 2 (컨텍스트 수집) 추가:**
- `docs/decisions/reviews/*.md` — 이전 RDR 스캔
- `docs/decisions/ADR-*.md` — 기존 ADR 스캔
- 이미 결정된 항목은 리뷰 범위에서 자동 제외

**단계 6 (최종 결과) 확장:**
1. 전체 수용/일부수용/기각 집계 테이블 출력
2. **RDR 자동 생성** — `docs/decisions/reviews/`에 저장
3. **ADR 승격 제안** — P0/P1 수용 중 설계 변경 수반 건
4. 사용자 다음 지시 대기

## 3. 제안: 태스크 크기 기준 재정의

### 현재 문제

기능 단위로 자르고 있으나 여전히 큼 → 리뷰 8라운드, 결과물 품질 저하

### 새로운 크기 기준 (3가지 중 하나라도 넘으면 쪼갬)

1. 설계 문서가 **A4 2장 이내**로 서술 가능한가
2. 관련 API가 **3개 이하**인가
3. 리뷰 지적이 **한 라운드 10개 이하** 예상되는가

### 목표

- 태스크당 총 지적 30개 이하
- 3라운드 이내 수렴

### 태스크 시작 전 사전 점검

오케스트레이터에게 먼저 질문:
> "이 태스크를 5명이 리뷰하면 지적이 몇 개 나올 것 같아? 30개 넘을 것 같으면 어떻게 쪼갤 수 있어?"

## 4. 제안: 수렴 메커니즘 (Sonnet 제안)

### 라운드별 역할 분리

| 라운드 | 모드 | 규칙 |
|---|---|---|
| 1~2 | 발산 | 전체 지적 허용 |
| 3~ | 수렴 | 신규 P0/P1만, 기존 건 재언급 금지 |
| 마지막 | 사인오프 | 각 에이전트 "사인오프 가능?" Yes/No만 |

### 전문가 선택적 투입 (현재 스킬에 이미 사용자 승인 단계 있음)

| 태스크 성격 | 투입 리뷰어 |
|---|---|
| 인증/권한/Secret | SEC + BE |
| 순수 API 설계 | BE + SRE |
| 프론트 컴포넌트 | FE + PO |
| DB 스키마 변경 | BE + SRE + SEC |
| 신규 기능 전체 | PO + BE + FE |
| 인프라/배포 | SRE + SEC |
| **전원 투입** | 아키텍처 변경, 보안 정책 변경 등 **진짜 큰 것만** |

## 5. 제안: 계층적 디렉토리 구조 (작업 단위 관리)

```
dashboard/
  00-index.md              ← 전체 조감도 (위젯 목록, 의존관계, 상태)
  01-monthly-stat/
    00-index.md            ← 위젯 범위, API, 상태, 관련 ADR
    design.md
    TASK-001.md
    TASK-002.md
  02-user-activity/
    00-index.md
    ...
  03-revenue-chart/
    00-index.md
    ...
```

- 새 세션 시작 시 `00-index.md` 하나만 읽으면 전체 맥락 복원
- 각 위젯(모듈) 단위가 독립 태스크 → 리뷰 범위도 자연스럽게 제한

## 6. 제안: 작업 요약 카드 (Sonnet 제안)

ADR이 "왜"를 기록한다면, 작업 요약 카드는 "무엇"을 기록.

```markdown
# TASK-A-01: 사용자 인증 API

**완료일:** 2026-04-02
**관련 ADR:** ADR-001, ADR-003
**구현 범위:** POST /auth/login, /auth/refresh
**의존:** 없음
**피의존:** 대시보드 모듈, 마이페이지 모듈
**특이사항:** refresh token rotation 적용
```

## 7. 이상적인 전체 사이클

```
태스크 분할 (단일 관심사, 크기 기준 충족)
  ↓
설계 초안
  ↓
review-hell (선택적 전문가 투입, 수렴 메커니즘 적용)
  ↓
RDR 자동 생성 + ADR 승격 (해당 시)
  ↓
설계 확정
  ↓
개발
  ↓
코드 리뷰 → RDR/ADR 기록
  ↓
(반복)
  ↓
최소 배포 단위 도달 → 배포
  ↓
다시 처음으로
```
