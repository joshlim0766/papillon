## tracing 2 — SHOULD 신호별 "신호 유무가 shackled 결과에 영향 준 정도"

SHOULD 3건: #3 에러 타입 응집 / #4 의존 DAG (단일) / #6 테스트 파일 경로.

### #3 에러 타입 응집

**exemplar 상태**: `pipeline/exceptions/` 패키지 명시 + `GroupLookupRaceFailureException` 위치 지정 + `DltErrorReason` 매핑 언급.

**실측**: 
- 구현 중 추가된 `TenantContextMismatchException` 도 동일 `pipeline/exceptions/` 패키지에 배치 → **응집 원칙 유지**.
- wtth ARCH P1-4 수용: 예외 타입 세분화로 2F `DefaultErrorHandler` 의 retryable/non-retryable 분류가 정확해짐.

**영향 정도**: **큼**. 이 신호가 있어서 shackled가 신규 예외 타입 배치 결정에서 혼동 없음. 없었으면 `lookup/` 이나 `upsert/` 내부에 분산 배치될 위험.

**판정**: **MUST 승격 후보** (실측 근거 기반).

### #4 의존 DAG (단일 태스크)

**exemplar 상태**: 2e 헤더에 "의존: 2D / 선행: use-block".

**실측**: 선행 조건 충족 상태에서 진입 → 문제 없음.

**영향 정도**: 작음 (선행이 이미 문제없이 완료된 상태였기 때문).

**판정**: **SHOULD 유지**. 단일 태스크에선 조건이 이미 충족된 경우가 많아 영향 작음.

### #6 테스트 파일 경로 사전 지정

**exemplar 상태**: §6 에 3 파일 분할 경로 지정 (Batch + Race + 필요 시 Empty).

**실측**: shackled 가 지시서 대로 2 파일 생성 (Batch + Race). 구현 중 추가로 `UpsertServiceTransactionIntegrationTest` (SpykBean 기반 Connection 재사용 증명) 신규 생성 — 이는 지시서 예상 밖.

**영향 정도**: 중간. 지시서가 제시한 2 파일은 곧바로 사용, 추가 1 파일은 구현 중 필요성 발견.

**판정**: **SHOULD 유지**. "테스트 파일 경로는 예측 가능 범위만 지정, 추가 발생은 자연스러움" 부연 권고.

## tracing 3 — AP-1/AP-2별 "exemplar RED에서 실제 위반 여부 + 결과"

### AP-1 (비결정적 RED 금지)

**exemplar 위반 여부**: 
- **RED 7** (race 2세션 병렬, 사전 적재된 동일 차원 ROW LOCK 직렬화) — 환경 의존 부분 존재. "두 세션 모두 정상 커밋, Deadlock 없음" 검증은 MySQL InnoDB 타이밍에 의존.
- **RED 9** (1000 엔트리 5초 timeout) — 지시서가 스스로 "절대 임계값 아님, CI 변동성 크면 @Tag 분리" 주석 달아 **사전 완화**. 실측에서 wtth TEST P2-2 수용으로 `@Tag("performance")` + 별도 Gradle task 분리.

**위반 시 결과**: 
- RED 7: wtth P0-3 수용으로 **범위 축소** — "ROW LOCK 직렬화 sanity" 로 정직화. Tier 1 insertMissing multi-session deadlock 은 별도 테스트로 빼지 않고 RDR F-15 (운영 관측) 로 이관.
- RED 9: `@Tag("performance")` 분리로 기본 test task 에서 제외. CI 변동성 방어.

**환류**: AP-1 유효. 2e RED 7 전환사(단건 시대 "실제 MySQL 데드락 유발" → 배치 시대 "race 상류 해소 회귀 증명" → wtth 후 "ROW LOCK 직렬화 sanity")가 **AP-1 해소 방향의 3단계 학습 사례**. v0.2.0 anti-pattern 섹션에 보존 가치.

### AP-2 (구현 상세 검증 RED 금지)

**exemplar 위반 여부**: 
- **RED 6** (트랜잭션 실패 시 rollback): 지시서가 "Tier 2 UPSERT 직전 예외 hook 주입" 제시. 구현은 **behavior 기반** (예외 전파 + rollback 검증) 으로 유지. 위반 없음.
- **Transaction Integration Test 초기 설계**: `System.identityHashCode(ctx.connection())` 로 Connection wrapper 동일성 검증 시도 → **AP-2 경향 발견** (구현 내부 상세 종속). `TransactionAwareDataSourceProxy` 가 호출마다 새 JDK proxy wrapper 반환하여 **FAIL**. behavior 검증(물리 `HikariDataSource.getConnection()` 카운트) 으로 전환하여 GREEN.
- **RED 3 SQL count 검증**: wtth TEST P1-1 수용으로 `ExecuteListener spy` 로 SQL 실행 횟수 검증. 이는 behavior (관측 가능한 외부 효과) 검증. AP-2 미해당.

**위반 시 결과**: 
- 초기 Transaction Test 설계에서 AP-2 위반 → FAIL → 교정 → GREEN. 즉 **컴파일러/테스트가 AP-2 위반을 자동 감지한 드문 사례**.

**환류**: AP-2 유효. Transaction Test 교정사(identityHashCode mock verify → physical getConnection counter)가 **"구현 상세 → observable effect" 전환의 학습 사례**. v0.2.0 anti-pattern 섹션에 보존.

## tracing 4 — 신호 #7 7A/7B 재분리 판단

**7A (TDD 플래그 헤더)**: 2e 경험상 단독 기능 — shackled `tdd_mode` 판정 근거. 명확한 단일 용도.

**7B (공통 제약 G1~G12)**: 2e 경험상 복합 기능 —
- **G1/G6/G8/G9**: 준수. 품질 문제 없음.
- **G3** (TenantContext.use("megabird") 고정): 준수.
- **G11** (재시도 금지): 준수 (2e 범위 내).
- **G12** (JooqConfig 수정 금지): **부분 예외**. Spring+jOOQ 연동 구조적 요구로 `TransactionAwareDataSourceProxy` 명시 배선이 불가피했음. 단서조항 추가로 해결.

**재분리 판단**:
- 7A 단독 신호는 "TDD 진입 여부 판정"이라는 명확한 축 하나만 담음.
- 7B 는 "여러 공통 제약의 집합" → 성격상 **체크리스트**. 일부 위반이 발생할 수 있으며, 이를 anti-pattern 과 구분해야 함.
- **결론**: 7A/7B 통합 유지 OK (둘 다 MUST). 단 **7B 에 "공통 제약의 예외는 단서조항으로 지시서에 명시" 메커니즘 요건 추가**. G12 사례가 직접 근거.

**재분리 불필요 근거**: 2e 1건 기준으로 7A/7B 가 서로 다른 품질 효과를 낸다는 증거는 약함. 둘 다 존재할 때 shackled 의 행동이 일관되게 TDD 규율을 준수했다. 통합 유지가 경제적.

## v0.2.0 draft 반영 권고사항 (2e 기준 가설)

**신규 요건 (2e 경험 기반)**:

1. **#8 상태·부작용 경계 — 권위 소스 교차 참조 요건 추가**: DDL/스키마/테이블 상수 등은 단일 권위 소스(`02-Design/02-table-schema.md` 류) 지정 + 지시서는 해당 소스를 명시 참조. 2e §2.4 UNIQUE KEY 오탈자 경험이 근거.

2. **#2 I/O 계약 — 예외 타입 카탈로그 확장**: 주 메서드 시그니처뿐 아니라 **방어 예외 타입 사전 열거** 요건화. 2e `TenantContextMismatchException` 미열거 경험이 근거. (신호 #3 과 일부 중복이나 "사전 카탈로그" 요건은 별도.)

3. **#7B 공통 제약 — 단서조항 메커니즘**: 공통 제약에 **예외가 발생할 수 있음을 전제**하고, 예외 사례는 지시서 본문에 "단서조항" 블록으로 박제하는 체계 도입. 2e §4.4 경험이 근거.

**재분류 후보 (단일 샘플이라 보류)**:

- **#3 에러 타입 응집 SHOULD → MUST 승격**: 2e 에서 신호 유무 차이가 큼(응집 규칙이 신규 예외 배치 결정을 자동화). 2f 데이터로 재확인.

**anti-pattern 학습 케이스 보존**:

- **AP-1 예**: 2e RED 7 3단계 전환사 (단건 데드락 유발 → 배치 race 해소 증명 → ROW LOCK 직렬화 sanity).
- **AP-2 예**: Transaction Integration Test 초기 설계(`identityHashCode` mock verify → 물리 `getConnection` counter)의 behavior 전환.
