## 요구사항 요약

### R-1. 라운드 간 상태 전달 원칙 (출처: AI-3 + AI-6 병합)

2라운드 이후 에이전트에게 전달하는 입력:
1. **최신 문서 원본** (전체)
2. **이전 라운드 finding 결정 목록** (수용/기각/일부수용 + 사유)
3. "결정 목록의 건은 재언급 금지" 지시

AI-3(전체 문서 + 결정 목록)과 AI-6(변경 섹션만 전달)이 충돌했으나, **A 방식(전체 문서)으로 결정.** 근거: shackled 리뷰에서 토큰이 녹은 근본 원인은 2라운드 전체 읽기가 아니라 M 체급에 7개 관심사를 밀어넣은 체급 판정 오류. 체급이 정확하면 문서 규모 자체가 적정하여 전체 읽기 비용도 수용 범위.

외부 도구(git/diff) 의존 제로.

### R-2. 조기 수렴 전환 경로 (출처: AI-6)

기존 자동 전환("신규 P0/P1 0건 → 수렴")과 **별개의 조기 전환 경로**를 추가:
- 조건: P0 0건 + P1 건수 직전 라운드 대비 50% 이상 감소
- 오케스트레이터가 사람에게 제안 → 사람이 검토·AI 의견 수렴 후 최종 결정 필수
- 사람이 거부하면 기존 프로세스대로 계속

### R-3. 사인오프 경량 모델 (출처: AI-6)

사인오프 모드에서 경량 모델(Haiku/Sonnet) 사용을 허용한다. P0/P1 유무 확인만 하면 되므로 Opus급이 불필요.

### R-4. 1라운드 분석 시 토큰 규모감 제시 (출처: AI-6)

§4.2 1라운드 결과 분석에서 예상 토큰 규모감을 정보 제공 톤으로 제시한다. "참고로 이 규모입니다" 수준.

### R-5. 관심사 분산도 분석 (출처: AI-7, 개입 지점 2 — wtth 측)

§4.2 분석 관점 테이블에 "관심사 분산" 행을 추가:
- finding을 관심사별 group by → 분산도 분석
- 관심사 3개 이상에서 P0/P1 분산 → 분할 재검토 제안
- 선택지: 관심사별 분할 / 현재 구조 유지 / 범위 축소

### R-6. 초안 완료 시 관심사 분포 규모감 제시 (출처: AI-7, 개입 지점 1 — papillon 측)

papillon Step 3(Draft 생성) 완료 시 관심사 분포를 요약하고 예상 규모감을 제시한다:
- 정보 제공 톤 — "분할을 권합니다"가 아니라 "참고로 이 규모입니다"
- 관심사 수, 예상 finding 범위
- 이중 안전망: 여기서 사람이 무시해도 R-5(1라운드 분석)에서 finding 데이터 기반으로 재확인
