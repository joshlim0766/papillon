# Issue-03-01: 라운드 간 수정 적용 시 통합 체크 부재

## 심각도: 높음

## 문제

wtth 리뷰에서 1라운드 finding을 수용하고 수정을 적용한 뒤, 2라운드로 넘기는 과정에 **수정 간 통합 체크**가 없다.

1라운드 finding은 개별 빈칸을 지적하고, 수정 적용자(오케스트레이터)는 각 finding을 독립적으로 반영한다. 이때 "수정 A와 수정 B가 결합했을 때 새로운 빈칸이 생기는가?"를 확인하는 단계가 설계에 존재하지 않는다.

### 실례

묶음 B 리뷰에서 발생:
- 1라운드: "§1.2 복합 모드 추가", "§1.3 excluded_experts 영속 추가", "mode 필드 유효값 정의" 등 개별 수용
- 수정 적용: 각 finding을 기계적으로 반영
- 2라운드: "복합 모드일 때 mode 필드가 단일값이라 복원 불가", "experts에 모드 매핑 없어 묶음 헤더 복원 불가" 등 **조합 엣지 케이스** 5건 발생

각 수정은 개별적으로 정확했지만, 공유 상태(`.wtth-review-state.json`)에서 결합했을 때 빈칸이 생겼다.

## 왜 중요한가

1. **사람의 역할 변화:** 도구가 고도화되면 사람은 세부보다 큰 흐름에 집중하게 된다. 조합 엣지 케이스를 사람이 매번 추적하는 건 도구를 쓰는 의미를 훼손한다.
2. **2라운드가 잡긴 잡았지만:** 다중 라운드 메커니즘이 안전망 역할을 했으나, 이건 결과론이다. 수정 적용 시점에서 선제적으로 잡을 수 있었다면 2라운드 finding이 줄어들고 전체 리뷰 비용(토큰, 사람 시간)이 감소했을 것.
3. **빈도는 낮지만 위험:** 조합 엣지 케이스는 크래시가 아니라 조용히 잘못된 결과를 만드는 유형이라 발견이 어렵다.

## 솔루션

### 라운드 간 경량 통합 체크

수정 적용 완료 후, 2라운드 에이전트에 넘기기 전에 오케스트레이터가 수행하는 경량 체크:

1. **공유 상태 식별:** 이번 라운드에서 수정된 N개 항목이 공유하는 상태(데이터 구조, 상태 파일, 공통 참조)를 식별
2. **조합 시나리오 확인:** 공유 상태에 대해 "수정 A + 수정 B가 동시에 적용된 상태에서 빈칸이 없는가" 확인
3. **빈칸 발견 시:** 2라운드로 넘기기 전에 오케스트레이터가 자체 보정하거나, 사람에게 확인

### 적용 위치

- `04-wtth/02-design/00-core.md` §4.1.1 라운드 간 상태 전달 원칙 — 수정 적용 후 통합 체크 단계 추가
- 또는 §2.1 수정 반영 단계 4 직후에 삽입

### 비용

- Opus/Sonnet 1회 호출 수준 (수정된 항목 목록 + 공유 상태만 전달)
- 전체 문서를 다시 읽지 않아도 됨 — 수정 항목 간 교차점만 확인
- 2라운드 finding 감소로 전체 리뷰 비용 절감 효과 기대

## 적용 시점

묶음 B 리뷰 완료 후 — 현재 리뷰에서 발견된 인사이트를 반영
