# 2: 컨텍스트 윈도우 제약이 설계에 구체적으로 반영되지 않음

**심각도:** 중간
**분류:** 제약
**상태:** Open

## 문제

파이프라인의 가장 큰 물리적 제약은 Claude의 컨텍스트 윈도우인데, 설계 어디에서도 이걸 정면으로 다루지 않는다.

papillon 오케스트레이터가 하나의 세션에서 inquisition → 체급 판정 → Draft 생성 → wtth 리뷰 → 종결 판단 → shackled 구현 → 코드 리뷰 → 커밋까지 돌리려면, 각 Phase의 프롬프트 + 누적 대화가 윈도우를 채울 수 있다.

- M 체급도 전체 파이프라인을 한 세션에서 소화하기 어려울 수 있음
- wtth 발산 2~3라운드 시 finding + 사람 응답이 상당한 토큰 소모
- L/XL은 세션 분할이 있지만, Phase 중간에 윈도우가 부족해지면 어떻게 하는지 미정의

## 제안된 솔루션: Phase별 토큰 버짓 + 컨텍스트 리셋 포인트

**Phase별 토큰 버짓 산정:**

정확할 필요 없다. 대략의 규모감이면 충분. papillon 프롬프트 N천 토큰, inquisition 인터뷰 5턴 기준 M천 토큰, wtth 1라운드 finding K천 토큰... 이런 식으로 쌓아보면 M 체급이 한 세션에 들어가는지 바로 보인다.

**안 들어가면 두 가지 선택지:**

1. M도 Phase 단위로 세션을 끊는다 (Phase 1 세션, Phase 2~3 세션, Phase 4~6 세션)
2. 각 Phase 시작 시 이전 Phase의 대화를 요약으로 압축한다

후자가 더 현실적. 이미 `HANDOFF.md` 패턴을 쓰고 있으니 Phase 간 핸드오프를 자동화하는 방향이 자연스럽다.

**구체적으로:** papillon 설계서에 "Phase 전환 시 오케스트레이터가 현재까지의 컨텍스트를 산출물 파일에 저장하고, 새 Phase는 산출물 파일만 읽어서 시작한다"는 원칙을 명시. L/XL에서는 이미 이걸 하고 있고(세션 분할), M에도 같은 원칙을 적용하되 세션은 안 끊는 — "컨텍스트 리셋 포인트"를 두는 것.

**적용 시점:** 파이프라인 1~2회 더 돌린 후. 실측 데이터가 있어야 버짓 산정이 의미 있음.
