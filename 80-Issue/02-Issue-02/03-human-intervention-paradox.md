# 3: PRD "사람은 의사결정만" vs 설계의 사람 확인 지점 과다

**심각도:** 높음
**분류:** 모순
**상태:** Open

## 문제

PRD G3이 "사람이 하는 행위는 리뷰 의사결정과 종결 승인뿐"이라고 했는데, 실제 설계는 이 목표를 달성하지 못했다.

★ 마크로 표시된 사람 개입 지점이 M 체급 기준으로도 최소 10곳 이상:
- 인터뷰 제안 승인
- 문맥 파일 생성 승인
- 종료 승인
- 미결 처리
- 체급 승인
- 초안 확인
- finding별 수용/기각 (매 건)
- ADR 승격 승인
- 종결 승인
- 구현 태스크별 확인
- 코드 리뷰 diff별 승인/반려
- 커밋 승인

## 왜 문제인가

"자동화된 파이프라인" 위에 "매 단계 사람 확인"이 덧붙여져서, 수동 파이프라인보다 사람의 개입 빈도가 높아질 수 있다. 각각의 확인 지점에는 이유가 있지만, 전체적으로 보면 "파이프라인은 자동, 사람은 의사결정만"이라는 핵심 목표와 모순된다.

PRD의 약속과 설계의 현실이 다르다는 걸 인식하고, 둘 중 하나를 조정해야 한다:
- PRD를 현실에 맞게 수정하거나
- 설계에서 사람 개입 지점을 줄이거나

## 제안된 솔루션: 확인 지점 3등급 분류

모든 ★를 동등하게 취급하지 말고 3등급으로 분류:

| 등급 | 설명 | 예시 | 동작 |
|---|---|---|---|
| **Gate (관문)** | 잘못 판단하면 이후 전체가 틀어짐 | 체급 판정, 설계 리뷰 종결, P0 finding 수용/기각 | 반드시 사람이 명시적 승인 |
| **Checkpoint (확인)** | 사람이 "OK" 또는 "잠깐" | 문맥 파일 생성, 인터뷰 종료 제안, 태스크 목록 확인 | "이의 없으면 진행합니다" 패턴 가능 |
| **FYI (알림)** | 봐야 하지만 승인 불필요 | 진행률 표시, P2/P3 일괄 처리 결과, 크로스 체크 이력 | 보여주되 멈추지 않음 |

이렇게 분류하면 파이프라인이 사람을 기다리며 멈추는 횟수가 Gate 수준으로 줄어든다. PRD G3의 약속에 훨씬 가까워짐.

**적용 시점:** papillon 설계서에 한 섹션 추가하면 끝. 지금 바로 가능.

## 심화: 실패 비용 기반 위임 판단

"어디까지 맡길 수 있는가"가 어려운 이유: 같은 작업에서도 상황에 따라 위임 가능 여부가 달라진다. 예) wtth finding 수용/기각은 P0이면 반드시 사람이 봐야 하지만, P3이면 AI 자동 수용해도 거의 문제 없음.

**기준: "위임했을 때 실패 비용이 얼마인가"**

| 실패 비용 | 특징 | 예시 | 판단 |
|---|---|---|---|
| **낮음** | 되돌리기 쉬움. "아 그거 아니야" 한 마디면 복구 | 문맥 파일 생성 승인, 인터뷰 종료 제안, P2/P3 일괄 수용, 크로스 체크 결과 확인 | **지금 당장 자동화** (Checkpoint/FYI 등급) |
| **높음** | 되돌리기 어려움. 후속 작업 전부에 영향 | 체급 판정(M→L이면 설계 재작업), P0 finding 기각, 설계 종결 승인 | **사람이 계속 유지** (Gate 등급) |
| **중간** | AI가 잘할 수도 못할 수도 있고, 판단 근거 부족 | 태스크 목록 확인, 태스크별 구현 결과 확인, ADR 승격 여부 | **사례 수집 타겟** (아래 참조) |

**중간 지대 — 오버라이드 빈도 추적:**

사례 수집의 진짜 타겟은 중간 지대. 파이프라인 돌릴 때마다 "이 지점에서 사람이 AI 제안을 그대로 수용했는가, 오버라이드했는가"만 기록하면 된다.

- 수용률 90%+ → 자동화 후보
- 수용률 50% 이하 → 사람 유지

별도 추적 불필요. RDR에 기각/수용 기록이 이미 남으니, 파이프라인 3~5회 돌린 후 "AI 제안을 오버라이드한 비율이 높은 지점이 어디인가"만 보면 자연스럽게 보인다.

**요약:** 모든 ★ 지점에 대해 수집할 필요 없음. 실패 비용 낮으면 자동화, 높으면 사람 유지 — 이건 이미 답이 나와 있음. 중간 지대에서의 오버라이드 빈도만 보면 된다.

## 참고: 현재 단계에서의 ROI

파이프라인 완성도를 높이는 것보다 실제 작업에 빨리 적용해보는 것이 ROI가 높다. 작은 S 체급 하나를 papillon으로 돌려보면, 설계서에서는 안 보이던 문제가 바로 드러날 것.
