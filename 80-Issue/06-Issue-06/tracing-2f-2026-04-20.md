## 관찰 1 — TDD 사이클 한 세션 수행 시 Sonnet 품질 저하

### 현상 (주인님 실측)

- 2f 를 Sonnet 으로 진행 중, 단일 세션 내에서 RED → GREEN → REFACTOR 사이클을 연속 수행.
- 진행이 누적될수록 Sonnet 판단 품질 저하 관찰. 주인님 표현 "빠르게 병신새끼가 되어감. 조금 커지면 착란 일으킬지도 모르겠어."
- Sonnet 자체가 도중 대화 자동 압축 시도 (auto-compaction 발동). 그러나 근본 해결 아님.

### 독립 검증

주인님이 Claude Web Sonnet 에 동일 맥락 공유 → 독립적으로 동일 분석 도달:

> TDD 사이클 자체가 의도적으로 상태를 전환하는 구조. Red → Green → Refactor 가 단순 코딩 순서가 아니라 "지금 뭘 보고 있는가" 의 **컨텍스트 전환**. 한 세션에 다 때려넣으면 모델 입장에서 실패 코드·성공 코드·중간 상태 신호가 뒤섞임.

### 원리 분석

TDD 3단계는 각각 다른 멘탈 모드:

| 단계 | 멘탈 모드 | 핵심 신호 |
|---|---|---|
| RED | "이 동작은 아직 구현 안 됨" 을 **의도적으로 기록** | 실패가 기대값 |
| GREEN | "최소 변경으로 통과만" | 성공이 기대값, 구조 최적화 금지 |
| REFACTOR | "테스트 유지하며 구조 개선" | 이미 녹색, 외부 관찰 불변 |

한 세션 안에 3 단계를 다 두면:
- RED 작성 시의 "실패 기대" 가 GREEN 작업 중에도 맥락에 남아 **과잉 방어 구현** 유도.
- GREEN 의 "최소 변경" 원칙이 REFACTOR 와 섞여 **구조 최적화 타이밍 혼선**.
- 이전 실패 케이스 주변 노이즈가 압축 시 비교 대상으로 끌려들어와 **잘못된 패턴 학습**.

### Issue-06 관점에서의 의미

현 spec v0.1.0-draft 의 **신호 #5 TDD 플래그 헤더** 와 **신호 #7 TDD 구현 규율**은 **지시서 품질** 에 초점. 하지만 이 관찰은 **shackled 소비 로직** (오케스트레이터의 세션 관리) 자체의 요건임을 시사.

즉 Issue-06 의 스코프가 확장될 필요:
- **기존**: TDD-ready 설계서 규격 (지시서 측).
- **추가**: TDD-ready 소비 로직 (shackled 오케스트레이터 측) — 세션 분할 메커니즘.

이 확장을 Issue-06 #4 "shackled tdd_mode 소비 로직 확인" 으로 처리하던 것에 **세션 분할 메커니즘** 을 신규 요건으로 추가.

### 임시 완화책 (2f 진행 중)

- auto-compaction 근접 신호 감지 시 수동 `/clear` + 핸드오프 노트 작성.
- 핸드오프 노트 형식 (Claude Web Sonnet 제안):
  ```
  ## 현재 상태: GREEN 진입
  - 완료된 테스트: [테스트명] ([개수] RED 확인)
  - 다음 목표: 테스트 통과만, 설계 최적화 금지
  - 건드리지 말 것: [다음 세션 범위]
  ```

세션 분할 메커니즘 정식 설계는 [`session-split-design-v0.1.0-draft.md`](session-split-design-v0.1.0-draft.md) 에 초안 박제. 2f 완료 후 실측 데이터로 설계 검증 → v0.2.0 통합.
