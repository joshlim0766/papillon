## 1. 심각도 기준 (P0~P3)

| 등급 | 기준 | 사용 맥락 |
|---|---|---|
| **P0** | 데이터 손실·보안 침해·시스템 다운 가능 | wtth: 강력 수용 권장 / papillon: 종결 불가 |
| **P1** | 운영 장애·감사 실패·주요 UX 붕괴 가능 | wtth: 수용 권장 / papillon: 종결 불가 |
| **P2** | 기술 부채·성능 저하·혼란 유발 | wtth: 수용 제안, 기각 존중 / papillon: 종결 가능 |
| **P3** | 개선 제안·코드 품질·선택적 강화 | wtth: 중립 제시 / papillon: 종결 가능 |

## 2. RDR (Review Decision Record) 포맷

리뷰 1회 = RDR 1개. wtth이 리뷰 종료 시 자동 생성한다.

**파일명**: `RDR-YYYY-MM-DD-{대상명}.md`
**저장 경로**: `{프로젝트}/docs/decisions/reviews/`

```markdown
# RDR-YYYY-MM-DD-{대상명}

**리뷰 대상:** {파일 경로 또는 설명}
**날짜:** YYYY-MM-DD
**리뷰 모드:** {PRD / 설계 / ADR / 운영 절차 / 코드}
**참여 전문가:** {도메인 목록}

| ID | 심각도 | 제목 | 결정 | 사유 요약 |
|---|---|---|---|---|
| SEC-01 | P0 | ... | 수용 | ... |
| BE-01 | P1 | ... | 기각 | ... |

### 주요 기각 사유
- **BE-01**: {사유}

### 주요 결정의 대안 비교 (선택적)
대안 비교가 있었던 수용 건에 한해 기록한다. 모든 수용 건에 작성할 필요 없음.
- **SEC-01**: {수용된 방안} vs {기각된 대안} — {선택 근거}
```

---

## 3. ADR (Architecture Decision Record) 포맷

P0/P1 수용 건 중 **설계 변경을 수반하는 항목**에 대해 wtth이 승격을 제안한다.
사람이 승인하면 생성한다.

**파일명**: `ADR-NNN-{제목}.md`
**저장 경로**: `{프로젝트}/docs/decisions/`

```markdown
# ADR-NNN: {제목}

**날짜:** YYYY-MM-DD | **상태:** 확정

**결정:** {한 줄 요약}
**이유:** {근거}
**기각:** {기각된 대안과 사유}
**영향:** {영향 범위}
```

---

## 4. 작업 요약 카드 포맷

태스크 완료 시 생성한다. ADR이 "왜"를 기록한다면, 작업 요약 카드는 **"무엇을 만들었는가"**를 기록한다.
새 세션에서 LLM이 모듈의 구현 범위와 의존 관계를 빠르게 파악하는 데 사용한다.

**파일명**: `TASK-{ID}-{제목}.md` (해당 태스크 파일에 추가 또는 별도 생성)
**저장 경로**: `{프로젝트}/03-tasks/` 또는 태스크 문서 내 하단

```markdown
# TASK-A-01: 사용자 인증 API

**완료일:** YYYY-MM-DD
**관련 ADR:** ADR-001, ADR-003
**관련 RDR:** RDR-YYYY-MM-DD-auth-api
**구현 범위:** POST /auth/login, /auth/refresh
**의존:** 없음
**피의존:** 대시보드 모듈, 마이페이지 모듈
**특이사항:** refresh token rotation 적용
```

## 5. Phase 요약 카드 포맷 (XL 체급)

XL 체급에서 하나의 Phase가 완료될 때 생성한다. 해당 Phase 내 작업 요약 카드들의 summary이다.
새 세션에서 index → Phase 요약 → 태스크 카드 순으로 drill-down하여 맥락을 복원한다.

**저장 위치**: 해당 Phase 디렉토리의 `00-phase-summary.md`

```markdown
# Phase N 요약: {Phase 제목}

**완료일:** YYYY-MM-DD
**포함 태스크:** TASK-001, TASK-002, TASK-003

## 달성 결과
- {이 Phase에서 무엇이 만들어졌는가 / 무엇이 완료되었는가}

## 주요 결정
- ADR-002: {한 줄 요약}
- RDR 3건: 설계 리뷰 2회, 코드 리뷰 1회

## 다음 Phase를 위한 전제
- {이 Phase 결과로 확보된 조건}
- {다음 Phase 시작 전 필요한 사항}
```

## 6. Why/How 제시 규칙

사람에게 결정을 요청하거나 결정을 기록할 때 반드시 "왜"와 근거를 포함한다.

### 6.1. wtth finding 제시

```markdown
**문제**: reviewer_id를 클라이언트에서 받는데 인증 토큰과 대조하지 않음
**왜 문제인가**: 다른 사용자의 reviewer_id를 넣으면 타인 명의로 리뷰가 생성됨 (권한 우회)
**참조**: 코드 분석 — src/api/review.ts:42 에서 req.body.reviewerId를 검증 없이 사용
**수정안**: 인증 미들웨어에서 토큰의 user_id와 reviewer_id 일치 여부 검증
```

- "왜 문제인가"는 **인과관계**로 서술한다 ("X하면 Y가 발생한다").
- "참조"는 **있으면 필수, 없으면 추론 근거를 명시**한다.
- 3줄 초과 시 finding을 분할한다.

### 6.2. papillon 결정 요청

```markdown
체급: M
근거: 독립 작업 단위 1개 (인증 API), 하위 작업 3개 (엔드포인트, 미들웨어, 테스트)
L이 아닌 이유: 하위 작업 간 의존성이 강해 분리 실행 불가
```

- 판단 근거 + **대안을 기각한 이유**를 포함한다.

### 6.3. RDR 기록

RDR의 "결정 근거" 컬럼에 인과관계 수준으로 기록한다.
단순 요약("불필요")이 아니라 조건과 맥락을 포함한다("현재 데이터 10만건 미만, 동시 수정 빈도 낮음. 50만건 도달 시 재검토").

### 6.4. 작업 요약 카드

설계 선택이 있는 경우에만 "구현 판단" 섹션을 기재한다.
단순 로그 추가, 설정 변경 등은 생략 가능.

### 6.5. 참조(Reference) 유형

| 출처 유형 | 예시 |
|---|---|
| 설계 문서 | 설계서 §2.3, PRD 목표 G2 |
| 기존 결정 기록 | ADR-005, RDR-2026-04-02-auth |
| 코드 | `src/api/review.ts:42` |
| 외부 기준 | OWASP Top 10 A01, K8s 공식 문서 |
| KB 축적 지식 | KB 검색 결과 (인시던트, 패턴 등) |

## 7. 파일 산출물 디렉토리 구조

파이프라인 실행 시 생성되는 산출물의 저장 구조:

```
{프로젝트}/docs/decisions/
  reviews/
    RDR-YYYY-MM-DD-{대상명}.md
  ADR-NNN-{제목}.md
{프로젝트}/docs/runbooks/
  {런북명}.md                      ← shackled 4-B 산출물
{프로젝트}/03-tasks/
  TASK-{ID}-{제목}.md              ← 작업 요약 카드 포함
```

---

## 7.1. 파일·디렉토리 네이밍 컨벤션

### 7.1.1. 기본 패턴

`NN-{kebab-case-name}.md` — 2자리 숫자 프리픽스 + 하이픈 + 케밥케이스.

```
00-index.md
01-interview/
02-design/
03-review/
```

### 7.1.2. 확장 파일 번호

기존 파일 사이에 파일을 추가할 때, 전체 번호를 재배치하지 않고 알파벳 접미사를 붙인다.

```
03-k8s-manifests.md
03b-k8s-workloads.md    ← 03 이후 추가된 관련 파일
04-ci-pipeline.md
```

접미사는 `a`, `b`, `c` 순서로 사용한다. 접미사가 `d` 이상 필요해지면 번호 재배치를 검토한다.

### 7.1.3. L/XL 태스크 디렉토리

L/XL 체급에서 분해된 태스크 디렉토리는 `T{N}-{kebab-case-name}` 패턴을 사용한다.

```
T0-design-revision/
T1-stats-consumer/
T2-stats-batch/
```

태스크 내부 파일은 기본 패턴(§7.1.1)을 따른다.

### 7.1.4. HANDOFF.md

세션 간 맥락 전달을 위한 핸드오프 파일. 숫자 프리픽스 없이 `HANDOFF.md`로 고정한다.

- **생성 주체:** papillon Phase 5.1 (Re-baselining) 또는 Phase 6 (커밋)
- **저장 위치:** 해당 태스크 디렉토리 루트
- **역할:** 다음 세션이 이전 세션의 진행 상태·변경 사항·잔여 작업을 파악하는 데 사용
- **소비 주체:** 다음 세션의 papillon 컨텍스트 스캔

## 8. L/XL 프로젝트 구조 템플릿

L/XL 체급 작업 시 papillon이 생성하는 프로젝트 구조:

```
{프로젝트}/
  00-index.md          ← 전체 조감도 + 현재 진행 상태 + 핵심 제약
  01-design/
    design.md          ← XL인 경우 PRD 포함
  docs/decisions/
    reviews/
      RDR-*.md
    ADR-*.md
  03-tasks/
    TASK-001.md  ✅ 완료
    TASK-002.md  ← 다음 세션에서 S/M으로 수행
    TASK-003.md
```

- `00-index.md`에 "다음 태스크" 정보가 기재된다.
- 다음 세션에서 papillon 호출 시 이 index를 읽고 태스크를 제안한다.

## 9. 네임스페이스 및 패키징 구조

papillon이 네임스페이스 최상위이며, 하위 스킬은 papillon 네임스페이스 안에 설치된다.

**설치 구조:**

```
~/.claude/commands/
  papillon.md           ← 오케스트레이터 (/papillon)
  papillon/
    inquisition.md      ← 요구사항 추출 (/papillon:inquisition)
    wtth.md             ← 리뷰 엔진 (/papillon:wtth)
    shackled.md         ← 구현 엔진 (/papillon:shackled)
```

**호출 방식:**

| 호출 | 동작 |
|---|---|
| `/papillon` | 파이프라인 오케스트레이터 실행 (인터뷰 → 체급 → Phase 자동 전환) |
| `/papillon:inquisition` | 요구사항 추출 단독 실행 |
| `/papillon:wtth` | 리뷰 엔진 단독 실행 |
| `/papillon:shackled` | 구현 엔진 단독 실행 |

**설계 원칙:**
- 하위 스킬은 단독 실행 가능해야 한다. papillon 오케스트레이터에 의존하지 않는다.
- papillon은 하위 스킬을 호출하여 파이프라인을 구성한다.
- 향후 하위 스킬이 추가될 수 있다.

### 9.1. 스킬 파일 포맷 (SKILL.md)

모든 papillon 스킬 파일(`~/.claude/commands/papillon/{NAME}.md` 및 오케스트레이터 `papillon.md`)은 다음 골격을 따른다:

```markdown
# /papillon:NAME — 한 줄 요약

## 개요

이 스킬이 무엇을 하는지 1~3 문단으로 기술한다.
내용은 해당 스킬 설계 문서의 `identity.purpose`를 자연어로 확장한 버전이어야 한다.

(이후 페르소나, 프로세스 등 스킬별 내용)
```

**`## 개요` 섹션 요구사항:**

- 스킬 파일의 H1 직후 **첫 번째 `##` 섹션**으로 존재.
- 내용은 서술문 중심. 테이블·코드·리스트 가능하나 최소 한 문단의 서술 포함.
- 설계 문서 `identity.purpose`의 핵심 키워드를 자연스럽게 포함해야 한다.
- AC 프레임워크의 `ART-IQ-02` 등 "SKILL.md description" 검증은 이 섹션 본문을 대상으로 한다.

<rationale>
설계 문서와 설치된 스킬 간 드리프트를 구조적으로 감지하려면, 스킬 파일에
"자기 설명" 역할의 표준 섹션이 필요하다. H1 title은 너무 짧고, 파일 전체를
대상으로 삼으면 드리프트 감지력이 약해진다.
</rationale>

## 10. 암묵적 피드백 수집 메커니즘

사람과 AI가 상호작용하는 모든 단계(inquisition, wtth 리뷰, 코드 리뷰 등)에 공통 적용한다.

**수집 방식:**
- AI가 인터뷰/리뷰 진행 중 사람의 반응에서 긍정/부정 시그널을 감지하여 내부적으로 기록한다 (예: "이 질문 뜬금없다", "아까 말했잖아", "좋은 질문이네" 등).
- 별도로 물어보지 않는다 — 진행 흐름을 끊지 않는다.
- 종료 시 감지된 피드백을 정리하여 사람에게 확인한다: "이번 세션에서 이런 피드백을 감지했는데 맞나요?"
- 사람이 확인/수정하면 확정한다.

**저장:**
- **프로젝트별 피드백** (이번 프로젝트에만 해당): 해당 산출물 디렉토리 내 `feedback.md`에 기록
- **스킬 피드백** (프로젝트 무관, 스킬 자체 개선): AI 메모리에 저장

**분류:** 종료 시 피드백 확인 단계에서 사람에게 "이건 이번 프로젝트에만 해당하는 건가요, 앞으로도 계속 적용할 건가요?" 물어본다.

**목적:** 완벽한 피드백 수집이 아니라 사람의 기억 회상 트리거. 감지된 내용을 보고 사람이 "아 맞다 그때 그랬지"를 떠올리면 성공이다.

## 11. 문서 작성 기본 규칙

papillon이 산출물(설계서, 요약, 문맥 파일 등)을 생성할 때 적용하는 최소 규칙이다. **사용자의 doc-standard가 있으면 그것을 우선 적용**하고, 없으면 아래 규칙을 기본으로 사용한다.

### 11.1. Context 헤더

모든 설계 문서의 최상단에 아래 헤더를 포함한다:

```markdown
## Context
- **Parent:** [상위 문서 링크]
- **Related:** [관련 문서 링크]
- **Status:**
  - Work: Draft | In-Progress | Completed
  - Review: None | Reviewing | Approved (Human)
```

### 11.2. Status 2축

Status는 Work(작업)과 Review(리뷰) 두 축으로 독립 관리한다.

- **Work:** Draft → In-Progress → Completed
- **Review:** None → Reviewing → Approved

문서가 종료되려면 `Work: Completed + Review: Approved (Human)` 두 조건을 모두 충족해야 한다.

### 11.3. 섹션 넘버링

모든 섹션은 소수점 넘버링(`1.`, `1.1.`, `1.1.1.`)을 사용하여 논리적 계층을 표현한다.

### 11.4. 상호참조

문서 간 연결이 필요하면 마크다운 링크 `[텍스트](./경로)`를 사용한다. 파일 이동/이름 변경 시 참조하는 모든 문서의 링크를 함께 수정한다.

### 11.5. 문서 분할 기준

아래 트리거 중 하나라도 해당되면 파일 분할을 검토한다:
- **T1. 독립 변경 가능성** — 서로 독립적으로 수정될 수 있는 개념이 2개 이상 공존
- **T2. 상호참조 밀도** — 문서 내 섹션 간 상호참조가 3회 이상 발생
- **T3. 200줄 초과** — T1·T2 기준을 재검토하는 경보 (분할 기준 자체는 아님)

### 11.6. 사용자 doc-standard 우선 적용

사용자의 CLAUDE.md 또는 프로젝트 설정에 doc-standard가 지정되어 있으면 §11.1~11.5 대신 사용자의 규칙을 따른다. papillon은 산출물 생성 시 사용자 doc-standard의 존재 여부를 확인하고, 있으면 그것을 로드한다.

## 12. 설계서 SPEC/RAT 태깅 규칙

설계서 본문은 기본적으로 **SPEC(프롬프트 스펙)**이다. 설계 근거(RAT)에 해당하는 부분만 XML 태그로 감싼다. 태그가 없는 본문 = SPEC.

### 12.1. 판별 테스트

> **"이 문장을 빼면 AI의 행동이 달라지는가?"**
> - 달라지면 → SPEC (태그 없음, 그대로 유지)
> - 안 달라지면 → 해당 태그로 감싸기

### 12.2. 태그 종류

| 태그 | 용도 | 예시 |
|---|---|---|
| `<rationale>` | 설계 판단 근거, 대안 기각 이유 | "Opus를 쓰는 이유는 추론력 우선" |
| `<calibration>` | 경험적 사례, 수치 근거 | 체급 캘리브레이션 데이터 |
| `<note>` | 미래 메모, TODO | "팀 확장 시 재검토" |

3종 이하로 유지한다. 태그를 세분화하면 태그 자체가 관리 비용이 된다.

### 12.3. 스킬 변환 시 활용

- `<rationale>` 블록을 제거하면 SPEC만 남는다. SPEC을 토큰 효율적으로 압축하여 스킬 파일을 작성한다.
- 또는 `<rationale>` 블록을 남겨두고 "rationale 태그 내용은 배경 지식이며 실행 지시가 아니다" 한 줄을 추가하는 방식도 가능하다.

### 12.4. 태깅 대상 외

Context 헤더, References, Status 등 **메타데이터**는 SPEC도 RAT도 아니다. 태그를 붙이지 않는다. 스킬 변환 시 구조적으로 걸러지는 영역이다.

### 12.5. 적용 시점

새로 쓰거나 수정하는 설계서부터 적용한다. 기존 문서는 스킬 변환 사이클에서 순차 태깅한다.

**상세 배경:** [Issue-02 #1: 설계서 vs 프롬프트 스펙 정체성 혼란](../80-Issue/02-Issue-02/01-design-vs-prompt-spec.md)

## 13. 파일럿 정량 지표 프로토콜

파일럿·풀사이클 실험의 평가가 서술형 관찰에 머무르면 설계 수정의 정당성이 일화에 의존한다. 이를 방지하기 위해 파일럿마다 공통 지표를 수집한다. 지표는 품질·비용·정확도 각 2종씩 총 6종으로 시작하며, 3~5회 축적 후 재조정한다.

### 13.1. 초기 지표 카탈로그

| 축 | ID | 지표 | 정의 | 데이터 소스 |
|---|---|---|---|---|
| 품질 | Q1 | 크로스 체커 catch 수 | 라운드별 유의미 findings 건수(P0/P1만) | shackled `cross_check_log` 또는 wtth RDR |
| 품질 | Q2 | 사인오프 인용 일치율 | (MCP 문자열 일치 성공 건수) / (P0/P1 수용 건수) | wtth 최종 확인 로그 + 회고 수동 집계 |
| 비용 | C1 | 수렴 라운드 수 | 사인오프 도달까지의 라운드 수 | wtth RDR 파일 수 |
| 비용 | C2 | 사람 개입 건수 | 파일럿 전 구간에서 사람에게 물은 횟수 (분류 없이 합계) | 회고 수동 기록 |
| 정확도 | A1 | 체급 판정 적정성 | inquisition 판정(S/M/L/XL)이 실제 진행 난이도와 일치했는가 — 3단계 평가(과소/적정/과대) | 회고 수동 평가 |
| 정확도 | A2 | output_type 오분류 | code/runbook/doc 판정이 실제와 달랐던 건수 | 회고 수동 기록 |

<rationale>
6종 한정 이유: 지표가 10종을 넘으면 수집 자체가 파일럿 부담이 되어 오히려 데이터 품질이 떨어진다. 축별 2종은 "하나는 정량, 하나는 정성" 원칙을 유지하기 위함(Q1 정량 / Q2 비율, C1 정량 / C2 개입, A1 주관 / A2 정량). 초기엔 자동화 없이 회고에서 수동 집계하는 것을 전제로 설계.
</rationale>

### 13.2. 수집 메커니즘

- **자동 수집 가능 지표(Q1, C1):** 기존 산출물(상태 파일·RDR)에서 집계 가능. 당장은 수동 집계, 의미 확정 후 자동화 검토.
- **수동 수집 지표(Q2, C2, A1, A2):** 파일럿 종료 회고 단계에서 사람이 기록. 템플릿: [07-metrics/retro-template.md](./07-metrics/retro-template.md).
- **저장 위치:** 파일럿 산출물 디렉토리 내 `99-retro.md` (파일럿별). 중앙 집계는 3~5회 후 필요성 재검토.

### 13.3. 운용 원칙

1. **최소 3회 축적 후 해석.** 1~2회 데이터로 설계 수정을 정당화하지 않는다(일화 방지).
2. **지표 제거 조건.** 5회 축적 후 분산이 거의 없거나 의사결정에 쓰이지 않은 지표는 제거.
3. **지표 추가 조건.** 특정 실패 패턴이 3회 이상 반복되고 기존 지표로 포착되지 않을 때만 추가.
4. **스키마화 조건.** 수동 수집 지표가 "반복 패턴이 확정" + "자동 집계 가능"의 두 조건을 모두 충족하면 해당 스킬 상태 파일 스키마에 편입.
