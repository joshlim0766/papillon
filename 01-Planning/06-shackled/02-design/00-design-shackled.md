## 1. 개요

shackled은 papillon 파이프라인의 Phase 4(구현)를 담당하는 독립 스킬이다. 설계서(또는 요구사항 정의서)를 기반으로 AI가 구현을 수행하고 사람이 방향을 확인하는 페어 프로그래밍 협업 도구이다.

### 1.1. 호출 방식

| 호출 방식 | 설명 |
|---|---|
| `/papillon:shackled` | 단독 실행. 파라미터를 직접 제공하거나 shackled이 사람에게 확인하여 진행 |
| papillon 경유 | papillon 오케스트레이터가 Phase 4에서 파라미터를 구성하여 자동 호출 |

### 1.2. 단독 실행 vs 파이프라인 내 실행

| 항목 | 단독 실행 (`/papillon:shackled`) | 파이프라인 내 실행 |
|---|---|---|
| 호출 주체 | 사람이 직접 | papillon 오케스트레이터 |
| 파라미터 제공 | 직접 입력하거나 shackled이 확인 | 오케스트레이터가 구성하여 전달 |
| wtth 호출 | 수행하지 않음. 구현 결과만 반환 | 오케스트레이터가 별도로 wtth 호출 |
| 이후 동작 | 구현 결과를 사람에게 반환. 필요 시 사람이 직접 `/papillon:wtth` 호출 | 오케스트레이터가 Phase 5(코드 리뷰)로 자동 전환 |

### 1.3. 모델

- **작성자·크로스 체커 모두 Sonnet**
- 크로스 체커에 더 높은 추론 모델을 적용하고 싶은 경우, `--cross-check-model` 옵션으로 오버라이드 가능 (확장 포인트)

<rationale>
Sonnet 선택 근거: 구현 작업은 코드 생성·편집이 주이고 설계 판단은 이미 완료된 상태이므로 속도·비용 효율 우선. Opus는 inquisition(추론 중심)에서 사용하고, shackled은 실행 중심이다.
</rationale>

### 1.4. 시블링 구조

shackled과 wtth은 시블링 관계이다. shackled은 구현에만 집중하고 wtth 호출은 papillon 오케스트레이터의 책임이다. shackled이 wtth을 직접 호출하지 않는다.

<rationale>
shackled이 wtth을 직접 호출하지 않는 이유: 각 스킬은 단독 실행 가능해야 한다. shackled이 wtth에 의존하면 단독 실행 시 불필요한 리뷰가 강제되거나, wtth 변경이 shackled에 파급된다.
</rationale>

### 1.5. 리뷰 계층 구분

두 리뷰는 역할이 다르며 중복이 아니다:

| 리뷰 주체 | 시점 | 역할 | 관점 |
|---|---|---|---|
| shackled 내부 크로스 체크 | 태스크별 구현 완료 후 | "설계대로 구현했나?" — 방향 이탈·누락 점검 | 자기 검수 |
| wtth (오케스트레이터 호출) | 전체 구현 완료 후 | "잘 만들었나?" — 품질·안전성·다각도 점검 | 제3자 검수 |

<rationale>
이중 리뷰를 유지하는 이유: 크로스 체크는 "설계대로 ��들었는가"만 확인하는 자기 검수이므로, 코드 품질·보안·성능 등 다각도 검증은 별도 제3자(wtth)가 필요하다. 자기 검수만으로는 시야 밖의 문제를 잡을 수 없다.
</rationale>

## 2. 입력 인터페이스

### 2.1. 파라미터 정의

| 파라미터 | 필수 | 설명 |
|---|---|---|
| `context_path` | 필수 | M 체급: 확정된 설계서 경로. S 체급: 인터뷰 요약(`00-summary.md`) 또는 요구사항 경로. 없으면 진행 거부 |
| `weight` | 필수 | 체급. 유효값: `S` \| `M`. 유효 범위 밖 입력 시 §2.4 참조 |
| `output_type` | 필수 | 산출물 유형 (`code` / `runbook`). 4-A/4-B 분기 및 페르소나 조합 결정에 사용 |
| `code_target` | 조건부 | `output_type: code`일 때 필수. 유효값: `backend` \| `frontend` \| `fullstack`. 크로스 체커 조합 결정에 사용. §5.1 참조 |
| `task_id` | 선택 | 중단 후 재개 시 이어받을 태스크 식별자 |
| `exec_mode` | 선택 | 실행 모드 (`auto` / `pair`). 기본값 `auto`. 페어 모드는 확장 포인트 예약 |

**체급명 사용자 제시 규칙:** 사용자에게 체급을 안내할 때는 S/M/L/XL 코드 대신 사용자 친화적 용어로 치환한다 (예: "소규모 작업", "중규모 설계 기반 작업" 등). 내부 파라미터값은 코드 그대로 사용한다.

### 2.2. context_path 없음 처리

`context_path`가 제공되지 않으면 shackled은 진행을 거부하고 사람에게 다음을 안내한다:
- 설계서 또는 요구사항 경로 제공 요청
- M 체급: 설계서가 없으면 papillon Phase 2~3(설계 리뷰)부터 시작 권장
- S 체급: `00-summary.md` 또는 요구사항 정의서 경로 제공 요청

### 2.3. 단독 실행 시 파라미터 처리

파라미터 없이 단독 실행된 경우, 2단계로 확인한다:

1. 구현할 작업에 대한 설계서 또는 요구사항 경로 (`context_path`)
2. context_path 분석 후 체급(`weight`), 산출물 유형(`output_type`), `code_target`(code인 경우)을 **동시에 제안**하고 사람이 확인

### 2.4. 유효 범위 밖 입력 처리

`weight`에 `S`/`M` 이외의 값(L, XL 등)이 입력된 경우:
- 진행을 거부하고 안내 메시지를 제시한다: "L/XL 체급은 shackled의 실행 단위가 아닙니다. papillon 오케스트레이터에서 S/M 단위로 분할한 뒤 호출하세요."
- 단독 실행 시: papillon 오케스트레이터 실행을 권장한다.

### 2.5. code_target 자동 판별

`code_target`이 제공되지 않은 경우, shackled이 context_path의 내용을 분석하여 판별하고 사람에게 확인받는다.

**판별 기준 (우선순위 순):**
1. 설계서에 대상 기술 스택이 명시된 경우 (예: "React 컴포넌트", "Spring Boot API")
2. 프로젝트 디렉토리의 기술 스택 파일 (예: `package.json`, `pom.xml`, `go.mod`)
3. 설계서 내 키워드 분석 (UI/컴포넌트 → frontend, API/DB → backend, 양쪽 → fullstack)
4. 위 기준으로 판별 불가 시 사람에게 직접 선택 요청

### 2.6. 태스크 ID 채번 규칙

**기본 규칙:** `TASK-{output_type 접두어}-{순번}`
- 접두어: `A` = code, `B` = runbook
- 순번: `01`부터 순차 부여
- 예: `TASK-A-01`, `TASK-A-02`, `TASK-B-01`

**사람 오버라이드:** 사람이 별도의 채번 규칙을 정의하여 기본 규칙을 대체할 수 있다. 오버라이드된 규칙이 부적절하다고 판단되면(예: 중복 가능성, 식별 불가능한 형식 등) shackled이 재고를 요청한다. 사람이 수용하면 기본 규칙으로 복귀하고, 기각하면 사람의 규칙을 수용한다.

## 3. 프로세스: 기본 모드 (auto)

### 3.1. 진입

`context_path`를 로드하여 컨텍스트를 파악한 뒤, 아래 테이블에 따라 진입 분기를 결정한다.

| 조건 | 처리 |
|---|---|
| `task_id` 있음 + 상태 파일 매칭 | 해당 태스크부터 재개 ([02-state-management.md §6](./02-state-management.md)) |
| `task_id` 있음 + 상태 파일 불일치 | 사람에게 불일치 알림 + 상태 파일의 태스크 목록 제시 → 사람이 결정 |
| `task_id` 있음 + 상태 파일 없음 | 에러: "해당 task_id의 상태 파일을 찾을 수 없습니다" + 신규 진입 제안 |
| `task_id` 없음 + 상태 파일 존재 | 사람에게 중단된 작업이 있음을 알리고 재개 여부 확인 |
| `task_id` 없음 + 상태 파일 없음 | 신규 진입 |

### 3.2. 태스크 파싱

#### 3.2.1. M 체급 (설계서 기반)

1. 설계서에서 구현 태스크 목록을 추출한다.
2. 태스크 간 의존 관계를 파악하고 실행 순서를 결정한다.
3. 추출된 태스크 목록을 사람에게 제시하고 확인받는다.
4. ★ 사람이 순서를 조정하거나 태스크를 추가/제외할 수 있다.

#### 3.2.2. S 체급 (요구사항 기반)

1. 인터뷰 요약(`00-summary.md`) 또는 요구사항 정의서를 읽는다.
2. 구현에 필요한 작업 단위를 도출한다.
3. 도출된 작업 목록을 사람에게 제시하고 확인받는다.
4. ★ 사람이 확인 또는 조정한다.

#### 3.2.3. 태스크 목록 제시 및 사람 확인

태스크 목록 제시 시 각 태스크에 아래 정보를 포함한다:

| 항목 | 설명 |
|---|---|
| 태스크 ID | §2.6 채번 규칙에 따라 부여 |
| 제목 | 태스크의 한 줄 제목 |
| 범위 요약 | 무엇을 구현하는가 (1~2문장) |
| 의존 관계 | 의존하는 태스크 ID + 근거 (없으면 "없음") |
| 크로스 체커 | §5.1에 따른 크로스 체커 페르소나 조합 |

**의존 관계 위반 경고:** 사람이 순서를 조정할 때, 의존 관계가 위반되면 shackled이 경고를 표시한다. 사람이 경고를 확인하고도 순서를 유지하면 그대로 따른다.

**페르소나 조합 판별:** `code_target` 기반으로 전체 태스크의 크로스 체커 조합을 이 단계에서 결정한다 (§5.1 참조). 태스크별로 다른 조합이 필요한 경우(풀스택 프로젝트에서 일부 태스크가 백엔드 전용 등) 개별 명시하고 사람에게 확인받는다.

**사람 응답 분기:**

| 응답 | 처리 |
|---|---|
| 승인 | 태스크 목록 확정, §3.3 순차 실행 시작 |
| 조정 | 순서 변경, 태스크 추가/제외 반영 후 재제시 |
| 재파싱 요청 | 컨텍스트를 다시 읽고 태스크 목록을 처음부터 재생성 |
| 구현 취소 | shackled 종료 (상태 파일 생성하지 않음) |

### 3.3. 태스크별 순차 실행

태스크 하나씩 순서대로 수행한다. 한 번에 전체 태스크를 구현하지 않는다.

<rationale>
순차 실행 이유: 한 번에 전체를 구현하면 크로스 체크 시점에 문제가 누적되어 수정 비용이 기하급수적으로 증가한다. 태스크 단위로 구현→크로스 체크→사람 확인을 반복하면 문제를 조기에 잡을 수 있다.
</rationale>

각 태스크에 대해 아래 루프를 반복한다:

#### 3.3.1. 구현

작성자 페르소나가 컨텍스트(설계서 또는 요구사항)를 펼쳐두고 해당 태스크를 구현한다.

- `output_type: code` — 코드 작성·편집
- `output_type: runbook` — 런북 섹션 작성 ([50-runbook-template.md](../../50-runbook-template.md) 형식 준수)

#### 3.3.2. 크로스 체크 (§6 참조)

크로스 체커 페르소나가 구현 결과를 설계서(또는 요구사항)와 대조하여 방향 이탈·누락을 점검한다.

- finding이 없으면 다음 단계로 진행한다.
- finding이 있으면 작성자 페르소나가 수정하고 크로스 체크를 재수행한다.
- 내부 크로스 체크를 통과해야 사람 확인 단계로 넘어간다.

#### 3.3.3. 사람 확인

구현 결과를 사람에게 제시한다. 제시 내용:
- **진행률**: "태스크 [N/전체] 완료: {태스크 제목}"
- **구현된 내용 요약**
- **크로스 체크 결과**: 이상 없음 확인, 또는 수정이 발생한 경우 수정 전/후 요약 (무엇이 지적되었고 어떻게 바뀌었는지). 사람이 상세 변천사를 요청하면 전체 크로스 체크 이력 제시
- **다음 태스크 예고**: 다음 태스크의 제목 + 1줄 범위 요약

사람 응답에 따라 분기한다:

| 응답 | 처리 |
|---|---|
| 승인 | 다음 태스크로 진행 |
| 피드백 제공 | 피드백을 반영하여 현재 태스크 재구현 → 크로스 체크 → 재제시. 재제시 시 사람의 피드백 요약 + 각 피드백 포인트별 반영 내역을 포함 |
| 거부 (전면 재작성 요청) | 구현을 폐기하고 태스크 처음부터 재시작 |
| 진행 불가 선언 | §7 진행 불가 처리 흐름으로 분기 (사람 주도 선언) |

### 3.4. 인테그레이션

전체 태스크 완료 후 수행한다.

#### 3.4.1. output_type별 인테그레이션 동작

| output_type | 동작 |
|---|---|
| `code` | 전체 변경 파일 목록 취합 + 빌드/lint 통과 확인(도구 없으면 스킵 후 사람에게 알림) + 태스크 간 인터페이스 일치 검증 |
| `runbook` | 전체 절차 순서 재검증 + 중복 사전조건 정리 + 단계 간 의존 관계 확인 |

#### 3.4.2. 수행 주체

크로스 체커 페르소나가 인테그레이션 확인을 수행한다. 태스크 단위 크로스 체크가 "개별 태스크 ↔ 설계" 대조라면, 인테그레이션 확인은 "태스크 간 결합 ↔ 설계 전체 목표" 대조이다.

**인테그레이션 체크 항목:**
- 태스크 간 인터페이스(함수 시그니처, 데이터 구조 등)가 일치하는가
- 통합 시 누락되거나 충돌하는 부분이 없는가
- 설계서(또는 요구사항)의 전체 목표가 달성되었는가

**인테그레이션 finding 처리:** [01-cross-check.md §5](./01-cross-check.md)와 동일한 루프(작성자 수정 → 재점검 → 반복 상한)를 적용한다. 단, 인테그레이션 finding은 여러 태스크에 걸칠 수 있으므로 관련 태스크를 명시하여 보고한다.

#### 3.4.3. 사람 제시 및 최종 확인

통합 결과를 사람에게 제시한다. 제시 내용:

- **전체 태스크 목록과 완료 상태**
- **설계 대비 달성 현황**: 구현됨 / 미구현 / 변경됨
- **통합 시 발생한 조정 사항** (있는 경우)
- **최종 산출물 목록**

★ 사람이 승인하면 산출물 반환 단계로 진행한다.

### 3.5. 4-B(runbook) 전용 프로세스

`output_type: runbook`일 때, §3.3~§3.4의 기본 프로세스에서 산출물이 런북 문서가 된다. shackled은 런북 **생성**까지만 담당하며, 런북 **실행**은 shackled의 책임이 아니다.

<rationale>
런북 생성과 실행을 분리하는 이유: 런북 실행은 인프라에 실제 영향을 주는 위험 행위다. 생성(설계 영역)과 실행(운영 영역)의 책임을 분리하여, 실행 전에 반드시 별도 검증(wtth 운영 절차 리뷰)을 거치도록 한다.
</rationale>

**shackled 내 흐름:**
1. 태스크별 런북 섹션 작성 (§3.3.1)
2. 크로스 체크: 절차 완결성, 롤백 포함 여부, 검증 커맨드 존재 여부 ([01-cross-check.md §4](./01-cross-check.md))
3. 사람 확인 (§3.3.3)
4. 인테그레이션: 전체 절차 순서 재검증 (§3.4)
5. 완성된 런북을 산출물로 반환

**이후 흐름 (shackled 외부):**
- 파이프라인 내 실행: 오케스트레이터가 wtth(운영 절차 리뷰) 호출 → 사람 의사결정 → 별도 실행 스킬 또는 오케스트레이터가 런북 실행 (papillon §3.4 참조)
- 단독 실행: 사람에게 런북을 반환하고 종료. 실행은 사람이 별도 판단

### 3.6. 산출물 반환

구현 결과물과 함께 작업 요약 카드(공통 규격 §4 포맷) 초안을 생성한다.

- `output_type: code` — 변경된 코드 파일 목록, 작업 요약 카드
- `output_type: runbook` — 완성된 런북 파일 경로, 작업 요약 카드

**단독 실행 종료 시:** 사람에게 결과를 반환하고 다음 행동 선택지를 제시한다:

| output_type | 선택지 |
|---|---|
| `code` | 1. 코드 리뷰 실행 (`/papillon:wtth`) 2. 커밋 진행 3. 종료 |
| `runbook` | 1. 운영 절차 리뷰 실행 (`/papillon:wtth`) 2. 종료 |

**파이프라인 내 실행 시:** 오케스트레이터에게 결과를 반환한다.

## 4. 산출물 유형별 분기

프로세스 골격(§3)은 4-A와 4-B가 공유한다. 분기점은 아래와 같다.

| 구분 | 4-A (code) | 4-B (runbook) |
|---|---|---|
| 대표 작업 | 기능 구현, 버그 수정, 리팩토링 | 인프라 조작, 데이터 마이그레이션, 환경 설정 |
| 구현 산출물 | 소스 코드 파일 | 런북 문서 ([50-runbook-template.md](../../50-runbook-template.md) 형식) |
| 페르소나 조합 | §5.1 참조 | §5.1 참조 |
| 크로스 체크 기준 | 설계서의 기능 명세, 인터페이스 정의 | 절차 완결성, 롤백 절차 포함 여부, 검증 커맨드 존재 여부 |
| shackled 산출물 | 코드 파일 + 작업 요약 카드 | 런북 문서 + 작업 요약 카드. 실행은 별도 스킬/오케스트레이터 책임 (papillon §3.4 참조) |

## 5. 전용 페르소나

shackled은 wtth의 전문가 풀과 별도로 구현 특화 페르소나를 사용한다. 페르소나는 작성자와 크로스 체커 쌍으로 구성된다.

<rationale>
wtth 페르소나와 별도로 구현 전용 페르소나를 두는 이유: wtth 전문가는 "리뷰 관점"(비판적 분석)으로 설계되어 있고, shackled 페르소나는 "구현 관점"(생산+자기검수)으로 설계되어 있다. 같은 도메인(BE 등)이라도 역할이 다르므로 페르소나를 공유하면 관점이 혼재된다.
</rationale>

각 페르소나의 4축 정의(관점/금기/판단기준/체크리스트)는 [personas/](./personas/) 디렉토리의 개별 파일에서 관리한다. wtth 페르소나(`01-Planning/personas/`)와는 독립된 정의이다.

### 5.1. 페르소나 조합 결정

`output_type`과 `code_target` 파라미터로 조합을 결정한다.

| output_type | code_target | 작성자 | 크로스 체커 |
|---|---|---|---|
| `code` | `backend` | shackled-CODE | shackled-BE |
| `code` | `frontend` | shackled-CODE | shackled-FE |
| `code` | `fullstack` | shackled-CODE | shackled-BE + shackled-FE |
| `runbook` | — | shackled-SRE | shackled-SEC |

조합은 §3.2(태스크 파싱) 단계에서 전체 태스크에 대해 한 번에 결정하고 사람에게 확인받는다.

### 5.2. 각 페르소나 역할

| 페르소나 | 역할 | 정의 파일 |
|---|---|---|
| shackled-CODE | 코드 작성자. 설계서의 인터페이스·명세를 기반으로 코드를 작성한다 | [personas/shackled-code.md](./personas/shackled-code.md) |
| shackled-BE | 백엔드 크로스 체커. API 설계 준수, 데이터 처리 로직, DB 연동 방식을 점검한다 | [personas/shackled-be.md](./personas/shackled-be.md) |
| shackled-FE | 프론트엔드 크로스 체커. UI 명세 준수, 상태 관리, API 연동 방식을 점검한다 | [personas/shackled-fe.md](./personas/shackled-fe.md) |
| shackled-SRE | 런북 작성자. 실행 절차를 단계별로 작성하고 검증 커맨드를 포함한다 | [personas/shackled-sre.md](./personas/shackled-sre.md) |
| shackled-SEC | 보안 크로스 체커. 절차의 보안 위험, 권한 과다 부여, 롤백 절차 누락을 점검한다 | [personas/shackled-sec.md](./personas/shackled-sec.md) |

## 6. 크로스 체크 메커니즘

상세: [01-cross-check.md](./01-cross-check.md)

### 9.2. 설계 방향 (상세 설계 미포함)

- JetBrains MCP 한계 및 tmux 연동 방식은 별도 설계 필요

</rationale>
