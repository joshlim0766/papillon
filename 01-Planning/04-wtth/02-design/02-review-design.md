## 2. ADR 리뷰

### 2.1. 대상
ADR (아키텍처 결정 기록)

### 2.2. 전문가 풀
- 기본 풀: ARCH, SEC
- 선택적 투입: BE, DBA, SRE (오케스트레이터가 ADR 내용을 분석하여 관련 도메인 전문가 추가 제안)

### 2.3. 의사결정 방식
finding별 수용/기각 (코어 [§2.1](./00-core.md#21-finding별-수용기각-prd--설계--운영-절차-리뷰) 참조)

### 2.4. ADR 승격 흐름
설계 리뷰에서 P0/P1 수용 건 중 설계 변경을 수반하는 항목에 대해 ADR 승격을 제안한다. 승격 기준 상세는 코어 [§3.2](./00-core.md#32-adr-승격-규칙) 참조.
ADR 승격 시 papillon Phase 2.1에서 본 모드(ADR 리뷰)로 호출된다.

## 3. 공통 프로세스와의 차이점 (설계 리뷰 + ADR 리뷰 공통)

### 스킵 단계
- 없음. 코어의 전체 프로세스를 그대로 사용한다.

### 변형 단계
- **전문가 풀 선별**: 설계 리뷰는 6인 기본 풀 + 3인 선택적 투입에서 추천 상한 3인 내로 선별한다 (코어 §1.3). ADR 리뷰는 ARCH + SEC 2인 기본이며, 선택적 투입(BE, DBA, SRE)은 ADR 내용 분석 후 제안한다.
- **수정 경로 리뷰 적용**: 설계 리뷰에서 기존 설계서 수정(papillon Step 3-B) 시, 코어 §2.3 수정 경로 리뷰를 적용한다. ADR 리뷰에서는 적용하지 않는다 (ADR은 신규 생성이므로).

### 추가 규칙
- **설계 리뷰 → ADR 승격 연계**: 설계 리뷰에서 도출된 P0/P1 수용 건이 ADR 승격 대상이면 코어 §3.2에 따라 승격을 제안한다. 승격 시 ADR 리뷰 모드로 재호출된다.
- **선택적 투입 조건 (설계 리뷰)**: SPEC은 워크플로우·프로세스 명세가 핵심일 때, IX는 사람-AI 상호작용 포함 시, DBA는 DB 스키마 변경 포함 시 투입 제안한다.

<rationale>
설계 리뷰의 기본 풀을 6인으로 넓게 잡은 이유: 설계서는 아키텍처, 백엔드, 프론트엔드, 인프라, 보안, 제품 관점 등 다양한 관심사가 교차한다. 넓은 풀에서 리뷰 대상에 맞는 3인을 선별하는 방식이, 고정 3인으로는 커버하기 어려운 다양한 설계서 유형에 대응할 수 있다.
</rationale>
