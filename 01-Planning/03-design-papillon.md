## 2. 파이프라인 공통 규칙

### 2.1. 작업 중 체급 변경(승격)

파이프라인 진행 중 사람 또는 AI가 체급이 맞지 않다고 판단하면 승격을 제안/요청할 수 있다. 사람이 **승인하면 승격 동작을 수행**하고, **거부하면 현재 체급으로 계속** 진행한다.

| 승격 | 트리거 | 승인 시 | 거부 시 |
|---|---|---|---|
| S → M | 구현 중 예상치 못한 설계 판단이 필요해짐 | Phase 1 Step 3(설계 초안)로 이동 | S로 계속 구현 |
| M → L | 크기 점검 또는 리뷰에서 태스크 분할이 필요해짐 | 조감도 + 태스크 목록 생성 → 파이프라인 종료 | M으로 계속 진행 |

## 3. Phase별 상세

### 3.0. 파이프라인 진입: 컨텍스트 스캔

papillon 호출 시 CWD에서 기존 산출물을 스캔하여 진입점을 결정한다.

| 발견된 산출물 | 추론 | 진입점 |
|---|---|---|
| `00-index.md` + 미완료 태스크 | L/XL 프로젝트 진행 중 | index 기반 다음 태스크 제안 |
| `docs/decisions/` (RDR/ADR) 존재, index 없음 | 이전 S/M 작업 이력 있음 | 마지막 RDR 요약 + "이어서 할 건지, 새 작업인지" 확인 |
| `00-interview/` 디렉토리 존재 | 인터뷰 진행 중 세션 끊김 | 미결 문맥 파일 기반 인터뷰 재개 제안 |
| 아무것도 없음 | 이 저장소에서 첫 사용 | Phase 1(인터뷰)부터 시작 |

- 스캔 결과를 사람에게 제시하고 진입점을 **제안**한다. 사람이 승인하면 해당 지점으로 진입하고, 오버라이드하면 사람이 지정한 지점으로 진입한다.
- 이 스캔은 papillon 오케스트레이터의 진입 로직이다. wtth의 "이전 결정 스캔"(Core §3.3)과는 역할이 다르다 — Core §3.3은 리뷰 시작 시 기결정 사항을 전문가에게 전달하는 것이고, 여기는 파이프라인 자체의 진입점을 결정하는 것이다.

### 3.1. Phase 1: 인터뷰 + 체급 판정 + 초안

#### Step 1: 구조화 인터뷰 (Opus)

**인터뷰 모델 선택 근거:** 인터뷰는 사람 응답 대기가 대부분이라 AI 응답 속도가 병목이 아니다. 반면 함의 감지, 암시적 제약 추출, 문맥 경계 판단은 추론력이 결과물 품질에 직접 영향을 준다. Opus를 사용한다.

**인터뷰 재개 (§2.0에서 `00-interview/` 발견 시):**
1. 기존 문맥 파일을 읽어 상태를 복원한다 (Open/Settled 현황 파악).
2. 컨텍스트 수집(항목 2)을 다시 수행한다 — 세션 간에 데이터가 변경되었거나 새 데이터가 추가되었을 수 있다.
3. 사람에게 현재 상태를 요약 제시한다 (미결 N건, 확정 N건, 변경된 컨텍스트가 있으면 함께).
4. 사람이 재개 지점을 결정한다.

**신규 인터뷰:**

1. 사람이 요구사항을 전달한다 (자연어, 구조화된 문서, 이슈 링크, 이미지 등 형태 무관).
2. **컨텍스트 수집**: Opus가 작업에 활용 가능한 데이터를 사람에게 확인한다.
   - 어떤 데이터가 있는지 (소스코드, 설계문서, DB 스키마, KB, API 명세 등)
   - 어떤 형태인지 (파일, 저장소, 외부 시스템 등)
   - 어떻게 접근하면 되는지 (경로, MCP, URL 등)
   - 사람이 지시한 데이터를 읽고 현황을 파악하여 답변한다. 매 턴 종료 시 "더 봐야 할 데이터가 있는지" 확인한다. 사람이 없다고 하면 항목 3으로 진행한다.
3. Opus가 요구사항 + 수집된 컨텍스트를 바탕으로 인터뷰 필요 여부를 판단하고 사람에게 **제안**한다.
   - 인터뷰 제안 시: 구체화가 필요한 포인트를 함께 제시한다.
   - 인터뷰 불필요 판단 시: 이유와 함께 Step 2(체급 판정) 직행을 제안한다. 수락 시 항목 4~11을 건너뛰고 컨텍스트 수집(항목 2) 결과가 Step 2의 입력이 된다.
   - ★ 사람이 최종 결정한다 (제안 수락 / 오버라이드).
4. Opus가 요구사항 + 수집된 컨텍스트를 분석하여 다음을 수행한다:
   - 기술 키워드, 도메인 개념 추출
   - 각 키워드 뒤에 숨겨진 함의(implication)와 암시적 제약 후보 감지
   - 기술적 결정의 의도, 실행 순서 제약, 핵심 전제 조건에 대해 사람에게 확인 질문
5. 사람이 답변하면, 답변 내용에서 추가 확인이 필요한 부분을 재질문한다.
6. **문맥 단위 기록**: 하나의 관심사에 대한 질답이 마무리되면, 해당 내용을 정리하여 문맥 파일을 생성한 뒤 다음 관심사로 넘어간다.
   - 저장 경로: `{프로젝트}/00-interview/`
   - Opus가 질답 흐름에서 관심사 경계를 판단하여 파일 생성을 **제안**한다 (예: "여기까지를 인증 전략으로 정리해도 될까요?"). 사람이 확인하면 파일을 생성한다 (예: `01-auth-strategy.md`, `02-data-model.md`).
   - 각 파일에 상태를 표기한다: **미결**(Open) — 아직 확인이 필요한 항목 있음 / **확정**(Settled) — 결론 도출 완료 / **보류**(Deferred) — 의도적으로 미뤄둠, 사유 기재.
   - 다음 관심사 진행 중 이전 관심사에 영향을 주는 내용이 나오면, 해당 문맥 파일로 돌아가 수정한 뒤 현재 관심사로 복귀한다.
   - 세션이 끊어져도 파일별 상태를 보고 재개 가능하다.
7. **문맥 전환 감지**: 사람이 다른 주제로 넘어갈 때, Opus는 다음을 수행한다:
   - 이전 문맥에 미결 항목이 있으면 환기한다 (예: "인증 방식은 아직 OAuth vs API Key 중 결론이 안 났는데, 먼저 데이터 모델로 넘어갈까요?"). 사람이 넘어가겠다고 하면 해당 문맥을 미결 상태로 두고 진행한다.
   - 지금까지 확인된 내용을 요약하여 사람에게 보여준다. 사람이 수정/보충하면 해당 문맥 파일에 반영 후 계속한다.
   - 사람이 직접 요약을 요청한 경우에도 동일하게 제시한다.
8. **확정 사안 번복 처리**: 사람이 확정(Settled)된 문맥을 뒤집으려 할 때, Opus는 다음을 수행한다:
   - 해당 변경이 다른 확정 문맥에 미치는 영향을 분석하여 사람에게 제시한다 (예: "인증을 API Key로 바꾸면 세션 관리, 권한 모델 문맥도 재검토가 필요합니다").
   - 사람이 영향 범위를 확인한 뒤 판단한다: 변경 범위를 축소 조정 / 영향받는 문맥과 함께 수정 진행 / 전체 인터뷰 재시작.
   - 수정 진행 시 영향받는 문맥 파일의 상태를 미결(Open)로 되돌리고, 해당 문맥들을 순차 재검토한다.
9. **사람의 표현 패턴에 따른 보정 질문**:
   - 주제를 건너뛰며 말함 → 미결 문맥 환기
   - 암시적으로만 말함 ("그냥 빠르면 돼") → 구체화 질문 ("응답 시간 기준이 있나요?")
   - 솔루션부터 말함 ("Redis 쓰자") → 의도 역추적 ("Redis를 쓰려는 이유가 캐시 때문인가요?")
   - 핵심 제약을 당연하게 생략 → 도메인 기반 체크리스트로 누락 감지
10. **인터뷰 종료 규칙**:
    - AI가 "추가 확인 사항이 없다"고 판단하면 종료를 **제안**한다. 사람이 거부하면 추가로 확인할 항목을 지시한다.
    - 사람이 직접 "이만하면 됐다"고 판단해도 종료를 개시한다.
    - 미결(Open) 상태인 문맥 파일이 있으면 종료 전에 항목별로 사람에게 처리를 요구한다: **지금 결정**(Settled) / **의도적 보류**(Deferred, 사유 필수). 모든 미결 항목이 Settled 또는 Deferred가 되어야 종료할 수 있다.
12. 종료 시 `00-interview/00-summary.md`에 전체 요약을 생성한다: 확인된 요구사항, 핵심 제약, 암시적 전제 조건 목록, Deferred 항목과 사유. 이 디렉토리가 이후 체급 판정과 Draft 생성의 입력이 된다.
13. **체급 판정**: 인터뷰 결과(또는 컨텍스트 수집 결과)를 바탕으로 체급을 판정한다. Opus가 아래 두 축을 기준으로 종합 판단한다:
    - **설계 판단의 복잡도**: 의사결정이 필요한 기술적/구조적 선택지가 얼마나 있는가
    - **독립 실행 가능한 단위 수**: 하나의 세션에서 완결할 수 있는가, 분할이 필요한가

    | 체급 | 핵심 기준 | 보조 참고 지표 | 예시 |
    |---|---|---|---|
    | **S** | 설계 판단 없음, 단일 작업으로 완결 | 변경 파일 수, 영향 범위 | 버그 수정, 설정 변경, 유틸 함수 추가 |
    | **M** | 단일 관심사 내 설계 판단 필요 | 문서 분량, API 수 | API 엔드포인트 1개 + 테스트 + DB 마이그레이션 |
    | **L** | 독립 실행 가능한 M급 태스크가 2개 이상 | 태스크 간 의존 관계 | 대시보드 위젯 6개 (각각은 M 이하) |
    | **XL** | 순차 Phase가 필요하거나, PRD 수준의 목표/범위 정의가 선행되어야 함 | Phase 간 의존성 | 시스템 이관, 신규 서비스 구축 |

    보조 참고 지표(파일 수, API 수 등)는 판정의 근거로 참고하되, 기계적으로 적용하지 않는다.
    경계 사례: Opus가 후보 체급과 근거를 why/how 포함하여 제시한다. ★ 사람이 최종 판단한다.
    확정된 체급과 판정 근거를 `00-interview/00-summary.md`에 기록한다.

#### Step 2: 체급별 분기 (Opus)

| 체급 | 동작 |
|---|---|
| **S** | 설계 생략 → 바로 Phase 4(구현) 진입 |
| **M** | 설계 초안 생성 → Phase 1.5(크기 점검) → Phase 2~6 수행 |
| **L** | 조감도(00-index.md) + 태스크 목록 생성 → **파일로 저장하고 종료**. 각 태스크는 별도 세션에서 S/M으로 수행 |
| **XL** | PRD 작성 → Phase 2(wtth PRD 리뷰 모드) → Phase 3(종결 판단) → Phase 분할 + 태스크 목록 생성 → **파일로 저장하고 종료**. 각 Phase 내 태스크는 별도 세션에서 L/M/S로 수행 |

**L/XL의 핵심 원칙:**
- L/XL은 파이프라인의 실행 단위가 아니다. **실행 결과물의 저장소이자 추적 구조**이다.
- 파이프라인이 실제로 도는 단위는 **항상 S/M**이다.
- L/XL의 산출물은 프로젝트 구조(index + 태스크 목록 + RDR/ADR + 작업 요약 카드)로 남는다.
- 다음 세션에서 papillon 호출 시 index를 읽고 "다음 태스크"를 제안한다.

L/XL 산출물 구조: [02-common-spec.md 섹션 8](./02-common-spec.md) 참조.

#### Step 3: Draft 생성 (M 체급, Sonnet)

1. 인터뷰 결과를 기반으로 doc-standard(01-doc-standard.md) 기준으로 설계 초안을 작성한다.
2. 인터뷰에서 확인된 **핵심 제약은 00-index.md에 명시적으로 기재**한다.
3. 초안을 사람에게 보여주고 Phase 1.5로 자동 전환한다.

### 3.2. Phase 1.5: 태스크 크기 점검 (1차, M 체급)

초안이 나온 직후, 리뷰 진입 전에 **실제 산출물 기반으로** 크기를 점검한다.

**Step 1: 단순 판별 (Haiku)**

| 기준 | 판단 방법 |
|---|---|
| 설계 문서 A4 2장 초과 | 초안 문서 길이 측정 |
| 관련 API 3개 초과 | 초안에서 API/엔드포인트 수 확인 |

**Step 2: 분할 제안 (Sonnet, 기준 초과 시만)**

- Haiku가 기준 초과를 판별하면 Sonnet이 분할안을 제안한다.
- ★ 사람이 승인하면 분할 후 Phase 2로, 거부하면 그대로 Phase 2로 진행.
- 리뷰 1라운드 후 wtth이 건수/심각도/패턴을 분석하여 태스크 크기 경고를 제안할 수 있다 (2차 안전망).

### 3.3. Phase 2: 설계 리뷰

1. wtth을 **설계 리뷰 모드**로 호출한다.
2. 기존 wtth 프로세스 그대로 수행 (전문가 제안 → 승인 → 병렬 실행 → 항목별 의사결정).
3. 리뷰 종료 시:
   - RDR 자동 생성 → `docs/decisions/reviews/`에 저장
   - finding 전부 처리 완료 후, P0/P1 수용 건 중 설계 변경 수반 항목 → ADR 승격 대상을 **일괄 제안**
4. ADR 승격 시 → Phase 2.1로 진입:
   - AI가 ADR 초안 자동 생성 → ★ 사람이 확인/수정 → wtth로 ADR 리뷰 (finding별 수용/기각 + RDR 생성) → ★ 사람이 ADR 자체에 대해 판단
   - **ADR 판단 분기 (wtth 리뷰 종료 후):**
     - 승인 → ADR 확정, 다음 ADR 승격 건 처리 또는 Phase 3로
     - 방향 기각 → ADR을 다른 방향으로 재작성 → Phase 2.1 재진입 (RDR에 기각 사유 기록)
     - ADR 자체 불필요 판정 → ADR 드롭, finding 수용은 유지하고 구현에서 처리 (RDR에 드롭 사유 기록)

### 3.4. Phase 3: 종결 판단

1. 오케스트레이터가 현재 상태를 분석한다:
   - 미해결 P0: 0건인가
   - 미해결 P1: 0건인가
   - 리뷰 대상 문서 Status: Work: Completed 가능한가
2. 조건 충족 시 종결을 제안한다.
3. 사람이 승인하면 설계 확정 (Status: Completed + Approved).
4. 사람이 거부하면 사유를 받고 Phase 2로 복귀.

### 3.5. Phase 4: 구현

오케스트레이터가 태스크 내용을 분석하여 4-A/4-B를 판별한다:
- **4-A** — 산출물이 코드인 경우 (기능 구현, 버그 수정, 리팩토링 등)
- **4-B** — 산출물이 실행 절차인 경우 (인프라 조작, 데이터 마이그레이션, 환경 설정 등)

판별 결과를 사람에게 제시하고 확인받는다.

**4-A. 개발 작업 (코드)**
1. 확정된 설계서의 태스크 목록을 순서대로 구현한다.
2. Phase 5로 자동 전환.

**4-B. 인프라/수행 작업**
1. 태스크 내용을 기반으로 **런북(절차서)을 생성**한다. 런북은 [50-runbook-template.md](./50-runbook-template.md)의 형식을 따른다.
2. wtth을 **운영 절차 리뷰 모드**로 호출한다.
3. ★ 사람: finding별 수용/기각.
4. 절차서 확정 후, 각 단계를 순서대로 실행한다.
5. 각 단계의 검증 커맨드를 수행하여 성공 여부를 판단한다.
6. 전체 완료 확인 후:
   - 커밋 대상이 있는 경우 (런북, 설정 파일, RDR 등) → Phase 6으로 전환
   - 커밋 대상이 없는 경우 → RDR 생성 후 파이프라인 종료

### 3.6. Phase 5: 코드 리뷰

wtth을 **코드 리뷰 모드**로 호출한다. 리뷰 대상은 `git diff` 기반 변경사항.
상세 프로세스(finding 도출 → diff 묶음 → 순차 제시 → 승인/반려/직접수정/기각)는 [wtth 코어 §2.2](./04-design-wtth/00-core.md) 참조.
종결 시 RDR 생성 → Phase 6.

### 3.7. Phase 6: 커밋 + PR + 완료

1. 커밋을 수행한다 (`/commit-plugin:commit` 사용, Human-Time/Ai-Driven-Time 확인).
2. **PR 생성 여부를 프로젝트 CLAUDE.md의 `Git.Platform` 값으로 결정**한다:

   ```markdown
   ## Git
   - **Platform**: github  # github | bitbucket | none
   ```

   | Platform 값 | 동작 |
   |---|---|
   | `github` | `gh pr create` — RDR/ADR/리뷰결과를 PR description에 자동 포함 |
   | `bitbucket` | Bitbucket MCP 또는 수동 안내 — 동일하게 PR description 생성 |
   | `none` / 미설정 | 커밋 + PR 본문 출력 후 종료 |

3. PR 생성 시 본문에 포함되는 내용:
   - 설계 RDR 요약
   - 코드 리뷰 RDR 요약
   - 관련 ADR 목록
   - 변경 파일 및 구현 요약
4. 사람에게 다음 행동을 묻는다:
   - "다음 태스크" → Phase 1로 복귀
   - "종료" → 파이프라인 종료
